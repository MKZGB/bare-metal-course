Module 10 Interrupts & the NVIC

How the Cortex-M3 takes an exception, how the NVIC enables and dispatches a peripheral IRQ, and a real, QEMU-testable demo: a USART1 receive interrupt (RXNEIE → USART1_IRQHandler, IRQ 37).

Dependencies This builds directly on Module 07 (UART Receive), which received bytes by polling the RXNE flag. Here we replace that poll with an interrupt. The vector table you will hook into lives in common/startup.c (explained in Modules 02–03); the SysTick interrupt of Module 08 was your first taste of an ISR.
Theory

1. What an interrupt is, and why you want one

Polling — sitting in while ((SR & RXNE) == 0) {} — wastes the CPU: it does nothing useful while it waits, and it can only watch one thing at a time. An interrupt inverts the relationship. The peripheral raises a request when it needs attention; the CPU drops what it is doing, runs a short Interrupt Service Routine (ISR), and then resumes exactly where it was. The foreground code never polls.

On a Cortex-M3 every interrupt and fault is an exception with a number. Exception numbers 1–15 are the system exceptions built into the core (Reset, NMI, HardFault, SVC, PendSV, SysTick, …). Exception numbers 16+ are the external interrupts from peripherals, which are also numbered from zero as IRQ 0, 1, 2, … (so IRQ n = exception n+16). The block that manages all of them is the NVIC.

2. The NVIC — Nested Vectored Interrupt Controller

The NVIC is part of the Cortex-M3 core (not an STM32 peripheral), mapped in the System Control Space at 0xE000Exxx. Its three jobs are in its name:

  • Nested — a higher-priority interrupt can pre-empt a running lower-priority ISR.
  • Vectored — hardware looks up the handler address directly from the vector table; you do not write a dispatcher that asks "who interrupted me?".
  • Interrupt Controller — it holds the enable, pending, and priority state for every IRQ.

2.1 What the hardware does when an IRQ fires

peripheral (USART1) sets RXNE, RXNEIE=1 ──► request to NVIC │ NVIC: is IRQ 37 enabled? high enough priority? ▼ CORE TAKES THE EXCEPTION (all in hardware, no code): 1. push R0-R3, R12, LR, PC, xPSR onto the current stack (8 words) 2. load PC from vector-table slot for IRQ 37 (g_vector_table[53]) 3. run USART1_IRQHandler() ── an ordinary C function 4. on return (BX LR with magic EXC_RETURN): pop those 8 registers ▼ resume the interrupted instruction as if nothing happened

Because the eight caller-saved registers are stacked and restored by hardware, an ISR on Cortex-M can be a plain C function — there is no prologue/epilogue assembly to write. That is why common/startup.c can declare every handler as a normal void f(void).

2.2 The vector table is the dispatch mechanism

"Vectored" means the handler is found by table index, not by a software switch. Slot 0 is the initial stack pointer, slot 1 is Reset_Handler, slots 2–15 are the system exceptions, and slot 16 + n is the handler for IRQ n. In common/startup.c every entry is a WEAK alias to Default_Handler (an infinite trap). When your module defines a function with the matching name — e.g. USART1_IRQHandler — the linker overrides the weak symbol, and slot 53 (IRQ 37) now points at your code. You "register" an ISR simply by defining a correctly-named function.

2.3 Enabling an IRQ: the ISER registers

An IRQ does nothing until you enable it in the NVIC. The set-enable registers are a small array; write a 1 to a bit to enable that line (writing 0 is ignored, so there is no read-modify-write and no race):

3. EXTI: making a GPIO pin interrupt (the button)

A GPIO pin cannot drive an IRQ on its own. The EXTI (External Interrupt/event) controller sits between the pins and the NVIC. To make the user button B1 = PA0 generate an interrupt you would, conceptually:

  1. Clock AFIO, then point EXTI line 0 at port A in AFIO_EXTICR1 (0x40010008).
  2. Unmask line 0 in EXTI_IMR (0x40010400+0x00, set bit 0).
  3. Select the edge: rising in EXTI_RTSR (offset 0x08, bit 0) — the button is active-high.
  4. Enable IRQ 6 (EXTI0) in the NVIC: ISER0 |= (1U << 6).
  5. In EXTI0_IRQHandler, clear the pending bit by writing 1 to bit 0 of EXTI_PR (offset 0x14). EXTI_PR is write-1-to-clear; if you forget this the handler re-fires forever.
QEMU cannot press the button Per the verified emulation matrix, qemu-system-arm -M stm32vldiscovery does not emulate GPIO or button input at all — IDR reads 0 and no pin edge is ever produced. So although the EXTI/NVIC setup above is correct for real hardware, you cannot trigger a button IRQ in QEMU. We therefore cover EXTI conceptually and make the working demo a peripheral QEMU does fully emulate: the USART.

4. The demonstrable interrupt: USART1 RX (RXNE → IRQ 37)

QEMU fully emulates the USARTs, including raising the USART1 interrupt when a byte arrives. So we turn Module 07's polled echo into an interrupt-driven one:

  • Set RXNEIE (USART1_CR1 bit 5): now whenever RXNE becomes 1, the USART asserts its IRQ line.
  • Enable IRQ 37 in the NVIC (ISER1 bit 5).
  • In USART1_IRQHandler: confirm RXNE, read DR (this returns the byte and clears RXNE, de-asserting the request), echo it, and bump a counter.

4.1 Sharing data with the handler: volatile

The ISR runs asynchronously to main(). Any variable both touch must be declared volatile, or the optimiser may cache it in a register inside main()'s loop and never see the handler's updates:

static volatile uint32_t g_rx_count = 0U;   /* written by ISR, read by main */

For data wider than one aligned 32-bit word, you also need a critical section — briefly disable interrupts so main() reads a consistent snapshot. On Cortex-M that is __asm volatile("cpsid i") (mask) … "cpsie i" (unmask), or the PRIMASK register. Our counter is a single word, so a plain volatile suffices here.

5.2 Build & run

From inside the 10-interrupts-nvic/ folder:

make -f ../common/Makefile SRC=main.c

Run interactively and type, or pipe input after a short delay so the bytes arrive once the interrupt is armed:

(sleep 1; printf 'AB') | qemu-system-arm -M stm32vldiscovery -display none \
    -serial stdio -serial null -serial null -kernel build/firmware.elf

You should see AB echoed by the ISR, then a [main] byte #… report line. Quit with Ctrl-A then X.

Why the sleep 1 matters in QEMU QEMU front-loads instantly-piped stdin and may deliver it before the guest finishes reset, so printf 'AB' | qemu… can miss the bytes entirely — this affects even the polled receiver of Module 07, so it is not an interrupt bug. A small sleep (or just typing interactively) makes the bytes arrive after uart_init() has armed RXNEIE and the NVIC. This is purely a host-side timing quirk; on real hardware the interrupt fires whenever a byte actually arrives.

5.3 Hints

Hint 1 — the NVIC enable (TODO 1)

The ISER registers form a 32-bit array at 0xE000E100. For IRQ n, the word is n / 32 and the bit is n % 32. Using shifts/masks: NVIC_ISER[irqn >> 5U] = (1U << (irqn & 0x1FU)); No read-modify-write: these are write-1-to-set registers, so writing the single bit cannot disturb others.

Hint 2 — arming the source (TODO 2 & 3)

One combined store does it: USART1_CR1 = USART_CR1_TE | USART_CR1_RE | USART_CR1_RXNEIE | USART_CR1_UE; Then nvic_enable_irq(USART1_IRQn);. Both halves are needed: RXNEIE lets the USART raise the request, the NVIC bit lets the core accept it. Omit either and nothing fires.

Hint 3 — the ISR body (TODO 4)

while ((USART1_SR & USART_SR_RXNE) != 0U) { uint8_t c = (uint8_t)(USART1_DR & 0xFFU); uart_putc(c); if (c=='\r') uart_putc('\n'); g_last_char=c; g_rx_count++; } The USART1_DR read is what clears RXNE — that is your acknowledgement to the hardware. No manual register save/restore is needed; the core stacked your context before entering.

Verification Questions

6. Check your understanding

Q1. TIM3 is IRQ 29 and USART2 is IRQ 38. Which ISER register and bit enables each?

Index = IRQ/32, bit = IRQ%32.
TIM3 (29): 29/32 = 0 → ISER0 (0xE000E100), bit 29%32 = 29.
USART2 (38): 38/32 = 1 → ISER1 (0xE000E104), bit 38%32 = 6.
So ISER0 = (1U << 29) and ISER1 = (1U << 6) respectively.

Q2. Why does a Cortex-M3 ISR need no manual register save/restore (no assembly prologue)?

On exception entry the core automatically pushes the eight caller-saved registers (R0–R3, R12, LR, PC, xPSR) onto the active stack, and on exception return it pops them back. The C calling convention says a function may freely use those registers, and the callee-saved ones it must preserve itself — which the compiler already does. So the hardware-stacked frame plus normal C codegen means a plain C function is a complete, correct handler. (This is why common/startup.c declares handlers as ordinary void f(void).)

Q3. Why must a variable shared between an ISR and main() be volatile?

The ISR runs asynchronously, so from the compiler's view main() appears to be the only writer of that variable. Without volatile it may legally hoist the variable into a register at the top of a loop and never re-read memory, so it never observes the ISR's updates (the loop would spin forever on a stale value). volatile forces an actual memory access on every read/write. (For multi-word data you additionally need a critical section — briefly mask interrupts with cpsid i/cpsie i — so the reader gets a consistent snapshot.)

Q4. What is the difference between a "system exception" like SysTick and an "external interrupt" like USART1?

System exceptions (numbers 1–15: Reset, NMI, HardFault, SVC, PendSV, SysTick, …) are built into the Cortex-M3 core itself; they are not in the NVIC's IRQ-numbering space and have no ISER bit. SysTick, for instance, is enabled through its own SYST_CSR.TICKINT bit (Module 08). External interrupts are the peripheral IRQs, numbered 0+ (exception 16+), and each one is individually enabled/disabled/prioritised through the NVIC's ISER/ICER/IPR registers. Both kinds dispatch through the same vector table.

Q5. For an EXTI (button) interrupt, how do you clear the pending flag, and what happens if you forget?

You write a 1 to the corresponding bit of EXTI_PR (Pending Register, offset 0x14) — it is write-1-to-clear, so writing 1 clears the flag and 0 leaves it alone. If you forget, the EXTI line stays pending: the moment the handler returns, the NVIC sees the still-asserted request and immediately re-enters the handler, so it appears to fire forever. The same principle applies to every interrupt — the handler must clear the source (for USART RX it is the act of reading DR; for TIM it is clearing the UIF in SR).
