Módulo 10 Interrupções e o NVIC

Como o Cortex-M3 aceita uma exceção, como o NVIC ativa e despacha uma IRQ de periférico, e uma demonstração real, testável em QEMU: uma interrupção de receção da USART1 (RXNEIE → USART1_IRQHandler, IRQ 37).

Dependências Isto assenta diretamente no Módulo 07 (Receção UART), que recebia bytes por polling da flag RXNE. Aqui substituímos esse polling por uma interrupção. A tabela de vetores onde se vai ligar está em common/startup.c (explicada nos Módulos 02–03); a interrupção do SysTick do Módulo 08 foi a sua primeira amostra de uma ISR.
Teoria

1. O que é uma interrupção, e porque a quer

Polling — ficar dentro de while ((SR & RXNE) == 0) {} — desperdiça o CPU: não faz nada de útil enquanto espera, e só consegue vigiar uma coisa de cada vez. Uma interrupção inverte a relação. O periférico levanta um pedido quando precisa de atenção; o CPU larga o que está a fazer, corre uma curta Rotina de Serviço de Interrupção (ISR), e depois retoma exatamente onde estava. O código de primeiro plano nunca faz polling.

Num Cortex-M3 cada interrupção e falha é uma exceção com um número. Os números de exceção 1–15 são as exceções de sistema integradas no núcleo (Reset, NMI, HardFault, SVC, PendSV, SysTick, …). Os números de exceção 16+ são as interrupções externas dos periféricos, que também são numeradas a partir de zero como IRQ 0, 1, 2, … (logo IRQ n = exceção n+16). O bloco que gere todas elas é o NVIC.

2. O NVIC — Nested Vectored Interrupt Controller

O NVIC faz parte do núcleo Cortex-M3 (não é um periférico STM32), mapeado no System Control Space em 0xE000Exxx. As suas três funções estão no próprio nome:

  • Nested (aninhado) — uma interrupção de prioridade mais alta pode preemptar uma ISR de prioridade mais baixa em execução.
  • Vectored (vetorizado) — o hardware procura o endereço do tratador diretamente na tabela de vetores; não se escreve um despachante que pergunta "quem me interrompeu?".
  • Interrupt Controller (controlador de interrupções) — guarda o estado de ativação, pendência e prioridade de cada IRQ.

2.1 O que o hardware faz quando uma IRQ dispara

periférico (USART1) põe RXNE, RXNEIE=1 ──► pedido ao NVIC │ NVIC: a IRQ 37 está ativada? prioridade suficiente? ▼ O NÚCLEO ACEITA A EXCEÇÃO (tudo em hardware, sem código): 1. empilha R0-R3, R12, LR, PC, xPSR na pilha atual (8 palavras) 2. carrega PC a partir da posição da tabela de vetores para IRQ 37 (g_vector_table[53]) 3. corre USART1_IRQHandler() ── uma função C vulgar 4. ao regressar (BX LR com EXC_RETURN mágico): desempilha esses 8 registos ▼ retoma a instrução interrompida como se nada tivesse acontecido

Como os oito registos guardados pelo chamador são empilhados e restaurados pelo hardware, uma ISR no Cortex-M pode ser uma função C simples — não há prólogo/epílogo em assembly para escrever. É por isso que common/startup.c pode declarar cada tratador como um void f(void) normal.

2.2 A tabela de vetores é o mecanismo de despacho

"Vetorizado" significa que o tratador é encontrado por índice na tabela, e não por um switch em software. A posição 0 é o ponteiro de pilha inicial, a posição 1 é o Reset_Handler, as posições 2–15 são as exceções de sistema, e a posição 16 + n é o tratador da IRQ n. Em common/startup.c cada entrada é um alias WEAK para Default_Handler (uma armadilha infinita). Quando o seu módulo define uma função com o nome correspondente — p. ex. USART1_IRQHandler — o linker sobrepõe-se ao símbolo fraco, e a posição 53 (IRQ 37) passa a apontar para o seu código. "Regista-se" uma ISR simplesmente definindo uma função com o nome correto.

2.3 Ativar uma IRQ: os registos ISER

Uma IRQ não faz nada até a ativar no NVIC. Os registos set-enable são um pequeno array; escreva um 1 num bit para ativar essa linha (escrever 0 é ignorado, por isso não há read-modify-write nem corrida):

RegistoEndereçoAbrange
NVIC_ISER00xE000E100IRQ 0 … 31 (bit n = IRQ n)
NVIC_ISER10xE000E104IRQ 32 … 63 (bit n = IRQ 32+n)
NVIC_ICER0/10xE000E180clear-enable (desativar uma IRQ)
NVIC_ISPR0/10xE000E200set-pending (disparar uma IRQ por software)
NVIC_IPRn0xE000E400prioridade, um byte por IRQ — só os 4 bits de topo são usados

A aritmética de índice/bit é o coração deste módulo. Para a IRQ n:

índice do registo = n / 32 (qual palavra ISER) posição do bit = n % 32 (qual bit nessa palavra) NVIC_ISER[n / 32] = (1U << (n % 32)); Exemplo — USART1 = IRQ 37: índice = 37 / 32 = 1 → ISER1 (0xE000E104) bit = 37 % 32 = 5 ⇒ *(volatile uint32_t*)0xE000E104 = (1U << 5);

Os números de IRQ são da Tabela 51 do RM0041. O SysTick é diferente: é uma exceção de sistema (número 15), não uma IRQ externa, por isso não tem qualquer bit ISER — ativa-se através de SYST_CSR.TICKINT (Módulo 08). As prioridades (IPR) estão fora do âmbito aqui; a prioridade de reset 0 funciona bem para uma única interrupção.

3. EXTI: fazer um pino GPIO interromper (o botão)

Um pino GPIO não consegue acionar uma IRQ sozinho. O controlador EXTI (External Interrupt/event) fica entre os pinos e o NVIC. Para fazer o botão de utilizador B1 = PA0 gerar uma interrupção, faria, conceptualmente:

  1. Dar relógio ao AFIO, depois apontar a linha 0 do EXTI para o porto A em AFIO_EXTICR1 (0x40010008).
  2. Desmascarar a linha 0 em EXTI_IMR (0x40010400+0x00, pôr o bit 0).
  3. Selecionar o flanco: ascendente em EXTI_RTSR (offset 0x08, bit 0) — o botão é ativo-alto.
  4. Ativar a IRQ 6 (EXTI0) no NVIC: ISER0 |= (1U << 6).
  5. No EXTI0_IRQHandler, limpar o bit de pendência escrevendo 1 no bit 0 de EXTI_PR (offset 0x14). O EXTI_PR é write-1-to-clear; se se esquecer disto o tratador volta a disparar para sempre.
O QEMU não consegue carregar no botão Segundo a matriz de emulação verificada, o qemu-system-arm -M stm32vldiscovery não emula GPIO nem entrada de botão de todo — o IDR lê 0 e nunca é produzido qualquer flanco no pino. Por isso, embora a configuração EXTI/NVIC acima esteja correta para hardware real, não é possível acionar uma IRQ de botão no QEMU. Cobrimos portanto o EXTI conceptualmente e fazemos da demonstração funcional um periférico que o QEMU de facto emula por completo: a USART.

4. A interrupção demonstrável: USART1 RX (RXNE → IRQ 37)

O QEMU emula por completo as USARTs, incluindo levantar a interrupção da USART1 quando chega um byte. Por isso transformamos o eco por polling do Módulo 07 num eco guiado por interrupção:

  • Pôr RXNEIE (USART1_CR1 bit 5): agora, sempre que RXNE passa a 1, a USART afirma a sua linha de IRQ.
  • Ativar a IRQ 37 no NVIC (ISER1 bit 5).
  • No USART1_IRQHandler: confirmar RXNE, ler DR (isto devolve o byte e limpa RXNE, retirando a afirmação do pedido), ecoá-lo, e incrementar um contador.

4.1 Partilhar dados com o tratador: volatile

A ISR corre de forma assíncrona em relação a main(). Qualquer variável que ambos toquem tem de ser declarada volatile, ou o otimizador pode mantê-la em cache num registo dentro do ciclo de main() e nunca ver as atualizações do tratador:

static volatile uint32_t g_rx_count = 0U;   /* escrito pela ISR, lido pelo main */

Para dados mais largos do que uma palavra alinhada de 32 bits, também precisa de uma secção crítica — desativar interrupções por breves instantes para que main() leia um instantâneo consistente. No Cortex-M isso é __asm volatile("cpsid i") (mascarar) … "cpsie i" (desmascarar), ou o registo PRIMASK. O nosso contador é uma única palavra, por isso um simples volatile basta aqui.

5.2 Compilar e correr

De dentro da pasta 10-interrupts-nvic/:

make -f ../common/Makefile SRC=main.c

Corra interativamente e escreva, ou canalize entrada após um pequeno atraso para que os bytes cheguem assim que a interrupção estiver armada:

(sleep 1; printf 'AB') | qemu-system-arm -M stm32vldiscovery -display none \
    -serial stdio -serial null -serial null -kernel build/firmware.elf

Deverá ver AB ecoado pela ISR, depois uma linha de relatório [main] byte #…. Saia com Ctrl-A seguido de X.

Porque é que o sleep 1 importa no QEMU O QEMU carrega instantaneamente o stdin canalizado e pode entregá-lo antes de o guest terminar o reset, por isso printf 'AB' | qemu… pode falhar os bytes por completo — isto afeta até o recetor por polling do Módulo 07, logo não é um bug de interrupção. Um pequeno sleep (ou simplesmente escrever interativamente) faz os bytes chegarem depois de uart_init() ter armado o RXNEIE e o NVIC. Isto é puramente uma peculiaridade de temporização do lado do host; em hardware real a interrupção dispara sempre que um byte realmente chega.

5.3 Pistas

Pista 1 — a ativação do NVIC (TODO 1)

Os registos ISER formam um array de 32 bits em 0xE000E100. Para a IRQ n, a palavra é n / 32 e o bit é n % 32. Usando shifts/máscaras: NVIC_ISER[irqn >> 5U] = (1U << (irqn & 0x1FU)); Sem read-modify-write: estes são registos write-1-to-set, por isso escrever o único bit não perturba os outros.

Pista 2 — armar a fonte (TODO 2 e 3)

Um store combinado resolve: USART1_CR1 = USART_CR1_TE | USART_CR1_RE | USART_CR1_RXNEIE | USART_CR1_UE; Depois nvic_enable_irq(USART1_IRQn);. São precisas as duas metades: RXNEIE deixa a USART levantar o pedido, o bit do NVIC deixa o núcleo aceitá-lo. Omita qualquer uma e nada dispara.

Perguntas de Verificação

6. Verifique a sua compreensão

P1. O TIM3 é IRQ 29 e a USART2 é IRQ 38. Que registo ISER e que bit ativa cada um?

Índice = IRQ/32, bit = IRQ%32.
TIM3 (29): 29/32 = 0 → ISER0 (0xE000E100), bit 29%32 = 29.
USART2 (38): 38/32 = 1 → ISER1 (0xE000E104), bit 38%32 = 6.
Logo ISER0 = (1U << 29) e ISER1 = (1U << 6) respetivamente.

P2. Porque é que uma ISR no Cortex-M3 não precisa de guardar/restaurar registos manualmente (sem prólogo em assembly)?

Na entrada da exceção o núcleo empilha automaticamente os oito registos guardados pelo chamador (R0–R3, R12, LR, PC, xPSR) na pilha ativa, e no regresso da exceção desempilha-os de volta. A convenção de chamada de C diz que uma função pode usar livremente esses registos, e os guardados pelo chamado tem de preservá-los ela própria — o que o compilador já faz. Por isso a frame empilhada pelo hardware mais a geração normal de código C significam que uma função C simples é um tratador completo e correto. (É por isso que common/startup.c declara os tratadores como void f(void) vulgares.)

P3. Porque é que uma variável partilhada entre uma ISR e o main() tem de ser volatile?

A ISR corre de forma assíncrona, por isso, do ponto de vista do compilador, o main() parece ser o único escritor dessa variável. Sem volatile pode legalmente içar a variável para um registo no topo de um ciclo e nunca reler a memória, por isso nunca observa as atualizações da ISR (o ciclo ficaria a girar para sempre num valor obsoleto). O volatile força um acesso real à memória em cada leitura/escrita. (Para dados de várias palavras precisa adicionalmente de uma secção crítica — mascarar brevemente as interrupções com cpsid i/cpsie i — para que o leitor obtenha um instantâneo consistente.)

P4. Qual é a diferença entre uma "exceção de sistema" como o SysTick e uma "interrupção externa" como a USART1?

As exceções de sistema (números 1–15: Reset, NMI, HardFault, SVC, PendSV, SysTick, …) estão integradas no próprio núcleo Cortex-M3; não estão no espaço de numeração de IRQ do NVIC e não têm bit ISER. O SysTick, por exemplo, é ativado através do seu próprio bit SYST_CSR.TICKINT (Módulo 08). As interrupções externas são as IRQs dos periféricos, numeradas a partir de 0 (exceção 16+), e cada uma é ativada/desativada/priorizada individualmente através dos registos ISER/ICER/IPR do NVIC. Ambos os tipos despacham pela mesma tabela de vetores.

P5. Para uma interrupção EXTI (botão), como se limpa a flag de pendência, e o que acontece se a esquecer?

Escreve-se um 1 no bit correspondente de EXTI_PR (Pending Register, offset 0x14) — é write-1-to-clear, por isso escrever 1 limpa a flag e 0 deixa-a inalterada. Se a esquecer, a linha EXTI fica pendente: no momento em que o tratador regressa, o NVIC vê o pedido ainda afirmado e reentra imediatamente no tratador, parecendo disparar para sempre. O mesmo princípio aplica-se a toda a interrupção — o tratador tem de limpar a fonte (para a receção USART é o ato de ler DR; para o TIM é limpar o UIF em SR).
