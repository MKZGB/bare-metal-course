Módulo 09 Temporizadores de Uso Geral — Base de Tempo, PWM, Captura de Entrada

Configura o TIM3 como uma base de tempo de 1 Hz a partir do relógio de 8 MHz usando PSC/ARR/CEN, observa a flag de atualização UIF e conhece a Comparação de Saída (PWM) e a Captura de Entrada — tudo isto sem deixar de ser honesto sobre o que o QEMU não emula.

Dependências Este módulo assenta no Módulo 06 (UART + printf) para ter saída visível e reutiliza a ideia de gating de relógio do Módulo 04. O Módulo 08 (SysTick) deu-te um temporizador do núcleo; aqui conheces um verdadeiro temporizador periférico com prescaler, auto-recarga e canais de captura/comparação.
Teoria

1. A unidade de base de tempo

O STM32F100 tem três temporizadores de uso geral — TIM2, TIM3, TIM4 — cada um um contador ascendente/descendente de 16 bits com quatro canais de captura/comparação. Estão no barramento APB1. Este módulo usa o TIM3 (base 0x40000400). No coração de cada temporizador está a unidade de base de tempo: três registos que transformam o relógio do barramento num tique programável.

RegistoEndereçoFunção
TIM3_PSC0x40000428Prescaler: divide o relógio de entrada por PSC + 1.
TIM3_ARR0x4000042CAuto-recarga: o contador dá a volta após atingir este valor.
TIM3_CNT0x40000424Contador: a contagem em tempo real (0 → ARR → 0 …).

O relógio de contagem é:

f_CK_CNT = f_CK_PSC / (PSC + 1) entrada de 8 MHz, PSC = 7999: 8 000 000 / (7999 + 1) = 1 000 Hz (um tique por milissegundo) CNT conta então 0,1,2,...,ARR e dá a volta. Com ARR = 999 dá a volta a cada (999 + 1) = 1000 tiques = uma vez por segundo: taxa de atualização = f_CK_CNT / (ARR + 1) = 1000 / 1000 = 1 Hz

Sempre que o contador dá a volta, ocorre um evento de atualização e o hardware coloca a UIF (Update Interrupt Flag) no TIM3_SR bit 0. Fazer polling — ou interromper — sobre esta flag é como se constrói um tique periódico. O software tem de limpar a UIF (escrever 0 nela) antes do próximo período, ou nunca conseguirás distinguir dois períodos um do outro.

1.1 Os bits de controlo: CEN e UG

BitRegistoSignificado
CEN (bit 0)TIM3_CR1 (0x00)Counter Enable — o interruptor principal. Enquanto CEN = 0 o contador está congelado.
UG (bit 0)TIM3_EGR (0x14)Update Generation — escrever 1 força um evento de atualização imediatamente.
UIF (bit 0)TIM3_SR (0x10)Flag de atualização — colocada pelo hardware em cada volta; limpa por software.
Porque é que tens de pulsar UG depois de escrever PSC O prescaler é bufferizado: um valor escrito no PSC só tem efeito no próximo evento de atualização. Se escreveres o PSC e depois arrancares o contador, o primeiro período ainda usa o prescaler antigo. Escrever UG = 1 força uma atualização imediata que recarrega o prescaler com o novo PSC e repõe o CNT a 0. (O UG também coloca a UIF como efeito secundário, por isso limpa a UIF depois.)
⚠ CRÍTICO: o QEMU NÃO emula o TIM2/3/4 No qemu-system-arm -M stm32vldiscovery, os temporizadores de uso geral não são emulados de todo. O contador CNT permanece congelado em 0 e a flag de atualização UIF nunca se ativa. Isto significa que um ingénuo while ((TIM3_SR & UIF) == 0U) {} ficaria preso para sempre no QEMU. O teu código nunca deve fazer uma espera sem limite sobre uma flag de temporizador. Usa o padrão de espera limitada (secção 4): faz spin sobre UIF com um contador de guarda e, se nunca se ativar, imprime que isto é a limitação esperada do QEMU. Em hardware STM32F100 real o temporizador corre e a UIF ativa-se a 1 Hz.

2. Comparação de Saída e PWM

Para além de uma simples base de tempo, cada canal do temporizador pode comparar o contador com um valor alvo guardado num registo de captura/comparação CCRx e acionar um pino de saída quando coincidem. Isto é a Comparação de Saída (OC), e o seu uso mais comum é o PWM (Modulação por Largura de Impulso).

RegistoOffsetFunção no PWM
CCMR10x18Modo por canal. Os bits OC1M[2:0] selecionam o modo PWM 1 (110): a saída está ativa enquanto CNT < CCR1.
CCER0x20CC1E (bit 0) ativa a saída do canal no pino; CC1P (bit 1) define a polaridade.
CCR10x34O valor de comparação — define o ciclo de trabalho (duty cycle).
frequência PWM = f_CK_CNT / (ARR + 1) (definida por PSC e ARR) ciclo de trabalho PWM = CCR1 / (ARR + 1) (definido por CCR1) ARR = 999, CCR1 = 250 -> 25% de duty ARR = 999, CCR1 = 500 -> 50% de duty CNT: 0 .......... 250 .......... 999 | 0 .... PA6: |__ALTO_____|____BAIXO________| |__ALTO... ^ CNT cruzou CCR1, a saída cai

Para o canal 1 do TIM3 o pino de saída é o PA6, configurado como função alternativa push-pull (nibble do CRL 0xB, tal como o pino TX da UART) para que seja o temporizador — e não o registo de dados do GPIO — a acioná-lo. Altera o ARR para definir a frequência, altera o CCR1 para definir o brilho/duty. O PWM é a forma de regular a intensidade de um LED ou definir o ângulo de um servo sem um DAC.

Ressalva do QEMU para o PWM Como o TIM2/3/4 não é emulado e o GPIO também não, não consegues observar uma forma de onda PWM no QEMU. A configuração de registos abaixo está correta para hardware real; neste curso mantemos a OC/PWM conceptual e verificamos apenas a lógica da base de tempo.

3. Captura de Entrada

A Captura de Entrada (IC) é a Comparação de Saída ao contrário. Em vez de o contador acionar um pino, um flanco externo num pino faz o hardware copiar o CNT atual para o CCRx — uma marca temporal de quando o flanco chegou. Configura-se através do mesmo registo CCMR1, mas com os bits CC1S[1:0] definidos para o modo de entrada (por exemplo, 01 = mapear IC1 para TI1).

  • Captura o CNT num flanco ascendente e depois num flanco descendente → a diferença é a largura do impulso.
  • Captura o CNT em dois flancos ascendentes sucessivos → a diferença é o período (logo 1 / período dá a frequência da entrada).
  • A flag CC1IF (SR bit 1) sinaliza "aconteceu uma captura"; ler o CCR1 limpa-a.

4. Faz o TIM3 dar tique a 1 Hz (espera limitada)

Objetivo: pôr o TIM3 a funcionar como uma base de tempo de 1 Hz, imprimir o que configuraste pela USART1, e observar alguns eventos de atualização — sem nunca ficar preso no QEMU. Vais:

  1. Ativar o relógio do TIM3 (RCC_APB1ENR bit 1).
  2. Escrever PSC = 7999 e ARR = 999 para uma atualização de 1 Hz.
  3. Pulsar o UG para carregar o prescaler, limpar a UIF, e depois ativar o CEN.
  4. Usar uma espera limitada sobre a UIF para que a demonstração termine no QEMU.

4.3 Pistas

Pista 1 — a matemática do PSC/ARR

Relógio de contagem = 8000000 / (PSC + 1). Para 1 kHz precisas de PSC + 1 = 8000 → PSC = 7999. Taxa de atualização = 1000 / (ARR + 1). Para 1 Hz precisas de ARR + 1 = 1000 → ARR = 999. Logo TIM3_PSC = 7999U; e TIM3_ARR = 999U;.

Pista 2 — carregar o prescaler e arrancar

Pulsa a geração de atualização, depois limpa a flag que ela ativou, depois ativa o contador: TIM3_EGR = TIM_EGR_UG; TIM3_SR &= ~TIM_SR_UIF; TIM3_CR1 |= TIM_CR1_CEN;

Pista 3 — a espera limitada (NÃO ficar preso)

Nunca escrevas while ((TIM3_SR & TIM_SR_UIF) == 0U) {} — no QEMU a UIF nunca se ativa e isso faz loop para sempre. Em vez disso:

uint32_t guard = 2000000U;
while (((TIM3_SR & TIM_SR_UIF) == 0U) && (guard > 0U)) { guard--; }
if (guard == 0U) { return 0U; }        /* UIF nunca se ativou (QEMU) */
TIM3_SR &= ~TIM_SR_UIF;                 /* limpar para o próximo período */
return 1U;

Em hardware real a UIF ativa-se dentro de ~1 s e o loop sai mais cedo com guard > 0.

Perguntas de Verificação

5. Verifica a tua compreensão

P1. Queres uma atualização de 2 Hz a partir do relógio de 8 MHz. Dá um par PSC/ARR válido.

Mantém o tique de 1 kHz: PSC = 7999 (8 000 000 / 8000 = 1 kHz). Então atualização = 1000 / (ARR + 1) = 2 → ARR + 1 = 500 → ARR = 499. Muitos pares funcionam; outro é PSC = 3999 (→ 2 kHz) com ARR = 999. O produto (PSC+1)·(ARR+1) tem de ser igual a 8 000 000 / 2 = 4 000 000.

P2. O que é a flag UIF e a quem cabe limpá-la?

A UIF (Update Interrupt Flag, TIM3_SR bit 0) é colocada pelo hardware em cada evento de atualização (volta do contador, ou um UG forçado). O software tem de limpá-la (escrever 0) antes do próximo período; caso contrário não consegues distinguir um período do seguinte, e um handler de interrupção voltaria a disparar infinitamente.

P3. Qual é a diferença entre Comparação de Saída (PWM) e Captura de Entrada?

Comparação de Saída / PWM: o contador aciona um pino. Escreves um alvo no CCR1 e o hardware comuta a saída quando o CNT o atinge — usado para gerar uma forma de onda (define o duty via CCR1, a frequência via ARR). Captura de Entrada: um flanco externo num pino aciona um registo. O hardware copia o CNT para o CCR1 no flanco, marcando-o temporalmente — usado para medir a largura de impulso ou a frequência de uma entrada.

P4. Porque é que a demonstração não pode contar com o QEMU para mostrar o temporizador a dar tique?

A máquina stm32vldiscovery do QEMU não emula o TIM2/3/4: o CNT permanece a 0 e a UIF nunca se ativa. Um while (!(TIM3_SR & UIF)) sem limite ficaria preso para sempre. Por isso usamos uma espera limitada com um contador de guarda que termina sempre, e imprimimos que a flag em falta é a limitação esperada do QEMU. A configuração de registos continua correta e o temporizador funciona em hardware STM32F100 real.

P5. O que faz cada um dos CEN e UG, e porquê pulsar o UG logo após escrever o PSC?

O CEN (CR1 bit 0) é a ativação principal — o contador não corre até estar ativo. O UG (EGR bit 0) força um evento de atualização imediato. O prescaler é bufferizado, por isso um PSC acabado de escrever só tem efeito na próxima atualização; pulsar o UG faz isso acontecer agora (recarregando também CNT = 0), de modo a que o primeiríssimo período já use o teu novo prescaler. Lembra-te de que o UG ativa a UIF, por isso limpa-a antes de arrancar.
