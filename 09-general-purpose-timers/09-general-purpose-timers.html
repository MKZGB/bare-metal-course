Module 09 General-Purpose Timers — Time Base, PWM, Input Capture

Configure TIM3 as a 1 Hz time base from the 8 MHz clock using PSC/ARR/CEN, watch the UIF update flag, and meet Output Compare (PWM) and Input Capture — all while staying honest about what QEMU does not emulate.

Dependencies This module builds on Module 06 (UART + printf) for visible output and reuses the clock-gating idea from Module 04. Module 08 (SysTick) gave you a core timer; here you meet a real peripheral timer with prescaler, auto-reload and capture/compare channels.
Theory

1. The time-base unit

The STM32F100 has three general-purpose timers — TIM2, TIM3, TIM4 — each a 16-bit up/down counter with four capture/compare channels. They sit on the APB1 bus. This module uses TIM3 (base 0x40000400). At the heart of every timer is the time-base unit: three registers that turn the bus clock into a programmable tick.

RegisterAddressRole
TIM3_PSC0x40000428Prescaler: divides the input clock by PSC + 1.
TIM3_ARR0x4000042CAuto-reload: the counter wraps after reaching this value.
TIM3_CNT0x40000424Counter: the live count (0 → ARR → 0 …).

The counting clock is:

f_CK_CNT = f_CK_PSC / (PSC + 1) 8 MHz input, PSC = 7999: 8 000 000 / (7999 + 1) = 1 000 Hz (one tick per millisecond) CNT then counts 0,1,2,...,ARR and wraps. With ARR = 999 it wraps every (999 + 1) = 1000 ticks = once per second: update rate = f_CK_CNT / (ARR + 1) = 1000 / 1000 = 1 Hz

Each time the counter wraps, an update event occurs and the hardware sets the UIF (Update Interrupt Flag) in TIM3_SR bit 0. Polling — or interrupting on — this flag is how you build a periodic tick. Software must clear UIF (write 0 to it) before the next period, or you can never tell two periods apart.

1.1 The control bits: CEN and UG

BitRegisterMeaning
CEN (bit 0)TIM3_CR1 (0x00)Counter Enable — the master switch. While CEN = 0 the counter is frozen.
UG (bit 0)TIM3_EGR (0x14)Update Generation — writing 1 forces an update event right now.
UIF (bit 0)TIM3_SR (0x10)Update flag — set by hardware on every wrap; cleared by software.
Why you must pulse UG after writing PSC The prescaler is buffered: a value written to PSC does not take effect until the next update event. If you write PSC and then start the counter, the first period still uses the old prescaler. Writing UG = 1 forces an immediate update that reloads the prescaler with the new PSC and resets CNT to 0. (UG also sets UIF as a side effect, so clear UIF afterwards.)
⚠ CRITICAL: QEMU does NOT emulate TIM2/3/4 On qemu-system-arm -M stm32vldiscovery, the general-purpose timers are not emulated at all. The counter CNT stays frozen at 0 and the update flag UIF never sets. This means a naive while ((TIM3_SR & UIF) == 0U) {} would hang forever in QEMU. Your code must never do an unbounded wait on a timer flag. Use the bounded-wait pattern (section 4): spin on UIF with a guard counter, and if it never sets, print that this is the expected QEMU limitation. On real STM32F100 hardware the timer runs and UIF sets at 1 Hz.

2. Output Compare and PWM

Beyond a plain time base, each timer channel can compare the counter against a target value held in a capture/compare register CCRx and drive an output pin when they match. This is Output Compare (OC), and its most common use is PWM (Pulse-Width Modulation).

RegisterOffsetRole in PWM
CCMR10x18Per-channel mode. The OC1M[2:0] bits select PWM mode 1 (110): output is active while CNT < CCR1.
CCER0x20CC1E (bit 0) enables the channel output on the pin; CC1P (bit 1) sets polarity.
CCR10x34The compare value — sets the duty cycle.
PWM frequency = f_CK_CNT / (ARR + 1) (set by PSC and ARR) PWM duty cycle = CCR1 / (ARR + 1) (set by CCR1) ARR = 999, CCR1 = 250 -> 25% duty ARR = 999, CCR1 = 500 -> 50% duty CNT: 0 .......... 250 .......... 999 | 0 .... PA6: |__HIGH_____|____LOW__________| |__HIGH... ^ CNT crossed CCR1, output drops

For TIM3 channel 1 the output pin is PA6, configured as alternate-function push-pull (CRL nibble 0xB, just like the UART TX pin) so the timer — not the GPIO data register — drives it. Change ARR to set frequency, change CCR1 to set brightness/duty. PWM is how you dim an LED or set a servo angle without a DAC.

QEMU caveat for PWM Because TIM2/3/4 are not emulated and GPIO is not emulated either, you cannot observe a PWM waveform in QEMU. The register configuration below is correct for real hardware; in this course we keep OC/PWM conceptual and verify only the time-base logic.

3. Input Capture

Input Capture (IC) is Output Compare run backwards. Instead of the counter driving a pin, an external edge on a pin causes the hardware to copy the current CNT into CCRx — a timestamp of when the edge arrived. Configure it through the same CCMR1 register, but with the CC1S[1:0] bits set to input mode (e.g. 01 = map IC1 to TI1).

  • Capture CNT on a rising edge, then on a falling edge → the difference is the pulse width.
  • Capture CNT on two successive rising edges → the difference is the period (so 1 / period gives the input frequency).
  • The CC1IF flag (SR bit 1) signals "a capture happened"; reading CCR1 clears it.

Guided Exercise

4. Make TIM3 tick at 1 Hz (bounded wait)

Goal: bring up TIM3 as a 1 Hz time base, print what you configured over USART1, and watch for a few update events — without ever hanging in QEMU. You will:

  1. Enable the TIM3 clock (RCC_APB1ENR bit 1).
  2. Write PSC = 7999 and ARR = 999 for a 1 Hz update.
  3. Pulse UG to load the prescaler, clear UIF, then set CEN.
  4. Use a bounded wait on UIF so the demo terminates in QEMU.

4.3 Hints

Hint 1 — the PSC/ARR math

Counting clock = 8000000 / (PSC + 1). For 1 kHz you need PSC + 1 = 8000 → PSC = 7999. Update rate = 1000 / (ARR + 1). For 1 Hz you need ARR + 1 = 1000 → ARR = 999. So TIM3_PSC = 7999U; and TIM3_ARR = 999U;.

Hint 2 — loading the prescaler and starting

Pulse update generation, then clear the flag it set, then enable the counter: TIM3_EGR = TIM_EGR_UG; TIM3_SR &= ~TIM_SR_UIF; TIM3_CR1 |= TIM_CR1_CEN;

Hint 3 — the bounded wait (do NOT hang)

Never write while ((TIM3_SR & TIM_SR_UIF) == 0U) {} — in QEMU UIF never sets and that loops forever. Instead:

uint32_t guard = 2000000U;
while (((TIM3_SR & TIM_SR_UIF) == 0U) && (guard > 0U)) { guard--; }
if (guard == 0U) { return 0U; }        /* UIF never set (QEMU) */
TIM3_SR &= ~TIM_SR_UIF;                 /* clear for next period */
return 1U;

On real hardware UIF sets within ~1 s and the loop exits early with guard > 0.

Verification Questions

5. Check your understanding

Q1. You want a 2 Hz update from the 8 MHz clock. Give one valid PSC/ARR pair.

Keep the 1 kHz tick: PSC = 7999 (8 000 000 / 8000 = 1 kHz). Then update = 1000 / (ARR + 1) = 2 → ARR + 1 = 500 → ARR = 499. Many pairs work; another is PSC = 3999 (→ 2 kHz) with ARR = 999. The product (PSC+1)·(ARR+1) must equal 8 000 000 / 2 = 4 000 000.

Q2. What is the UIF flag and whose job is it to clear it?

UIF (Update Interrupt Flag, TIM3_SR bit 0) is set by hardware on every update event (counter wrap, or a forced UG). The software must clear it (write 0) before the next period; otherwise you cannot distinguish one period from the next, and an interrupt handler would re-fire endlessly.

Q3. What is the difference between Output Compare (PWM) and Input Capture?

Output Compare / PWM: the counter drives a pin. You write a target into CCR1 and the hardware toggles the output when CNT reaches it — used to generate a waveform (set duty via CCR1, frequency via ARR). Input Capture: an external edge on a pin drives a register. The hardware copies CNT into CCR1 at the edge, timestamping it — used to measure an input's pulse width or frequency.

Q4. Why can't the demo rely on QEMU to show the timer ticking?

QEMU's stm32vldiscovery machine does not emulate TIM2/3/4: CNT stays 0 and UIF never sets. An unbounded while (!(TIM3_SR & UIF)) would hang forever. We therefore use a bounded wait with a guard counter that always terminates, and print that the missing flag is the expected QEMU limitation. The register configuration is still correct and the timer runs on real STM32F100 hardware.

Q5. What do CEN and UG each do, and why pulse UG right after writing PSC?

CEN (CR1 bit 0) is the master enable — the counter does not run until it is set. UG (EGR bit 0) forces an immediate update event. The prescaler is buffered, so a freshly written PSC only takes effect at the next update; pulsing UG makes that happen now (also reloading CNT = 0), so the very first period already uses your new prescaler. Remember UG sets UIF, so clear it before starting.
