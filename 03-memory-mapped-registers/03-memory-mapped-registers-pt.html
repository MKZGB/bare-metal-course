Módulo 03 Registos Mapeados em Memória (MMIO)

Como o hardware periférico aparece sob a forma de endereços de memória, porque é que volatile é inegociável, e como modelar um bloco de registos como uma struct C. Esta é a fundação que todos os módulos seguintes reutilizam.

O que este módulo te dá Ainda sem LED, sem UART, sem saída visível — este é um módulo de design. Vais aprender as duas formas de dar nome a um registo de hardware em C, os idiomas de manipulação de bits usados em todo o lado neste curso, e vais escrever à mão um pequeno ficheiro de cabeçalho com o mapa de registos do porto GPIO C. Do Módulo 04 em diante assume-se que dominas tudo o que está aqui.
Teoria

1. O que significa "E/S mapeada em memória"

Um Cortex-M3 tem um único espaço de endereçamento plano de 4 GB (endereços 0x00000000–0xFFFFFFFF). Alguns desses endereços são memória real — Flash e SRAM. Mas grandes regiões do mapa não são memória de todo: estão ligadas diretamente a hardware periférico. Ler ou escrever num desses endereços não armazena um valor; aciona um circuito.

Isto é E/S mapeada em memória (MMIO). No ARM não existe nenhuma "instrução de E/S" separada (ao contrário do IN/OUT do x86). Para acender um LED escreves num endereço; para ler um botão lês um endereço. Um registo é apenas uma localização de 32 bits, com nome, dentro de um periférico, que controla ou reporta algo.

Espaço de endereçamento do STM32F100 (as partes que interessam) 0xFFFFFFFF +---------------------------+ | ... | 0xE000E000 | Cortex-M3 core periph. | NVIC, SysTick (PM0056) | ... | 0x40018000 +---------------------------+ | AHB peripherals | RCC @0x40021000, DMA @0x40020000 0x40010000 +---------------------------+ | APB2 peripherals | GPIOA..D, USART1, SPI1, ADC, AFIO 0x40000000 +---------------------------+ | APB1 peripherals | TIM2..4, USART2/3, I2C, SPI2 +---------------------------+ | ... (unmapped) ... | 0x20000000 +---------------------------+ | SRAM (8 KB) | variables, stack 0x08000000 +---------------------------+ | Flash (128 KB) | your code + constants 0x00000000 +---------------------------+

Três marcos que vale a pena memorizar (RM0041 §3 / PM0056):

RegiãoEndereço baseContém
Periféricos APB10x40000000TIM2–4, USART2/3, I2C1/2, SPI2
Periféricos APB20x40010000GPIOA–D, USART1, SPI1, ADC1, AFIO

Núcleo Cortex-M30xE000E000NVIC, SysTick (parte do CPU, não do fabricante do chip)

Cada periférico ocupa uma fatia fixa deste espaço, e dentro dessa fatia cada registo fica num deslocamento (offset) fixo a partir da base do periférico. Por exemplo, o porto GPIO C tem base 0x40011000, e o seu registo de dados de saída ODR está no offset 0x0C, ou seja, no endereço absoluto 0x4001100C. O manual de referência (RM0041) é a autoridade para cada base e cada offset.

2. Porque é que volatile é obrigatório

O compilador C assume que variáveis vulgares se comportam como memória: se lês o mesmo endereço duas vezes sem nada pelo meio, o valor não pode ter mudado, logo é livre de ler uma vez e reutilizar a cópia em cache. Também é livre de descartar uma escrita que é imediatamente sobreposta, ou de reordenar leituras/escritas independentes. Essas otimizações são corretas para RAM e catastróficas para registos de hardware, cujo valor pode mudar por si só e em que cada acesso tem um efeito secundário.

O qualificador volatile diz ao compilador: esta localização pode mudar a qualquer momento, cada acesso é observável, faz exatamente os acessos que escrevi, por ordem, sem caching, sem eliminar, sem reordenar relativamente a outros acessos volatile.

⚠ O clássico bug do ciclo infinito Supõe que sondas (poll) uma flag de "dados prontos" num registo de estado. Sem volatile o compilador lê o registo uma vez, vê que o bit é 0 e — porque nada no ciclo o altera — transforma a tua sondagem em while (1) { }. O hardware põe o bit a 1 um microssegundo depois, mas o teu programa nunca mais olha. Fica preso para sempre. 
/* BROKEN: plain pointer, no volatile */
uint32_t *sr = (uint32_t *)0x40013800U;   /* USART1_SR */
while ((*sr & (1U << 5)) == 0U) { }        /* wait for RXNE */
/* compiler hoists the read out of the loop -> spins on a stale 0 forever */
A correção é uma única palavra-chave: 
/* CORRECT: volatile forces a real bus read every iteration */
volatile uint32_t *sr = (volatile uint32_t *)0x40013800U;
while ((*sr & (1U << 5)) == 0U) { }        /* re-reads hardware each pass */
O mesmo raciocínio aplica-se às escritas: sem volatile um compilador pode descartar uma escrita que "nunca voltas a ler", mesmo que essa escrita seja exatamente o que comuta um bit de hardware.

Regra para este curso: todo o ponteiro que dá nome a um registo de hardware é volatile. Usamos tipos de largura fixa (uint32_t) para que a largura do acesso coincida exatamente com a do registo de 32 bits.

3. Duas formas de dar nome a um registo em C

3.1 Idioma A — a macro #define

A abordagem mais simples: converte (cast) o endereço literal para um volatile uint32_t * e dereferencia-o. Envolve-o numa macro para que o nome se leia como uma variável.

#define GPIOC_ODR  (*(volatile uint32_t *)0x4001100CU)

GPIOC_ODR = 0x200U;      /* writes the 32-bit register */
uint32_t v = GPIOC_ODR;  /* reads it back               */

Lê-o de dentro para fora: (volatile uint32_t *)0x4001100CU é "um ponteiro para um valor volatile de 32 bits que vive nesse endereço"; o * à frente dereferencia-o, de modo que a macro é o registo, utilizável de qualquer dos lados do =. É exatamente isto que o Módulo 04 usa. Serve bem para um ou dois registos, mas com sete registos por porto GPIO e quatro portos torna-se numa parede de endereços fixos no código, fáceis de escrever mal.

3.2 Idioma B — a sobreposição com struct volatile

Os registos de um periférico são palavras consecutivas de 32 bits em offsets conhecidos — que é exatamente a disposição em memória de uma struct C de campos uint32_t. Por isso declaramos uma struct cujos campos, por ordem, espelham o mapa de registos, e depois apontamo-la para o endereço base do periférico:

typedef struct {
    volatile uint32_t CRL;    /* offset 0x00 */
    volatile uint32_t CRH;    /* offset 0x04 */
    volatile uint32_t IDR;    /* offset 0x08 */
    volatile uint32_t ODR;    /* offset 0x0C */
    volatile uint32_t BSRR;   /* offset 0x10 */
    volatile uint32_t BRR;    /* offset 0x14 */
    volatile uint32_t LCKR;   /* offset 0x18 */
} GPIO_TypeDef;

#define GPIOC ((GPIO_TypeDef *)0x40011000U)

GPIOC->ODR  = 0x200U;     /* address = base + 0x0C automatically */
GPIOC->BSRR = (1U << 9);  /* address = base + 0x10               */

Como cada uint32_t tem 4 bytes e o compilador dispõe os campos de forma contígua sem padding (todos os campos são do mesmo tipo de 4 bytes naturalmente alinhado), os offsets dos campos saem como 0x00, 0x04, 0x08, … — e o compilador calcula base + offset por ti. Escreves GPIOC->BSRR; o compilador emite um store para 0x40011010.

⚠ A struct tem de coincidir com o mapa de hardware exatamente A struct só funciona porque a ordem e o tamanho dos campos reproduzem o mapa de registos byte a byte. Se omitires um registo, trocares a ordem de dois, ou usares a largura errada, cada campo a seguir ao erro cai no endereço errado e fala silenciosamente com o hardware errado. Não há nenhuma verificação do compilador — a struct é apenas uma interpretação sobreposta a endereços em bruto. Quando um mapa de registos tem uma lacuna (um endereço reservado sem registo), tens de inserir um campo de padding explícito, p. ex. volatile uint32_t RESERVED0;, para manter os offsets seguintes alinhados. (O GPIO não tem lacunas; vais ver padding mais tarde nos temporizadores/DMA.)

Porque é que a struct escala melhor: um endereço base em vez de sete literais; nomes de campos verificados pelo compilador (uma gralha como GPIOC->OOR não compila, ao passo que um número errado numa macro compila alegremente); a mesma GPIO_TypeDef descreve todos os portos — GPIOA, GPIOB, GPIOC, GPIOD são apenas quatro ponteiros para o único tipo. É precisamente assim que os cabeçalhos CMSIS dos fabricantes são construídos. Deste módulo em diante preferimos o idioma da struct.

4. Idiomas de manipulação de bits

Os registos empacotam muitos controlos independentes numa só palavra de 32 bits, por isso raramente escreves um registo inteiro — alteras bits específicos deixando os restantes intactos. Estes cinco idiomas cobrem quase tudo no curso. R é um registo, n uma posição de bit.

ObjetivoCódigoPorquê
Pôr bit n a 1R |= (1U << n);OR com um 1 força esse bit a 1, os outros bits inalterados.
Limpar bit nR &= ~(1U << n);AND com um 0 nessa posição força-o a 0, os outros mantêm-se (o ~ cria uma máscara de tudo a 1 exceto o bit n).
Comutar bit nR ^= (1U << n);XOR com 1 inverte o bit; XOR com 0 mantém-no.
Testar bit n((R >> n) & 1U)Desloca o bit para a posição 0, mascara tudo o resto → 0 ou 1.
Escrever um campo de múltiplos bitsR = (R & ~(MASK << s)) | (VAL << s);Read-modify-write: limpa o campo com a máscara invertida, depois faz OR do novo valor deslocado para a posição.

O último — limpar um campo antes de fazer OR com um novo valor — é essencial porque um simples R |= (VAL << s) só consegue pôr bits a 1; não consegue voltar a pôr a 0 um 1 antigo. Para um campo de 4 bits (como o nibble CNF/MODE de um pino GPIO) a máscara é 0xFU:

/* set the 4-bit field for pin 9 (in CRH) to value 0x2, untouched elsewhere */
uint32_t shift = (9U - 8U) * 4U;            /* = 4 */
GPIOC->CRH = (GPIOC->CRH & ~(0xFU << shift)) /* clear old nibble  */
           | (0x2U << shift);               /* OR in new value   */
Estilo alinhado com MISRA (usado em todo o curso)
  • Sufixa sempre os literais unsigned com U: 1U, 0xFU. Um simples 1 é um int com sinal; 1 << 31 é comportamento indefinido, enquanto 1U << 31 é bem definido.
  • Usa tipos de largura fixa de <stdint.h> (uint32_t, uint8_t) — nunca um simples int para valores de registos.
  • Prefere máscaras/deslocamentos com nome a números mágicos, e faz sempre read-modify-write para nunca destruíres bits não relacionados.

5. RCC: clock gating como exemplo real de MMIO

Eis MMIO posto em ação. Para poupar energia, cada periférico STM32 arranca com o seu clock desligado, e um periférico sem clock ignora todas as escritas em registos — o clássico bug do "o meu código não faz nada". O bloco RCC (Reset and Clock Control) em 0x40021000 contém os bits de ativação. Para usar o porto GPIO C, primeiro pões a 1 o seu bit de ativação de clock em RCC_APB2ENR (offset 0x18):

RegistoEndereçoBitSignificado
RCC_APB2ENR0x40021018bit 4 — IOPCENAtivação de clock para o porto GPIO C

(bit 2 = IOPAEN porto A, bit 3 = IOPBEN porto B, bit 4 = IOPCEN porto C, bit 5 = IOPDEN porto D — RM0041 §7.3.7.)

#define RCC_APB2ENR (*(volatile uint32_t *)0x40021018U)
RCC_APB2ENR |= (1U << 4);   /* IOPCEN: turn on the port C clock */

Isto é exatamente o idioma "pôr bit n a 1" do §4, aplicado a um registo real. Vais escrever esta linha no topo de cada programa GPIO do curso.

6.3 Dicas

Dica 1 — dispor a struct

Cada registo é uma palavra de 32 bits, por isso cada campo é volatile uint32_t. Lista-os de cima para baixo na mesma ordem em que os offsets aumentam. O primeiro campo fica no offset 0, o seguinte no +4, e assim por diante automaticamente:

typedef struct {
    volatile uint32_t CRL;   /* 0x00 */
    volatile uint32_t CRH;   /* 0x04 */
    volatile uint32_t IDR;   /* 0x08 */
    volatile uint32_t ODR;   /* 0x0C */
    volatile uint32_t BSRR;  /* 0x10 */
    volatile uint32_t BRR;   /* 0x14 */
    volatile uint32_t LCKR;  /* 0x18 */
} GPIO_TypeDef;
Dica 2 — o ponteiro GPIOC

Converte (cast) o endereço base para o teu tipo ponteiro-para-struct, de modo que GPIOC->FIELD resolva para base + offset. O sufixo U mantém o literal unsigned:

#define GPIOC  ((GPIO_TypeDef *)0x40011000U)
Dica 3 — ativar o clock (TODO 3)

Read-modify-write com OR para não perturbares os outros bits de ativação:

Dica 4 — configurar PC9 em CRH (TODO 4)

O nibble do pino 9 está no shift (9 - 8) * 4 = 4. Limpa-o com a máscara de 4 bits invertida, depois faz OR de 0x2 (MODE=10 → saída 2 MHz, CNF=00 → push-pull), tudo deslocado para a posição:

GPIOC->CRH = (GPIOC->CRH & ~(0xFU << shift))      /* clear nibble */
           | (0x2U << shift);                    /* push-pull, 2 MHz */

Perguntas de Verificação

7. Verifica a tua compreensão

P1. Porque é que um ponteiro de registo tem de ser volatile? Indica o modo de falha sem ele.

Um registo de hardware pode mudar por si só (o hardware põe/limpa bits) e cada acesso tem um efeito secundário. Sem volatile o compilador trata-o como memória vulgar: pode pôr uma leitura em cache e reutilizar o valor antigo, descartar uma escrita que julga redundante, ou reordenar acessos. A falha clássica é sondar uma flag de estado num ciclo — o compilador retira a leitura para fora do ciclo, fica preso num 0 antigo, e o programa bloqueia para sempre mesmo depois de o hardware pôr o bit a 1. O volatile força um acesso real ao barramento, por ordem, em cada leitura e escrita.

P2. Como é que os offsets dos campos de uma struct mapeiam para endereços de registos?

O compilador dispõe os campos da struct de forma contígua. Com sete campos volatile uint32_t (cada um de 4 bytes, naturalmente alinhado, sem padding) os offsets são 0x00, 0x04, 0x08, 0x0C, 0x10, 0x14, 0x18. Quando o ponteiro da struct contém o endereço base do periférico, aceder a um campo calcula base + offset: p. ex. GPIOC->BSRR com base 0x40011000 e BSRR no offset 0x10 resolve para o endereço 0x40011010. É por isto que a ordem e o tamanho dos campos têm de espelhar o mapa de hardware exatamente — e porque lacunas no mapa precisam de campos de padding explícitos.

P3. Qual é a diferença entre escrever em ODR e escrever em BSRR?

ODR é o registo completo de dados de saída de 16 bits; alterar um pino significa um read-modify-write de ODR (lê-o, inverte um bit, escreve-o de volta), que pode entrar em corrida (race) com uma interrupção que modifique outro pino do mesmo porto e perder essa alteração. BSRR é só de escrita e atómico: uma única escrita de 32 bits põe a 1 os pinos cujo bit escreveste 1 na metade baixa (bits 0–15) e limpa os pinos cujo bit escreveste 1 na metade alta (bits 16–31); os bits escritos a 0 ficam intactos. Assim, BSRR altera exatamente os pinos que nomeias, numa só escrita de barramento, sem leitura e sem corrida.

P4. Porque é que os periféricos arrancam com os seus clocks desligados (gated off)?

Poupança de energia. Um bloco digital com clock consome corrente dinâmica em cada flanco de clock, por isso deixar os periféricos não usados sem clock corta o consumo — importante nos microcontroladores de baixo consumo que este design visa. O efeito secundário para o programador: um periférico sem clock ignora todas as escritas em registos, por isso tens de ativar o seu clock no RCC (p. ex. pôr IOPCEN a 1 em RCC_APB2ENR para o porto GPIO C) antes de o configurares. Esquecer isto é o bug mais comum do "o meu código corre mas nada acontece".

P5. O porto GPIO B tem base 0x40010C00. Calcula o endereço de GPIOB_ODR.

ODR está no offset 0x0C no mapa de registos do GPIO. Logo GPIOB_ODR = 0x40010C00 + 0x0C = 0x40010C0C. Com o idioma da struct bastava escreveres GPIOB->ODR depois de definir #define GPIOB ((GPIO_TypeDef *)0x40010C00U), e o compilador produz esse mesmo endereço por ti.
