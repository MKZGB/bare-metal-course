Module 03 Memory-Mapped Registers (MMIO)

How peripheral hardware appears as memory addresses, why volatile is non-negotiable, and how to model a register block as a C struct. This is the foundation every later module reuses.

What this module gives you No LED, no UART, no visible output yet — this is a design module. You will learn the two ways to name a hardware register in C, the bit-manipulation idioms used everywhere in this course, and you will hand-write a small register-map header for GPIO port C. Modules 04 onward assume you are fluent with everything here.
Theory

1. What "memory-mapped I/O" means

A Cortex-M3 has a single, flat 4 GB address space (addresses 0x00000000–0xFFFFFFFF). Some of those addresses are real memory — Flash and SRAM. But large regions of the map are not memory at all: they are wired directly to peripheral hardware. Reading or writing one of those addresses does not store a value; it pokes a circuit.

This is memory-mapped I/O (MMIO). There is no separate "I/O instruction" on ARM (unlike x86 IN/OUT). To turn on an LED you write to an address; to read a button you read an address. A register is just a named 32-bit location inside a peripheral that controls or reports something.

STM32F100 address space (the parts that matter) 0xFFFFFFFF +---------------------------+ | ... | 0xE000E000 | Cortex-M3 core periph. | NVIC, SysTick (PM0056) | ... | 0x40018000 +---------------------------+ | AHB peripherals | RCC @0x40021000, DMA @0x40020000 0x40010000 +---------------------------+ | APB2 peripherals | GPIOA..D, USART1, SPI1, ADC, AFIO 0x40000000 +---------------------------+ | APB1 peripherals | TIM2..4, USART2/3, I2C, SPI2 +---------------------------+ | ... (unmapped) ... | 0x20000000 +---------------------------+ | SRAM (8 KB) | variables, stack 0x08000000 +---------------------------+ | Flash (128 KB) | your code + constants 0x00000000 +---------------------------+

Three landmarks worth memorizing (RM0041 §3 / PM0056):

RegionBase addressHolds
APB1 peripherals0x40000000TIM2–4, USART2/3, I2C1/2, SPI2
APB2 peripherals0x40010000GPIOA–D, USART1, SPI1, ADC1, AFIO
AHB peripherals0x40018000+RCC (0x40021000), DMA1 (0x40020000)

Each peripheral occupies a fixed slice of this space, and inside that slice each register sits at a fixed offset from the peripheral's base. For example GPIO port C has base 0x40011000, and its output-data register ODR is at offset 0x0C, i.e. absolute address 0x4001100C. The reference manual (RM0041) is the authority for every base and offset.

2. Why volatile is mandatory

The C compiler assumes ordinary variables behave like memory: if you read the same address twice with nothing in between, the value cannot have changed, so it is free to read once and reuse the cached copy. It is also free to drop a write that is immediately overwritten, or to reorder independent reads/writes. Those optimizations are correct for RAM and catastrophic for hardware registers, whose value can change on its own and whose every access has a side effect.

The volatile qualifier tells the compiler: this location may change at any time, every access is observable, do exactly the accesses I wrote, in order, no caching, no eliding, no reordering relative to other volatile accesses.

⚠ The classic infinite-loop bug Suppose you poll a "data ready" flag in a status register. Without volatile the compiler reads the register once, sees the bit is 0, and — because nothing in the loop changes it — turns your poll into while (1) { }. The hardware sets the bit a microsecond later, but your program never looks again. It hangs forever. 
/* BROKEN: plain pointer, no volatile */
uint32_t *sr = (uint32_t *)0x40013800U;   /* USART1_SR */
while ((*sr & (1U << 5)) == 0U) { }        /* wait for RXNE */
/* compiler hoists the read out of the loop -> spins on a stale 0 forever */
The fix is one keyword: 
/* CORRECT: volatile forces a real bus read every iteration */
volatile uint32_t *sr = (volatile uint32_t *)0x40013800U;
while ((*sr & (1U << 5)) == 0U) { }        /* re-reads hardware each pass */
The same reasoning applies to writes: without volatile a compiler may drop a write you "never read back," even though that write is exactly what flips a hardware bit.

Rule for this course: every pointer that names a hardware register is volatile. We use fixed-width types (uint32_t) so the access width matches the 32-bit register exactly.

3. Two ways to name a register in C

3.1 Idiom A — the #define macro

The simplest approach: cast the literal address to a volatile uint32_t * and dereference it. Wrap it in a macro so the name reads like a variable.

#define GPIOC_ODR  (*(volatile uint32_t *)0x4001100CU)

GPIOC_ODR = 0x200U;      /* writes the 32-bit register */
uint32_t v = GPIOC_ODR;  /* reads it back               */

Read it inside-out: (volatile uint32_t *)0x4001100CU is "a pointer to a volatile 32-bit value living at that address"; the leading * dereferences it so the macro is the register, usable on either side of =. This is exactly what Module 04 uses. It is fine for one or two registers, but with seven registers per GPIO port and four ports it becomes a wall of hard-coded addresses that are easy to mistype.

3.2 Idiom B — the volatile struct overlay

A peripheral's registers are consecutive 32-bit words at known offsets — which is exactly the memory layout of a C struct of uint32_t fields. So we declare a struct whose fields, in order, mirror the register map, then point it at the peripheral's base address:

typedef struct {
    volatile uint32_t CRL;    /* offset 0x00 */
    volatile uint32_t CRH;    /* offset 0x04 */
    volatile uint32_t IDR;    /* offset 0x08 */
    volatile uint32_t ODR;    /* offset 0x0C */
    volatile uint32_t BSRR;   /* offset 0x10 */
    volatile uint32_t BRR;    /* offset 0x14 */
    volatile uint32_t LCKR;   /* offset 0x18 */
} GPIO_TypeDef;

#define GPIOC ((GPIO_TypeDef *)0x40011000U)

GPIOC->ODR  = 0x200U;     /* address = base + 0x0C automatically */
GPIOC->BSRR = (1U << 9);  /* address = base + 0x10               */

Because each uint32_t is 4 bytes and the compiler lays the fields out contiguously with no padding (all fields are the same naturally-aligned 4-byte type), field offsets become 0x00, 0x04, 0x08, … — and the compiler computes base + offset for you. You write GPIOC->BSRR; the compiler emits a store to 0x40011010.

⚠ The struct must match the hardware map exactly The struct only works because field order and size reproduce the register map byte-for-byte. If you omit a register, reorder two, or use the wrong width, every field after the mistake lands on the wrong address and silently talks to the wrong hardware. There is no compiler check — the struct is just an interpretation laid over raw addresses. When a register map has a gap (a reserved address with no register), you must insert an explicit padding field, e.g. volatile uint32_t RESERVED0;, to keep later offsets aligned. (GPIO has no gaps; you will see padding later in timers/DMA.)

Why the struct scales better: one base address instead of seven literals; field names checked by the compiler (a typo like GPIOC->OOR fails to compile, whereas a wrong number in a macro compiles happily); the same GPIO_TypeDef describes every port — GPIOA, GPIOB, GPIOC, GPIOD are just four pointers to the one type. This is precisely how vendor CMSIS headers are built. From this module on we prefer the struct idiom.

4. Bit-manipulation idioms

Registers pack many independent controls into one 32-bit word, so you rarely write a whole register — you change specific bits while leaving the rest alone. These five idioms cover almost everything in the course. R is a register, n a bit position.

GoalCodeWhy
Set bit nR |= (1U << n);OR with a 1 forces that bit to 1, other bits unchanged.
Clear bit nR &= ~(1U << n);AND with a 0 in that position forces it to 0, others kept (the ~ makes a mask of all-1s except bit n).
Toggle bit nR ^= (1U << n);XOR with 1 flips the bit; XOR with 0 keeps it.
Test bit n((R >> n) & 1U)Shift the bit down to position 0, mask off everything else → 0 or 1.
Write a multi-bit fieldR = (R & ~(MASK << s)) | (VAL << s);Read-modify-write: clear the field with the inverted mask, then OR in the new value shifted into place.

The last one — clearing a field before OR-ing in a new value — is essential because a bare R |= (VAL << s) can only ever set bits to 1; it cannot turn a stale 1 back to 0. For a 4-bit field (like a GPIO pin's CNF/MODE nibble) the mask is 0xFU:

/* set the 4-bit field for pin 9 (in CRH) to value 0x2, untouched elsewhere */
uint32_t shift = (9U - 8U) * 4U;            /* = 4 */
GPIOC->CRH = (GPIOC->CRH & ~(0xFU << shift)) /* clear old nibble  */
           | (0x2U << shift);               /* OR in new value   */
MISRA-aligned style (used throughout)
  • Always suffix unsigned literals with U: 1U, 0xFU. A bare 1 is a signed int; 1 << 31 is undefined behavior, while 1U << 31 is well-defined.
  • Use fixed-width types from <stdint.h> (uint32_t, uint8_t) — never bare int for register values.
  • Prefer named masks/shifts over magic numbers, and always read-modify-write so you never clobber unrelated bits.

5. RCC: clock-gating as a real MMIO example

Here is MMIO put to work. To save power, every STM32 peripheral powers up with its clock switched off, and a clock-less peripheral ignores all register writes — the textbook "my code does nothing" bug. The RCC (Reset and Clock Control) block at 0x40021000 contains the enable bits. To use GPIO port C you first set its clock-enable bit in RCC_APB2ENR (offset 0x18):

RegisterAddressBitMeaning
RCC_APB2ENR0x40021018bit 4 — IOPCENClock enable for GPIO port C

(bit 2 = IOPAEN port A, bit 3 = IOPBEN port B, bit 4 = IOPCEN port C, bit 5 = IOPDEN port D — RM0041 §7.3.7.)

#define RCC_APB2ENR (*(volatile uint32_t *)0x40021018U)
RCC_APB2ENR |= (1U << 4);   /* IOPCEN: turn on the port C clock */

This is exactly the "set bit n" idiom from §4, applied to a real register. You will write this line at the top of every GPIO program in the course.

6.3 Hints

Hint 1 — laying out the struct

Each register is one 32-bit word, so each field is volatile uint32_t. List them top-to-bottom in the same order the offsets increase. The first field sits at offset 0, the next at +4, and so on automatically:

typedef struct {
    volatile uint32_t CRL;   /* 0x00 */
    volatile uint32_t CRH;   /* 0x04 */
    volatile uint32_t IDR;   /* 0x08 */
    volatile uint32_t ODR;   /* 0x0C */
    volatile uint32_t BSRR;  /* 0x10 */
    volatile uint32_t BRR;   /* 0x14 */
    volatile uint32_t LCKR;  /* 0x18 */
} GPIO_TypeDef;
Hint 2 — the GPIOC pointer

Verification Questions

7. Check your understanding

Q1. Why must a register pointer be volatile? Give the failure mode without it.

A hardware register can change on its own (hardware sets/clears bits) and every access has a side effect. Without volatile the compiler treats it like ordinary memory: it may cache a read and reuse the stale value, drop a write it thinks is redundant, or reorder accesses. The classic failure is polling a status flag in a loop — the compiler hoists the read out of the loop, spins on a stale 0, and the program hangs forever even though the hardware set the bit. volatile forces a real bus access, in order, for every read and write.

Q2. How do a struct's field offsets map onto register addresses?

The compiler lays out struct fields contiguously. With seven volatile uint32_t fields (each 4 bytes, naturally aligned, no padding) the offsets are 0x00, 0x04, 0x08, 0x0C, 0x10, 0x14, 0x18. When the struct pointer holds the peripheral base address, accessing a field computes base + offset: e.g. GPIOC->BSRR with base 0x40011000 and BSRR at offset 0x10 resolves to address 0x40011010. This is why field order and size must mirror the hardware map exactly — and why gaps in the map need explicit padding fields.

Q3. What is the difference between writing ODR and writing BSRR?

ODR is the full 16-bit output-data register; changing one pin means a read-modify-write of ODR (read it, flip one bit, write it back), which can race with an interrupt that modifies another pin in the same port and lose that change. BSRR is write-only and atomic: a single 32-bit write sets the pins whose bit you wrote 1 to in the low half (bits 0–15) and clears the pins whose bit you wrote 1 to in the high half (bits 16–31); bits written 0 are left untouched. So BSRR changes exactly the pins you name, in one bus write, with no read and no race.

Q4. Why do peripherals power up with their clocks gated off?

Power saving. A clocked digital block draws dynamic current on every clock edge, so leaving unused peripherals unclocked cuts power consumption — important on the low-power microcontrollers this design targets. The side effect for the programmer: a clock-gated peripheral ignores all register writes, so you must enable its clock in the RCC (e.g. set IOPCEN in RCC_APB2ENR for GPIO port C) before configuring it. Forgetting this is the most common "my code runs but nothing happens" bug.

Q5. GPIO port B has base 0x40010C00. Compute the address of GPIOB_ODR.

ODR is at offset 0x0C in the GPIO register map. So GPIOB_ODR = 0x40010C00 + 0x0C = 0x40010C0C. With the struct idiom you would just write GPIOB->ODR after defining #define GPIOB ((GPIO_TypeDef *)0x40010C00U), and the compiler produces that same address for you.
