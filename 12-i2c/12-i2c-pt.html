Módulo 12 Driver I2C — A Máquina de Estados do Mestre

Coloca o I2C1 a funcionar ao nível dos registos (CR1/CR2/CCR/TRISE) e percorre a sequência de transmissão do mestre — START → SB → endereço → ADDR → dados → BTF → STOP — passo a passo. Narramos cada passo através da UART porque o QEMU não emula I2C.

Dependências Este módulo reutiliza o driver UART por sondagem (polled) e o redireccionamento do printf do Módulo 06 para narrar a sequência I2C, e o padrão de espera limitada introduzido para periféricos não emulados no Módulo 11 (DMA). A configuração dos pinos GPIO (CRL/CRH) vem do Módulo 04.
Teoria

1. O que é o I2C

O I2C (Inter-Integrated Circuit) é um barramento série de dois fios e múltiplos pontos (multi-drop). Duas linhas open-drain são partilhadas por todos os dispositivos:

  • SCL — o relógio, sempre conduzido pelo mestre.
  • SDA — dados bidireccionais.

Como as linhas são open-drain com resistências de pull-up externas, um dispositivo só pode puxar uma linha para baixo; os pull-ups levam-na para nível alto quando ninguém a conduz. É por isto que os pinos do STM32 têm de ser configurados como função alternada open-drain (não push-pull). Na STM32VLDISCOVERY, o I2C1 usa SCL = PB6 e SDA = PB7.

1.1 Mestres, escravos e endereçamento

  • Um mestre inicia e termina cada transacção e fornece o relógio. Um escravo responde quando é endereçado.
  • Cada escravo tem um endereço de 7 bits. O mestre envia esse endereço no primeiro byte, deslocado um bit para a esquerda, sendo o bit menos significativo a flag R/W (0 = escrita, 1 = leitura). Assim, uma escrita no endereço 0x53 coloca (0x53 << 1) | 0 = 0xA6 no barramento.
  • Após cada byte, o receptor puxa SDA para baixo durante um ciclo de relógio para ACK (confirmar). Deixá-la em nível alto é um NACK (sem dispositivo, ou "para de enviar").

1.2 Os quatro eventos do barramento

START : o mestre puxa SDA para baixo com SCL em nivel alto -> "estou a tomar o barramento" STOP : o mestre liberta SDA para alto com SCL em nivel alto -> "o barramento esta livre" ACK : o receptor puxa SDA para baixo no 9o ciclo -> "byte recebido" NACK : o receptor deixa SDA em nivel alto no 9o ciclo -> "nenhum/ultimo byte" Clock stretching: um escravo que precisa de tempo mantem SCL em nivel BAIXO. O mestre tem de esperar ate SCL ser libertada antes de continuar - o hardware faz isto por ti, mas e por isto que uma transferencia pode pausar a meio de um byte.

2. A máquina de estados dos registos do I2C1

Não fazes bit-bang das linhas. O periférico executa o protocolo; tu conduzes uma pequena máquina de estados escrevendo bits de controlo e sondando flags de estado. (RM0041 §26.)

2.1 Configuração de temporização (relógios)

O I2C1 reside no APB1, com relógio a 8 MHz nesta placa (HSI, sem PLL). Três números configuram o barramento:

  • CR2.FREQ indica ao periférico o seu relógio de entrada em MHz para que possa derivar a temporização interna. FREQ = 8.
  • CCR divide esse relógio até ao período de SCL. Para o modo padrão de 100 kHz: CCR = PCLK1 / (2 × SCL) = 8 000 000 / (2 × 100 000) = 40.
  • TRISE limita o tempo de subida: (1000 ns / TPCLK1) + 1 = (1000 / 125) + 1 = 9.

2.2 A sequência de transmissão do mestre

Este é o coração do módulo. Cada passo escreve um bit de controlo ou DR e depois espera por uma flag de estado antes do seguinte:

1. CR1.PE = 1 activa o periferico 2. CR1.START = 1 pede uma condicao de START 3. espera SR1.SB == 1 START esta na linha 4. DR = (addr << 1) | 0 envia endereco de 7 bits + bit de escrita 5. espera SR1.ADDR == 1 o escravo fez ACK ao seu endereco 6. le SR1, depois le SR2 <-- isto LIMPA a flag ADDR 7. DR = data carrega um byte de dados 8. espera SR1.TxE / SR1.BTF byte enviado 9. CR1.STOP = 1 liberta o barramento

O passo 6 é o famoso truque: ADDR é limpa lendo SR1 e depois SR2, por essa ordem. Até o fazeres, SCL fica esticada em nível baixo e a transferência está parada. Este estranho padrão de "ler dois registos para limpar uma flag" existe para que o hardware possa apresentar ambas as palavras de estado de forma atómica; vê as perguntas de verificação.

⚠ O QEMU NÃO emula I2C — e não há dispositivo escravo

O qemu-system-arm -M stm32vldiscovery emula nenhum periférico I2C e não há nenhum dispositivo no barramento (o acelerómetro ADXL345 usado como exemplo abaixo não está presente). Concretamente:

  • as escritas nos registos I2C são silenciosamente descartadas;
  • as leituras de SR1/SR2 devolvem 0;
  • SB, ADDR, BTF, TxE nunca são activadas.

Por isso a sequência não consegue passar do passo 3 no QEMU — e isso é esperado. O objectivo deste módulo é compreender a máquina de estados dos registos, não ler um sensor real. Um ingénuo while(!(SR1 & SB)){} faria o programa bloquear para sempre. TENS de usar uma espera limitada em cada sondagem de flag e imprimir até que passo chegaste.

A regra da espera limitada Nunca escrevas um while(!(REG & FLAG)){} ilimitado para uma flag que o QEMU nunca activa. Usa um contador de guarda que decrementa e, em caso de timeout, imprime uma explicação e continua: 
uint32_t guard = 200000U;
while (((I2C1_SR1 & I2C_SR1_SB) == 0U) && (guard > 0U)) { guard--; }
if (guard == 0U) { printf("SB not set - I2C not emulated in QEMU; correct on hardware\n"); }
Em hardware real a flag activa rapidamente e o ciclo sai cedo — o mesmo código está correcto nesse caso.

make -f ../common/Makefile SRC=main.c run

Perguntas de Verificação

4. Verifica a tua compreensão

P1. O ADXL345 tem o endereço de 7 bits 0x53. Que byte vai para o barramento numa escrita, e o que é ACK/NACK?

O byte de endereço é o endereço de 7 bits deslocado um bit para a esquerda com o bit R/W acrescentado: para uma escrita, (0x53 << 1) | 0 = 0xA6. Depois de o mestre enviar esses 8 bits, o escravo endereçado puxa SDA para baixo no 9.º ciclo para ACK ("existo e estou à escuta"). Se nenhum dispositivo corresponder ao endereço, SDA mantém-se em nível alto — um NACK — e em hardware o mestre abortaria. No QEMU não há escravo, por isso nada faz ACK.

P2. Porque é que tens de ler SR1 e depois SR2 para limpar a flag ADDR?

O I2C do STM32F1 limpa ADDR apenas numa leitura de SR1 seguida de uma leitura de SR2 (RM0041 §26.6.3). A leitura em dois passos existe porque o hardware expõe estado relacionado (por exemplo, papel mestre/escravo, BUSY) entre os dois registos e quer que sejam amostrados em conjunto. Até ADDR ser limpa, o periférico mantém SCL em nível baixo (clock stretch), por isso esquecer este passo bloqueia a transferência. Ler apenas SR1 não é suficiente.

P3. O que configura o campo FREQ em CR2, e porquê 8 aqui?

CR2.FREQ[5:0] indica ao periférico I2C a frequência do seu relógio de entrada (APB1 / PCLK1) em MHz. O hardware usa isto para temporizar eventos internos como o setup/hold dos dados. Não define a frequência de SCL — é o CCR que faz isso. Nesta placa o APB1 corre a 8 MHz (HSI, sem PLL), por isso FREQ = 8. Defini-lo errado torna a temporização derivada do barramento incorrecta em hardware real.

P4. O que fazem realmente START e STOP nas linhas, e quem os gera?

O mestre gera ambos. Um START é SDA a descer enquanto SCL está em nível alto — toma o barramento e (no STM32) activa SR1.SB. Um STOP é SDA a subir enquanto SCL está em nível alto — liberta o barramento para que outro mestre o possa usar. Entre um START e um STOP, o mestre é dono do barramento. Definir CR1.START pede o START; definir CR1.STOP pede o STOP.

P5. Porque é que este driver não consegue concluir no QEMU, e como o impedimos de bloquear?

A máquina stm32vldiscovery não emula qualquer periférico I2C e não tem escravo no barramento, por isso os registos I2C lêem 0 e SB/ADDR/BTF/TxE nunca são activadas. A primeira espera de flag (por SB após o START) nunca pode, portanto, ter sucesso. Um ingénuo while(!(SR1 & SB)){} rodaria para sempre. Usamos uma espera limitada: um contador de guarda que decrementa a cada iteração para que o ciclo saia sempre, após o que imprimimos que passo foi atingido e abortamos de forma limpa. Em hardware real a flag activa em poucas iterações, por isso o mesmo código corre correctamente e conclui a transferência.
