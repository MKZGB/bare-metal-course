Module 12 I2C Driver — The Master State Machine

Bring up I2C1 at the register level (CR1/CR2/CCR/TRISE) and walk the master-transmit sequence — START → SB → address → ADDR → data → BTF → STOP — step by step. We narrate every step over UART because QEMU does not emulate I2C.

Dependencies This module reuses the polled UART driver and the printf retarget from Module 06 to narrate the I2C sequence, and the bounded-wait pattern introduced for unemulated peripherals in Module 11 (DMA). GPIO pin configuration (CRL/CRH) comes from Module 04.
Theory

1. What I2C is

I2C (Inter-Integrated Circuit) is a two-wire, multi-drop serial bus. Two open-drain lines are shared by every device:

  • SCL — the clock, always driven by the master.
  • SDA — bidirectional data.

Because the lines are open-drain with external pull-up resistors, a device can only ever pull a line low; the pull-ups bring it high when no one drives it. This is why the STM32 pins must be configured as alternate-function open-drain (not push-pull). On the STM32VLDISCOVERY, I2C1 uses SCL = PB6 and SDA = PB7.

1.1 Masters, slaves, and addressing

  • A master starts and ends every transaction and supplies the clock. A slave responds when addressed.
  • Each slave has a 7-bit address. The master sends that address in the first byte, shifted left one bit, with the lowest bit being the R/W flag (0 = write, 1 = read). So a write to address 0x53 puts (0x53 << 1) | 0 = 0xA6 on the bus.
  • After each byte the receiver pulls SDA low for one clock to ACK (acknowledge). Leaving it high is a NACK (no device, or "stop sending").

1.2 The four bus events

START : master pulls SDA low while SCL is high -> "I am taking the bus" STOP : master releases SDA high while SCL is high -> "bus is free again" ACK : receiver pulls SDA low on the 9th clock -> "byte received" NACK : receiver leaves SDA high on the 9th clock -> "no/last byte" Clock stretching: a slave that needs time holds SCL LOW. The master must wait until SCL is released before continuing - the hardware does this for you, but it is why a transfer can pause mid-byte.

2. The I2C1 register state machine

You do not bit-bang the wires. The peripheral runs the protocol; you drive a small state machine by writing control bits and polling status flags. (RM0041 §26.)

2.1 Timing setup (clocks)

I2C1 lives on APB1, clocked at 8 MHz on this board (HSI, no PLL). Three numbers configure the bus:

  • CR2.FREQ tells the peripheral its input clock in MHz so it can derive internal timing. FREQ = 8.
  • CCR divides that clock down to the SCL period. For 100 kHz standard mode: CCR = PCLK1 / (2 × SCL) = 8 000 000 / (2 × 100 000) = 40.
  • TRISE bounds the rise time: (1000 ns / TPCLK1) + 1 = (1000 / 125) + 1 = 9.

2.2 The master-transmit sequence

This is the heart of the module. Each step writes a control bit or DR, then waits for a status flag before the next:

1. CR1.PE = 1 enable the peripheral 2. CR1.START = 1 request a START condition 3. wait SR1.SB == 1 START is on the wire 4. DR = (addr << 1) | 0 send 7-bit address + write bit 5. wait SR1.ADDR == 1 slave ACKed its address 6. read SR1, then read SR2 <-- this CLEARS the ADDR flag 7. DR = data load a data byte 8. wait SR1.TxE / SR1.BTF byte shifted out 9. CR1.STOP = 1 release the bus

Step 6 is the famous trick: ADDR is cleared by reading SR1 and then SR2, in that order. Until you do, SCL is stretched low and the transfer is stalled. This odd "read two registers to clear a flag" pattern exists so the hardware can present both status words atomically; see the verification questions.

⚠ QEMU does NOT emulate I2C — and there is no slave device

qemu-system-arm -M stm32vldiscovery emulates no I2C peripheral, and there is no device on the bus (the ADXL345 accelerometer used as the example below is not present). Concretely:

  • writes to the I2C registers are silently dropped;
  • reads of SR1/SR2 return 0;
  • SB, ADDR, BTF, TxE never set.

So the sequence cannot get past step 3 in QEMU — and that is expected. The goal of this module is to understand the register state machine, not to read a real sensor. A naïve while(!(SR1 & SB)){} would hang the program forever. You MUST use a bounded wait at every flag poll and print which step you reached.

The bounded-wait rule Never write an unbounded while(!(REG & FLAG)){} for a flag QEMU never sets. Use a guard counter that decrements, and on timeout print an explanation and continue: 
uint32_t guard = 200000U;
while (((I2C1_SR1 & I2C_SR1_SB) == 0U) && (guard > 0U)) { guard--; }
if (guard == 0U) { printf("SB not set - I2C not emulated in QEMU; correct on hardware\n"); }
On real hardware the flag sets quickly and the loop exits early — the same code is correct there.

Verification Questions

4. Check your understanding

Q1. The ADXL345 has 7-bit address 0x53. What byte goes on the bus for a write, and what is ACK/NACK?

The address byte is the 7-bit address shifted left one bit with the R/W bit appended: for a write, (0x53 << 1) | 0 = 0xA6. After the master sends those 8 bits, the addressed slave pulls SDA low for the 9th clock to ACK ("I exist and I'm listening"). If no device matches the address, SDA stays high — a NACK — and on hardware the master would abort. In QEMU there is no slave, so nothing ever ACKs.

Q2. Why must you read SR1 and then SR2 to clear the ADDR flag?

The STM32F1 I2C clears ADDR only on a read of SR1 followed by a read of SR2 (RM0041 §26.6.3). The two-step read exists because the hardware exposes related status (e.g. master/slave role, BUSY) across the two registers and wants them sampled together. Until ADDR is cleared the peripheral holds SCL low (clock stretch), so forgetting this step stalls the transfer. Reading just SR1 is not enough.

Q3. What does the FREQ field in CR2 configure, and why 8 here?

CR2.FREQ[5:0] tells the I2C peripheral the frequency of its input clock (APB1 / PCLK1) in MHz. The hardware uses this to time internal events such as data setup/hold. It does not set the SCL frequency — CCR does that. On this board APB1 runs at 8 MHz (HSI, no PLL), so FREQ = 8. Setting it wrong makes the derived bus timing incorrect on real hardware.

Q4. What do START and STOP actually do on the wires, and who generates them?

The master generates both. A START is SDA falling while SCL is high — it seizes the bus and (in the STM32) sets SR1.SB. A STOP is SDA rising while SCL is high — it releases the bus so another master may use it. Between a START and a STOP the master owns the bus. Setting CR1.START requests the START; setting CR1.STOP requests the STOP.

Q5. Why can't this driver complete in QEMU, and how do we keep it from hanging?

The stm32vldiscovery machine emulates no I2C peripheral and has no slave on the bus, so the I2C registers read 0 and SB/ADDR/BTF/TxE never set. The very first flag wait (for SB after START) can therefore never succeed. A naïve while(!(SR1 & SB)){} would spin forever. We use a bounded wait: a guard counter that decrements every iteration so the loop always exits, after which we print which step was reached and abort cleanly. On real hardware the flag sets within a few iterations, so the same code runs correctly and completes the transfer.
