Module 11 DMA — Moving Bytes Without the CPU

Use DMA1 Channel 4 to stream a string straight into USART1_DR — the data mover runs while the CPU is free. Contrast this with the byte-at-a-time polled TX of Module 06.

Dependencies This module builds directly on Module 06 (UART Transmit) (the polled uart_putc driver and the printf retarget) and on Module 10 (Interrupts / NVIC) (the transfer-complete interrupt is wired the same way). Make sure you are comfortable with both.
Theory

1. What DMA is, and why you want it

In Module 06 the CPU transmitted a string by spinning on the TXE flag and writing every single byte into USART1_DR itself:

Module 06 (CPU-polled TX): for each byte: CPU: while (!(USART1_SR & TXE)) { spin } <- CPU burns cycles waiting CPU: USART1_DR = byte <- CPU does the copy The CPU is the data mover. It can do nothing else meanwhile.

Direct Memory Access is a separate hardware engine that moves data between memory and a peripheral (or memory↔memory) without the CPU copying each byte. You tell it three things — where from, where to, and how many — flip a switch, and it does the rest, raising an interrupt when finished.

Module 11 (DMA TX): CPU (once): set source, destination, count, enable channel DMA engine: byte0 -> USART1_DR, byte1 -> USART1_DR, ... <- no CPU DMA engine: raises "transfer complete" when count hits 0 The DMA controller is the data mover. The CPU is free.

This matters for high-throughput or latency-sensitive work (audio, ADC sampling, long UART/SPI bursts): the CPU sets it up and walks away.

1.1 Channels and DMA requests

DMA1 on the STM32F100 has 7 channels. A channel is one independent transfer "stream" with its own source, destination, and count. On the STM32F1 each peripheral DMA request line is hard-wired to a fixed channel — you do not get to choose. From RM0041 Table 78, USART1_TX is mapped to DMA1 Channel 4. That is why this module uses channel 4 specifically.

1.2 The per-channel register block

DMA1 lives at base 0x40020000 (it is on the AHB bus, so its clock enable is RCC_AHBENR bit 0, not an APB register). It has two shared status registers followed by a block of four configuration registers per channel:

RegisterOffsetRole
DMA_ISR0x00Interrupt status flags (4 per channel)
DMA_IFCR0x04Clear flags (write 1 to clear)
CCRx0x08 + (x−1)·0x14Channel configuration
CNDTRxCCRx + 0x04Number of data items to transfer
CPARxCCRx + 0x08Peripheral address
CMARxCCRx + 0x0CMemory address

Each channel's block is 0x14 (20) bytes wide, starting at 0x08 + (channel − 1) × 0x14. For channel 4:

channel base = 0x08 + (4 - 1) * 0x14 = 0x08 + 0x3C = 0x44 CCR4 = 0x40020000 + 0x44 = 0x40020044 (config) CNDTR4 = 0x40020000 + 0x48 = 0x40020048 (count) CPAR4 = 0x40020000 + 0x4C = 0x4002004C (peripheral addr) CMAR4 = 0x40020000 + 0x50 = 0x40020050 (memory addr)

1.3 The CCRx configuration bits

BitNameMeaning
0ENChannel enable. Set this LAST. CPAR/CMAR/CNDTR may only be written while EN = 0.
1TCIETransfer-complete interrupt enable (fires the NVIC IRQ — see §1.5).
4DIRDirection. 1 = read from memory (memory → peripheral). For TX we set it.
5CIRCCircular mode (auto-reload count and restart). Off for a one-shot send.
7MINCMemory increment: bump the memory pointer after each item so we walk the buffer.
9:8PSIZEPeripheral data size (00 = 8-bit). Leave 00 for byte UART.
11:10MSIZEMemory data size (00 = 8-bit). Leave 00.
13:12PLChannel priority. Leave 00 (low) — only one channel is active here.
DIR and MINC: the two settings that matter most DIR=1 says the source is memory and the destination is the peripheral (CPAR). MINC=1 says "after each byte, advance the memory pointer" so successive bytes come from successive buffer addresses. We deliberately leave peripheral increment (PINC, bit 6) at 0, because USART1_DR is a single fixed register — every byte goes to the same address.

1.4 Wiring it to USART1

The DMA controller only moves a byte when the peripheral asks for one. USART1 must be told to raise a DMA request whenever its TX register is empty. That is the DMAT bit (bit 7) in USART1_CR3 (0x40013814). The destination address for the channel is the USART data register, USART1_DR = 0x40013804.

Setup order for a memory -> USART1_TX transfer: 1. RCC_AHBENR |= DMA1EN enable the DMA1 clock 2. CCR4.EN = 0 disable channel before configuring 3. CPAR4 = 0x40013804 dest = USART1_DR 4. CMAR4 = &buffer src = our string in RAM 5. CNDTR4 = length how many bytes 6. CCR4 = DIR | MINC mem->periph, walk the buffer 7. USART1_CR3 |= DMAT let USART1 issue DMA requests 8. CCR4.EN = 1 GO

1.5 Knowing when it is done

The shared DMA_ISR packs four flags per channel. For channel 4 they sit in bits [15:12]: GIF4(12), TCIF4(13), HTIF4(14), TEIF4(15). TCIF4 (transfer complete, bit 13) sets when CNDTR4 reaches 0. You either poll it or, with CCR4.TCIE set, take the DMA1_Channel4 interrupt (IRQ 14) via the NVIC (Module 10). You clear the flag by writing 1 to the matching bit in DMA_IFCR.

⚠ QEMU does NOT emulate DMA — read this before you write a wait loop Per the verified emulation matrix, qemu-system-arm -M stm32vldiscovery does not emulate the DMA controller at all. No transfer ever happens: the bytes in your buffer never reach USART1_DR, the DMA-sent string does not appear on the console, and TCIF4 is NEVER set.

Therefore a naive while (!(DMA_ISR & TCIF4)) { } would hang forever in QEMU. You must use a bounded wait (a guard counter) so the program completes and you can print an explanation. On real STM32 hardware the very same register code transmits the string correctly and TCIF4 sets.
How we still get something observable Since the DMA path produces no visible output in QEMU, main() first prints a banner using the CPU-polled uart_putc/printf from Module 06 (USART1 itself is fully emulated). It explains exactly what registers were programmed, then configures DMA, then bounded-waits on TCIF4 and prints the QEMU-limitation note. So you see the program run to completion and reason about the DMA setup, without depending on a transfer QEMU can't perform.
Guided Exercise

2. Transmit a string with DMA1 Channel 4

Goal: reuse the Module 06 polled driver for an explanatory banner, then set up DMA1 Channel 4 to stream a static string into USART1_DR, and finish with a bounded wait on TCIF4. You will:

  1. Enable the DMA1 clock in RCC_AHBENR (bit 0, the AHB bus).
  2. Program CPAR4 (= USART1_DR), CMAR4 (= buffer), CNDTR4 (= length).
  3. Set CCR4 bits DIR and MINC, then enable USART1 DMAT, then set CCR4.EN.
  4. Bounded-wait on TCIF4 and print the outcome (it will time out in QEMU — expected).

#define USART_CR3_DMAT (1U << 7)
#define USART1_DR_ADDR (0x40013804U)   /* DMA destination */

/* ---- DMA1 (base 0x40020000) ---- */

#define DMA1_IFCR     (*(volatile uint32_t *)0x40020004U)

{
    uint32_t guard;

    uart_init();
    (void)setvbuf(stdout, NULL, _IONBF, 0);

    printf("\r\n=== Module 11: DMA Driver (UART TX) ===\n");
    printf("This banner is sent by the CPU (polled), like Module 06.\n");

    len = (uint16_t)(sizeof(dma_message) - 1U);

From inside the 11-dma/ folder:

You should see the banner and the register summary, then "transfer did not complete - DMA not emulated by QEMU", then === done ===. Crucially the program does not hang — the bounded wait expires and execution continues. Quit QEMU with Ctrl-A then X. Success criterion: builds with no warnings, prints the banner, and reaches the idle loop without freezing.

2.3 Hints

Hint 1 — computing the channel-4 register addresses (TODO A)

Channel base offset is 0x08 + (4−1)·0x14 = 0x44. So: CCR4 @ 0x40020044, CNDTR4 @ 0x40020048, CPAR4 @ 0x4002004C, CMAR4 @ 0x40020050. For example: #define DMA1_CCR4 (*(volatile uint32_t *)0x40020044U) and the other three at +4/+8/+0xC.

Hint 2 — addresses and count (TODO B/C/D)

DMA1_CPAR4 = USART1_DR_ADDR; (the fixed peripheral register). DMA1_CMAR4 = (uint32_t)buf; (the start of your string). DMA1_CNDTR4 = (uint32_t)len; (bytes to move — exclude the terminating NUL).

Hint 3 — direction / increment and the start sequence (TODO E/F/G)

Write the config in one go: DMA1_CCR4 = (DMA_CCR_DIR | DMA_CCR_MINC); — DIR=1 means memory→peripheral, MINC=1 walks the buffer; leave PSIZE/MSIZE/PL at 0 (8-bit, low priority), and do NOT set PINC since USART1_DR is fixed. Then USART1_CR3 |= USART_CR3_DMAT; and finally DMA1_CCR4 |= DMA_CCR_EN;. Order matters: configure everything while EN=0, set EN last.

Hint 4 — why the bounded wait, not a normal flag poll

QEMU never sets TCIF4, so while(!(DMA1_ISR & DMA_ISR_TCIF4)){} spins forever. The guard counter forces the loop to exit after a fixed number of iterations so the program completes and prints the QEMU note. On real hardware the flag sets first and the loop exits early.

Verification Questions

3. Check your understanding

Q1. How does DMA offload the CPU compared with Module 06's polled TX?

In Module 06 the CPU was the data mover: for every byte it spun on TXE and wrote USART1_DR itself, so it could do nothing else during the transfer. With DMA the CPU configures source/destination/count once and enables the channel; the DMA engine then copies each byte (triggered by the USART's TX-empty DMA request) with zero CPU involvement, and only signals the CPU via TCIF4/the IRQ when finished. The CPU is free to run other code meanwhile.

Q2. Compute the four channel-4 register addresses from the DMA1 base.

Channel base offset = 0x08 + (4−1)·0x14 = 0x08 + 0x3C = 0x44. Adding the DMA1 base 0x40020000: CCR4 = 0x40020044, CNDTR4 = 0x40020048, CPAR4 = 0x4002004C, CMAR4 = 0x40020050. The shared DMA_ISR (0x40020000) and DMA_IFCR (0x40020004) sit before the channel blocks.

Q3. What are the roles of the DIR and MINC bits for a UART TX transfer?

DIR=1 sets the transfer direction to "read from memory" — source = memory (CMAR), destination = peripheral (CPAR). MINC=1 increments the memory address after each item so successive bytes are fetched from successive buffer locations. We leave peripheral increment (PINC) off because every byte is written to the same fixed register, USART1_DR.

Q4. Which flag signals completion, where does it live, and how do you clear it?

TCIF4 (transfer complete for channel 4) — bit 13 of the shared DMA_ISR (channel-4 flags occupy bits [15:12]). It sets when CNDTR4 counts down to 0. You clear it by writing 1 to the corresponding bit in DMA_IFCR (CTCIF4, bit 13). If CCR4.TCIE is set, the same event also raises the NVIC DMA1_Channel4 interrupt, IRQ 14.

Q5. You run it in QEMU and the DMA string never appears, and a plain while(!(ISR&TCIF4)){} hangs. Why, and what's the fix?

QEMU's stm32vldiscovery machine does not emulate the DMA controller, so no transfer occurs, the bytes never reach USART1_DR, and TCIF4 is never set — the unbounded loop would wait forever. The fix is a bounded wait: a guard counter that forces the loop to exit, after which you print an explanation. The register code is still correct and works on real STM32 hardware. (This is the same "QEMU doesn't emulate it" honesty as the GPIO, timer, and I2C modules.)
