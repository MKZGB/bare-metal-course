Módulo 11 DMA — Mover Bytes Sem o CPU

Usa o DMA1 Canal 4 para transmitir uma string diretamente para USART1_DR — o motor de transferência de dados funciona enquanto o CPU está livre. Contrasta isto com a TX por polling, byte a byte, do Módulo 06.

Dependências Este módulo assenta diretamente no Módulo 06 (Transmissão UART) (o driver uart_putc por polling e o redirecionamento do printf) e no Módulo 10 (Interrupções / NVIC) (a interrupção de transferência completa é ligada da mesma forma). Certifica-te de que estás à vontade com ambos.
Teoria

1. O que é o DMA, e porque o queres

No Módulo 06 o CPU transmitia uma string fazendo polling à flag TXE e escrevendo ele próprio cada byte em USART1_DR:

Módulo 06 (TX por polling do CPU): para cada byte: CPU: while (!(USART1_SR & TXE)) { spin } <- o CPU gasta ciclos à espera CPU: USART1_DR = byte <- o CPU faz a cópia O CPU é o motor de transferência de dados. Não pode fazer mais nada entretanto.

Direct Memory Access (Acesso Direto à Memória) é um motor de hardware separado que move dados entre a memória e um periférico (ou memória↔memória) sem que o CPU copie cada byte. Dizes-lhe três coisas — de onde, para onde e quantos — acionas um interruptor, e ele faz o resto, gerando uma interrupção quando terminar.

Módulo 11 (TX por DMA): CPU (uma vez): define origem, destino, contagem, ativa o canal motor DMA: byte0 -> USART1_DR, byte1 -> USART1_DR, ... <- sem CPU motor DMA: gera "transferência completa" quando a contagem chega a 0 O controlador DMA é o motor de transferência de dados. O CPU está livre.

Isto importa para trabalho de alto débito ou sensível à latência (áudio, amostragem de ADC, rajadas longas de UART/SPI): o CPU configura e afasta-se.

1.1 Canais e pedidos de DMA

O DMA1 no STM32F100 tem 7 canais. Um canal é um "fluxo" de transferência independente com a sua própria origem, destino e contagem. No STM32F1 cada linha de pedido de DMA de um periférico está fisicamente ligada a um canal fixo — não podes escolher. Pela Tabela 78 do RM0041, USART1_TX está mapeado para o DMA1 Canal 4. É por isso que este módulo usa especificamente o canal 4.

1.2 O bloco de registos por canal

O DMA1 reside na base 0x40020000 (está no barramento AHB, por isso o seu clock enable é o bit 0 do RCC_AHBENR, não um registo APB). Tem dois registos de estado partilhados seguidos de um bloco de quatro registos de configuração por canal:

RegistoOffsetFunção
DMA_ISR0x00Flags de estado de interrupção (4 por canal)
DMA_IFCR0x04Limpar flags (escrever 1 para limpar)
CCRx0x08 + (x−1)·0x14Configuração do canal
CNDTRxCCRx + 0x04Número de itens de dados a transferir
CPARxCCRx + 0x08Endereço do periférico
CMARxCCRx + 0x0CEndereço de memória

O bloco de cada canal tem 0x14 (20) bytes de largura, começando em 0x08 + (canal − 1) × 0x14. Para o canal 4:

base do canal = 0x08 + (4 - 1) * 0x14 = 0x08 + 0x3C = 0x44 CCR4 = 0x40020000 + 0x44 = 0x40020044 (config) CNDTR4 = 0x40020000 + 0x48 = 0x40020048 (contagem) CPAR4 = 0x40020000 + 0x4C = 0x4002004C (endereço do periférico) CMAR4 = 0x40020000 + 0x50 = 0x40020050 (endereço de memória)

1.3 Os bits de configuração do CCRx

BitNomeSignificado
0ENAtivação do canal. Define este por ÚLTIMO. CPAR/CMAR/CNDTR só podem ser escritos enquanto EN = 0.
1TCIEAtivação da interrupção de transferência completa (dispara a IRQ do NVIC — ver §1.5).
4DIRDireção. 1 = ler a partir da memória (memória → periférico). Para TX definimo-lo.
5CIRCModo circular (recarrega a contagem e reinicia). Desligado para um envio único.
7MINCIncremento de memória: avança o ponteiro de memória após cada item, para percorrermos o buffer.
9:8PSIZETamanho dos dados do periférico (00 = 8 bits). Deixa 00 para UART de bytes.
11:10MSIZETamanho dos dados de memória (00 = 8 bits). Deixa 00.
13:12PLPrioridade do canal. Deixa 00 (baixa) — só um canal está ativo aqui.
DIR e MINC: as duas definições que mais importam DIR=1 diz que a origem é a memória e o destino é o periférico (CPAR). MINC=1 diz "após cada byte, avança o ponteiro de memória" para que bytes sucessivos venham de endereços sucessivos do buffer. Deixamos deliberadamente o incremento do periférico (PINC, bit 6) a 0, porque USART1_DR é um único registo fixo — todos os bytes vão para o mesmo endereço.

1.4 Ligá-lo ao USART1

O controlador DMA só move um byte quando o periférico o pede. É preciso dizer ao USART1 para gerar um pedido de DMA sempre que o seu registo de TX estiver vazio. Esse é o bit DMAT (bit 7) em USART1_CR3 (0x40013814). O endereço de destino do canal é o registo de dados do USART, USART1_DR = 0x40013804.

Ordem de configuração para uma transferência memória -> USART1_TX: 1. RCC_AHBENR |= DMA1EN ativa o clock do DMA1 2. CCR4.EN = 0 desativa o canal antes de configurar 3. CPAR4 = 0x40013804 destino = USART1_DR 4. CMAR4 = &buffer origem = a nossa string na RAM 5. CNDTR4 = length quantos bytes 6. CCR4 = DIR | MINC mem->periph, percorre o buffer 7. USART1_CR3 |= DMAT deixa o USART1 emitir pedidos de DMA 8. CCR4.EN = 1 ARRANCA

1.5 Saber quando terminou

O DMA_ISR partilhado agrupa quatro flags por canal. Para o canal 4 estão nos bits [15:12]: GIF4(12), TCIF4(13), HTIF4(14), TEIF4(15). TCIF4 (transferência completa, bit 13) ativa-se quando CNDTR4 chega a 0. Ou fazes polling a ela ou, com CCR4.TCIE definido, atendes a interrupção DMA1_Channel4 (IRQ 14) via NVIC (Módulo 10). Limpas a flag escrevendo 1 no bit correspondente em DMA_IFCR.

⚠ O QEMU NÃO emula o DMA — lê isto antes de escreveres um ciclo de espera Segundo a matriz de emulação verificada, o qemu-system-arm -M stm32vldiscovery não emula de todo o controlador DMA. Nenhuma transferência acontece: os bytes no teu buffer nunca chegam a USART1_DR, a string enviada por DMA não aparece na consola, e TCIF4 NUNCA é ativada.

Por isso um ingénuo while (!(DMA_ISR & TCIF4)) { } bloquearia para sempre no QEMU. Tens de usar uma espera limitada (um contador de guarda) para que o programa termine e possas imprimir uma explicação. Em hardware STM32 real, este mesmo código de registos transmite a string corretamente e o TCIF4 ativa-se.
Como mesmo assim obtemos algo observável Como o caminho do DMA não produz saída visível no QEMU, o main() imprime primeiro um banner usando o uart_putc/printf por polling do CPU do Módulo 06 (o próprio USART1 é totalmente emulado). Explica exatamente que registos foram programados, depois configura o DMA, depois faz a espera limitada em TCIF4 e imprime a nota da limitação do QEMU. Assim vês o programa correr até ao fim e raciocinas sobre a configuração do DMA, sem depender de uma transferência que o QEMU não consegue executar.
Exercício Guiado

2. Transmitir uma string com o DMA1 Canal 4

Objetivo: reutilizar o driver por polling do Módulo 06 para um banner explicativo, depois configurar o DMA1 Canal 4 para transmitir uma string estática para USART1_DR, e terminar com uma espera limitada em TCIF4. Vais:

  1. Ativar o clock do DMA1 em RCC_AHBENR (bit 0, o barramento AHB).
  2. Programar CPAR4 (= USART1_DR), CMAR4 (= buffer), CNDTR4 (= comprimento).
  3. Definir os bits DIR e MINC do CCR4, depois ativar o DMAT do USART1, depois definir CCR4.EN.
  4. Fazer espera limitada em TCIF4 e imprimir o resultado (vai expirar no QEMU — esperado).

2.1 Esqueleto — cria 11-dma/main.c

Preenche cada TODO. O bloco por polling uart_init / uart_putc / _write / syscall-stub é idêntico ao do Módulo 06 — copia o teu. Apenas as partes do DMA são mostradas abaixo.

#include <stdint.h>
#include <stdio.h>
#include <sys/stat.h>

/* --- Reutilizado do Módulo 06: uart_init(), uart_putc(), _write(), e os
 *     stubs de syscall da newlib (_read/_close/_lseek/_isatty/_fstat).
 *     Dão-nos um printf funcional pelo caminho de TX por polling do CPU. --- */

/* ---- RCC ---- */
#define RCC_AHBENR    (*(volatile uint32_t *)0x40021014U)
#define RCC_DMA1EN    (1U << 0)

/* ---- USART1 (CR3.DMAT deixa o USART1 gerar pedidos de DMA) ---- */
#define USART1_CR3     (*(volatile uint32_t *)0x40013814U)
#define USART_CR3_DMAT (1U << 7)
#define USART1_DR_ADDR (0x40013804U)   /* destino do DMA */

/* ---- DMA1 (base 0x40020000) ---- */
#define DMA1_ISR      (*(volatile uint32_t *)0x40020000U)
#define DMA1_IFCR     (*(volatile uint32_t *)0x40020004U)
/* TODO A: define CCR4, CNDTR4, CPAR4, CMAR4 a partir da base do canal 4.
 *         base do canal = 0x08 + (4-1)*0x14 = 0x44, por isso CCR4 está em
 *         0x40020000 + 0x44. Os três seguintes seguem em +4, +8, +0xC. */

#define DMA_CCR_EN    (1U << 0)
#define DMA_CCR_DIR   (1U << 4)   /* 1 = ler a partir da memória  */
#define DMA_CCR_MINC  (1U << 7)   /* incremento de memória        */
#define DMA_ISR_TCIF4   (1U << 13)
#define DMA_IFCR_CTCIF4 (1U << 13)

static const char dma_message[] = "Hello via DMA1 Channel 4 -> USART1_TX\r\n";

static void dma_uart_tx_start(const char *buf, uint16_t len)
{
    /* 1. Ativa o clock do DMA1 (barramento AHB). */
    RCC_AHBENR |= RCC_DMA1EN;

    /* 2. Desativa o canal antes de (re)configurá-lo. */
    DMA1_CCR4 &= ~DMA_CCR_EN;

    /* TODO B: CPAR4 = o endereço do registo de dados do USART1 (USART1_DR_ADDR). */

    /* TODO C: CMAR4 = o endereço do buffer (faz cast de buf para uint32_t). */

    /* TODO D: CNDTR4 = o número de bytes a enviar (len). */

    /* TODO E: CCR4 = DIR | MINC  (mem->periph, incrementa o ponteiro de memória). */

    /* TODO F: define o bit DMAT do USART1_CR3 para o USART1 emitir pedidos de DMA. */

    /* TODO G: finalmente define CCR4.EN para iniciar a transferência. */
}

int main(void)
{
    uint32_t guard;
    uint16_t len;

    uart_init();
    (void)setvbuf(stdout, NULL, _IONBF, 0);

    printf("\r\n=== Module 11: DMA Driver (UART TX) ===\n");
    printf("This banner is sent by the CPU (polled), like Module 06.\n");
    printf("Now configuring DMA1 Channel 4...\n");

    len = (uint16_t)(sizeof(dma_message) - 1U);
    dma_uart_tx_start(dma_message, len);

    /* Espera LIMITADA: TCIF4 nunca ativa no QEMU, por isso um while() simples
     * BLOQUEARIA. A guarda faz o programa terminar para podermos explicar o sucedido. */
    guard = 1000000U;
    while (((DMA1_ISR & DMA_ISR_TCIF4) == 0U) && (guard > 0U))
    {
        guard--;
    }

    if (guard == 0U)
    {
        printf("Result: transfer did not complete - DMA not emulated by QEMU;\n");
        printf("        correct on hardware.\n");
    }
    else
    {
        DMA1_IFCR = DMA_IFCR_CTCIF4;   /* limpa a flag em HW real */
        printf("Result: DMA transfer complete (TCIF4 set).\n");
    }

    for (;;) { }
    return 0;
}

2.2 Compilar & correr

De dentro da pasta 11-dma/:

make -f ../common/Makefile SRC=main.c run

Deverás ver o banner e o resumo dos registos, depois "transfer did not complete - DMA not emulated by QEMU", depois === done ===. Crucialmente, o programa não bloqueia — a espera limitada expira e a execução continua. Sai do QEMU com Ctrl-A seguido de X. Critério de sucesso: compila sem avisos, imprime o banner e chega ao ciclo de espera sem congelar.

2.3 Pistas

Pista 1 — calcular os endereços dos registos do canal 4 (TODO A)

O offset da base do canal é 0x08 + (4−1)·0x14 = 0x44. Portanto: CCR4 @ 0x40020044, CNDTR4 @ 0x40020048, CPAR4 @ 0x4002004C, CMAR4 @ 0x40020050. Por exemplo: #define DMA1_CCR4 (*(volatile uint32_t *)0x40020044U) e os outros três em +4/+8/+0xC.

Pista 2 — endereços e contagem (TODO B/C/D)

DMA1_CPAR4 = USART1_DR_ADDR; (o registo fixo do periférico). DMA1_CMAR4 = (uint32_t)buf; (o início da tua string). DMA1_CNDTR4 = (uint32_t)len; (bytes a mover — exclui o NUL terminador).

Pista 3 — direção / incremento e a sequência de arranque (TODO E/F/G)

Escreve a configuração de uma só vez: DMA1_CCR4 = (DMA_CCR_DIR | DMA_CCR_MINC); — DIR=1 significa memória→periférico, MINC=1 percorre o buffer; deixa PSIZE/MSIZE/PL a 0 (8 bits, prioridade baixa) e NÃO definas PINC, já que USART1_DR é fixo. Depois USART1_CR3 |= USART_CR3_DMAT; e finalmente DMA1_CCR4 |= DMA_CCR_EN;. A ordem importa: configura tudo enquanto EN=0, define EN por último.

Pista 4 — porquê a espera limitada, e não um polling normal de flag

O QEMU nunca ativa TCIF4, por isso while(!(DMA1_ISR & DMA_ISR_TCIF4)){} gira para sempre. O contador de guarda força o ciclo a sair após um número fixo de iterações, para que o programa termine e imprima a nota do QEMU. Em hardware real a flag ativa-se primeiro e o ciclo sai mais cedo.

Perguntas de Verificação

3. Verifica a tua compreensão

Q1. Como é que o DMA alivia o CPU em comparação com a TX por polling do Módulo 06?

No Módulo 06 o CPU era o motor de transferência de dados: para cada byte fazia polling a TXE e escrevia ele próprio USART1_DR, por isso não podia fazer mais nada durante a transferência. Com o DMA, o CPU configura origem/destino/contagem uma vez e ativa o canal; o motor DMA copia depois cada byte (acionado pelo pedido de DMA de TX-vazia do USART) sem qualquer envolvimento do CPU, e só sinaliza o CPU via TCIF4/a IRQ quando termina. O CPU fica livre para correr outro código entretanto.

Q2. Calcula os quatro endereços de registo do canal 4 a partir da base do DMA1.

Offset da base do canal = 0x08 + (4−1)·0x14 = 0x08 + 0x3C = 0x44. Somando a base do DMA1 0x40020000: CCR4 = 0x40020044, CNDTR4 = 0x40020048, CPAR4 = 0x4002004C, CMAR4 = 0x40020050. O DMA_ISR partilhado (0x40020000) e o DMA_IFCR (0x40020004) ficam antes dos blocos dos canais.

Q3. Quais são as funções dos bits DIR e MINC numa transferência de TX por UART?

DIR=1 define a direção da transferência para "ler a partir da memória" — origem = memória (CMAR), destino = periférico (CPAR). MINC=1 incrementa o endereço de memória após cada item, para que bytes sucessivos sejam lidos de localizações sucessivas do buffer. Deixamos o incremento do periférico (PINC) desligado porque cada byte é escrito no mesmo registo fixo, USART1_DR.

Q4. Que flag sinaliza a conclusão, onde reside, e como a limpas?

TCIF4 (transferência completa do canal 4) — bit 13 do DMA_ISR partilhado (as flags do canal 4 ocupam os bits [15:12]). Ativa-se quando CNDTR4 conta até 0. Limpa-la escrevendo 1 no bit correspondente em DMA_IFCR (CTCIF4, bit 13). Se CCR4.TCIE estiver definido, o mesmo evento também gera a interrupção DMA1_Channel4 do NVIC, IRQ 14.

Q5. Corre-lo no QEMU e a string do DMA nunca aparece, e um while(!(ISR&TCIF4)){} simples bloqueia. Porquê, e qual é a correção?

A máquina stm32vldiscovery do QEMU não emula o controlador DMA, por isso nenhuma transferência ocorre, os bytes nunca chegam a USART1_DR, e TCIF4 nunca é ativada — o ciclo ilimitado esperaria para sempre. A correção é uma espera limitada: um contador de guarda que força o ciclo a sair, após o qual imprimes uma explicação. O código de registos continua correto e funciona em hardware STM32 real. (É a mesma honestidade do "o QEMU não o emula" dos módulos de GPIO, timer e I2C.)
