Módulo 02 O Modelo de Programação ARM

O núcleo Cortex-M3 tal como o programador o vê: o ficheiro de registos, os registos de estado do programa, os modos de operação e a sequência de reset que transforma um chip frio em código C em execução — percorrido através do startup.c e do linker.ld reais que este curso disponibiliza.

O que é este módulo Este é um módulo teórico — não há esqueleto para preencher nem ficheiro solutions/. Em vez disso, vais ler sobre o núcleo e depois executar uma breve investigação contra um ELF compilado real para ver a maquinaria de reset com os teus próprios olhos. Tudo o que está aqui é a base de todos os módulos seguintes: quando o Módulo 04 diz "main() executa", isto é como lá chegou.
Teoria

1. O ficheiro de registos do núcleo (R0–R15)

O Cortex-M3 é uma máquina load/store: o CPU opera sobre registos, e só instruções dedicadas de load/store movem dados de e para a memória. Existem dezasseis registos de núcleo de 32 bits visíveis em qualquer instante. Nem todos são iguais — três deles têm uma função fixa no hardware.

RegistoFunçãoNotas
R0–R12Uso geralAuxiliares / variáveis. A ABI do C passa os primeiros 4 argumentos inteiros em R0–R3 e devolve o resultado em R0.
R13 = SPStack PointerBanked: existem fisicamente dois — MSP (Main) e PSP (Process). Apenas um é "o SP" de cada vez (ver §3).
R14 = LRLink RegisterUm BL (branch-with-link) guarda aqui o endereço de retorno. À entrada de uma exceção, o CPU escreve um valor especial EXC_RETURN em LR.
R15 = PCProgram CounterEndereço da instrução a ser carregada. Escrever nele é um salto. Lê-se sempre com o bit 0 = 0, embora as instruções tenham um bit Thumb (ver §5).

1.1 A pilha é full-descending

A pilha do Cortex-M3 cresce para baixo (em direção a endereços menores) e o SP aponta para o último item empilhado — isto é uma pilha full-descending. Um PUSH decrementa primeiro o SP em 4 e depois guarda; um POP carrega e depois incrementa. É exatamente por isso que o linker define o stack pointer inicial para o topo da RAM: o primeiro push move-o para baixo, para o espaço livre.

RAM (8 KB) 0x20002000 <- _estack (initial MSP, top of RAM) | PUSH r0 -> SP -= 4, then store stack grows DOWN ===========> | ... v 0x20000000 <- bottom of RAM (.data / .bss live here)

2. Estado do programa: o xPSR

As flags de condição, o número da exceção atual e o bit de estado de execução vivem todos num único registo combinado de 32 bits, o xPSR. Na verdade são três vistas lógicas do mesmo registo que podes ler/escrever separadamente:

VistaContémCampos principais
APSREstado da aplicação — as flags aritméticasN (negativo), Z (zero), C (carry), V (overflow) e Q (saturação)
IPSREstado de interrupção — qual exceção está em execuçãoNúmero da exceção. 0 = modo Thread (sem exceção). Dentro de SysTick_Handler lê-se 15; dentro de uma IRQ n lê-se 16 + n.
EPSREstado de execuçãoO bit T (estado Thumb) e o estado de continuação da instrução interrompida. T tem de ser 1 num Cortex-M3 — só executa código Thumb.
xPSR (bit 31 .................................................. bit 0) |31 30 29 28 27| ... |26 ........ 16| ... |8 ..... 0| | N Z C V Q| | ICI/IT, T | | IPSR # | \___APSR_____/ \___EPSR_____/ \__IPSR_/

As flags N/Z/C/V são a forma como as comparações em C funcionam: if (a == b) compila para um CMP que define Z, seguido de um branch condicional que o testa. Raramente mexes no xPSR à mão em C — mas tens de compreender o IPSR para saber, dentro de um handler, qual interrupção disparou (abordado nos módulos de interrupções), e o bit T é a razão de um detalhe subtil que verás em §5.

3. Modos de operação e as duas pilhas

O Cortex-M3 tem exatamente dois modos e dois níveis de privilégio:

EixoEstadoSignificado
ModoThreadCódigo normal — o teu main() e tudo o que ele chama. Entrado no reset.
HandlerA executar um handler de exceção/interrupção. Entrado automaticamente quando uma exceção é tomada; sempre privilegiado.
PrivilégioPrivilegiadoAcesso total a todos os registos especiais e ao System Control Space (NVIC, SysTick, etc.).
Não privilegiadoRestrito; um RTOS pode executar aqui as threads da aplicação. Não consegue alcançar os registos de controlo do núcleo.

Qual stack pointer físico está ativo depende do modo e de um bit de controlo:

  • O modo Handler usa sempre o MSP. Os handlers de exceção executam na pilha principal.
  • O modo Thread usa o MSP por omissão, mas pode ser comutado para PSP através do bit SPSEL do registo CONTROL.
Porque existem duas pilhas Um RTOS dá a cada tarefa a sua própria pilha via PSP, enquanto os handlers de interrupção executam numa MSP separada controlada pelo kernel — assim uma ISR nunca pode fazer overflow da pilha de uma tarefa, e uma tarefa com pilha esgotada não pode corromper o kernel. Este curso bare-metal nunca comuta para PSP: vivemos inteiramente em modo Thread privilegiado sobre o MSP. Mas deves saber que o PSP existe, porque a primeira coisa que o hardware faz no reset é carregar o MSP (§4).

4. Do chip frio ao main(): reset e a tabela de vetores

Não há BIOS nem bootloader a fazer a configuração por ti. Quando o STM32F100 sai do reset, o hardware do Cortex-M3 faz exatamente duas coisas, ambas lendo a tabela de vetores que está logo no início da Flash, no endereço 0x08000000:

  1. Carrega o MSP inicial a partir da palavra de 32 bits em 0x08000000 (entrada de vetor 0).
  2. Carrega o PC a partir da palavra em 0x08000004 (entrada de vetor 1) e começa a executar aí — esse endereço é o nosso Reset_Handler.

Esse é todo o contributo do hardware. A partir do Reset_Handler em diante, cada byte de configuração é software que nós escrevemos. (RM0041 §10; Cortex-M3 PM0056.)

4.1 A tabela de vetores em common/startup.c

A tabela é apenas um array C colocado em 0x08000000 pelo linker. O detalhe crucial: a entrada 0 não é um ponteiro para função — é o valor inicial do stack pointer. Cada entrada seguinte é o endereço de um handler. Aqui está o início de g_vector_table do ficheiro de startup partilhado:

__attribute__((section(".isr_vector"), used))
void (* const g_vector_table[])(void) =
{
    (void (*)(void))(&_estack),  /*   0: Initial Main Stack Pointer       */
    Reset_Handler,               /*   1: Reset                            */
    NMI_Handler,                 /*   2: Non-maskable interrupt           */
    HardFault_Handler,           /*   3: Hard fault                       */
    MemManage_Handler,           /*   4: Memory management fault          */
    BusFault_Handler,            /*   5: Bus fault                        */
    UsageFault_Handler,          /*   6: Usage fault                      */
    0, 0, 0, 0,                  /* 7-10: Reserved                        */
    SVC_Handler,                 /*  11: Supervisor call (SVC)            */
    DebugMon_Handler,            /*  12: Debug monitor                    */
    0,                           /*  13: Reserved                         */
    PendSV_Handler,              /*  14: Pendable service request         */
    SysTick_Handler,             /*  15: System tick timer                */
    /* index 16+ : the STM32F100 peripheral IRQs (IRQ0 = index 16) ...    */
};

A disposição é fixada pela arquitetura, não por nós:

ÍndiceEndereço da palavraConteúdo
00x08000000MSP inicial = _estack (um valor, não um handler)
10x08000004Reset_Handler (o vetor de reset)
2–60x08000008+NMI, HardFault, MemManage, BusFault, UsageFault
110x0800002CSVC
14–150x08000038+PendSV, SysTick (índice 15)
16+0x08000040+IRQs de periféricos — a IRQ n está no índice 16 + n (WWDG = IRQ0 = índice 16)

A tag __attribute__((section(".isr_vector"))) coloca este array numa secção com nome que o linker força para o início absoluto da Flash. Em common/linker.ld:

.isr_vector :
{
    . = ALIGN(4);
    KEEP(*(.isr_vector))     /* KEEP = do not garbage-collect: nothing "calls" it */
    . = ALIGN(4);
} > FLASH                    /* FLASH ORIGIN = 0x08000000 */

Como .isr_vector é a primeira secção em FLASH, a entrada 0 fica exatamente em 0x08000000 — precisamente onde o hardware procura. O KEEP() impede o garbage collector do linker de descartar a tabela (nenhum código alguma vez a referencia, por isso sem KEEP ela desapareceria).

4.2 De onde vem _estack

_estack não é uma variável — é um símbolo de linker cujo endereço é o valor que queremos. Do linker.ld:

MEMORY
{
    FLASH (rx)  : ORIGIN = 0x08000000, LENGTH = 128K
    RAM   (rwx) : ORIGIN = 0x20000000, LENGTH = 8K
}

_estack = ORIGIN(RAM) + LENGTH(RAM);   /* 0x20000000 + 0x2000 = 0x20002000 */

Portanto o MSP inicial é 0x20002000 — uma posição acima do topo da SRAM de 8 KB. O primeiro PUSH decrementa-o para 0x20001FFC e guarda aí, em segurança dentro da RAM. Vais verificar este valor exato no exercício abaixo.

4.3 O que o Reset_Handler realmente faz

Assim que o hardware salta para o Reset_Handler, o software tem de construir o ambiente de execução do C antes de o main() poder correr. Os três passos clássicos de startup, diretamente do startup.c:

void Reset_Handler(void)
{
    uint32_t *src;
    uint32_t *dst;

    /* 1. Copy initialised data: Flash image (_sidata) -> RAM (_sdata.._edata) */
    src = &_sidata;
    for (dst = &_sdata; dst < &_edata; )
    {
        *dst = *src;
        dst++;
        src++;
    }

    /* 2. Zero-initialise the .bss section. */
    for (dst = &_sbss; dst < &_ebss; )
    {
        *dst = 0U;
        dst++;
    }

    /* 3. Hand control to the student's program. */
    (void)main();

    /* 4. Should never get here on bare metal. */
    for (;;) { /* trap */ }
}

Passo 1 — copiar .data. Um global inicializado como int x = 5; tem um problema: o seu valor inicial (5) tem de sobreviver ao corte de energia, por isso tem de viver na Flash não volátil; mas x é escrevível, por isso em tempo de execução tem de viver na RAM. O linker resolve isto dando a .data dois endereços — um endereço de load (LMA) na Flash e um endereço de run (VMA) na RAM:

_sidata = LOADADDR(.data);          /* LMA: where the init image sits in Flash */

.data :
{
    . = ALIGN(4);
    _sdata = .;                     /* VMA start: where the variable lives in RAM */
    *(.data) *(.data*)
    . = ALIGN(4);
    _edata = .;                     /* VMA end                                    */
} > RAM AT > FLASH                   /* run in RAM, load image in FLASH            */

Ao ligar a energia, a RAM contém lixo e os valores iniciais reais estão apenas na Flash. O ciclo do Passo 1 copia _edata − _sdata bytes de _sidata (Flash) para _sdata (RAM). Depois de correr, x na RAM é igual a 5.

Passo 2 — zerar .bss. A norma do C diz que os globais não inicializados (int y;) e os explicitamente postos a zero começam a 0. Guardar milhares de bytes a zero na Flash seria um desperdício, por isso .bss não ocupa nenhum espaço na imagem — o linker apenas regista a sua extensão na RAM (_sbss…_ebss), e o ciclo de startup escreve zeros sobre essa região.

Passo 3 — chamar main(). Agora o ambiente do C corresponde ao que o teu código assume (globais inicializados corretos, globais a zero zerados, pilha pronta), por isso é finalmente seguro entrar em main(). Se o main() alguma vez retornar não há SO para onde retornar, por isso o passo 4 fica preso num ciclo infinito.

Não há aqui run-time do C automático Num sistema hospedado (Linux, Windows) o crt0 da biblioteca C faz tudo isto por ti antes do main. Em bare metal tu és o crt0. É por isso que o Makefile passa -nostartfiles e liga o startup.c explicitamente: não há código de configuração escondido — o que vês em startup.c é a história toda.
Exercício Guiado

5. Ler o vetor de reset a partir de um ELF real

Objetivo: provar a ti próprio que o MSP inicial e o vetor de reset estão realmente em 0x08000000/0x08000004 com os valores que a teoria prevê. Vais construir uma imagem de firmware (reutilizando a solução do Módulo 04 como carga útil — a carga útil não importa; a maquinaria de startup é idêntica para cada módulo) e inspecionar a sua tabela de vetores com os GNU binutils.

5.1 Construir e fazer dump

Executa estes comandos a partir da pasta deste módulo. PREFIX aponta para a toolchain bare-metal:

PREFIX=/path/to/arm-gnu-toolchain-13.2.Rel1/bin/arm-none-eabi-
cd 02-arm-programmer-model

# Build an ELF using the shared Makefile + the module-04 solution as payload.
make -f ../common/Makefile PREFIX=$PREFIX SRC=../solutions/04-gpio-output.c

# (a) Disassemble: see the entry symbols and their addresses.
${PREFIX}objdump -d build/firmware.elf | head -40

# (b) Hex-dump the raw vector table section.
${PREFIX}readelf -x .isr_vector build/firmware.elf

5.2 O que procurar — a tua tarefa

O hex dump do readelf imprime os bytes de .isr_vector começando em 0x08000000. Os bytes são mostrados em little-endian, por isso uma palavra guardada 00 20 00 20 lê-se como o valor 0x20002000. A tua tarefa:

  • TODO B: Encontra a segunda palavra de 32 bits (em 0x08000004). Este é o vetor de reset — o endereço de Reset_Handler. Descodifica-o. Depois confirma que corresponde ao endereço de Reset_Handler impresso pela desmontagem.
  • TODO C: A palavra do vetor de reset é ímpar (o seu bit mais baixo é 1), mas o endereço de Reset_Handler dado pelo objdump é par. Explica a diferença de um bit (pista: §2, o bit T).

5.3 Pistas

Pista 1 — descodificar a primeira palavra (TODO A)

O dump mostra 00200020 em 0x08000000. O readelf -x agrupa os bytes tal como estão guardados. Volta a lê-los como um valor de 32 bits little-endian: a ordem de bytes guardada 00 20 00 20 torna-se o número 0x20002000. Esse é o topo da SRAM de 8 KB (0x20000000 + 0x2000) — exatamente o _estack que o linker calculou.

Pista 2 — descodificar o vetor de reset (TODO B)

A segunda palavra é 81010008. Em little-endian → 0x08000181. Compara com o nm: Reset_Handler está em 0x08000180. Diferem apenas no bit 0: 0x08000181 vs 0x08000180. A função está no endereço par; o vetor guarda o valor ímpar. Porquê? Ver a Pista 3.

Pista 3 — o endereço ímpar (TODO C)

O Cortex-M3 só executa instruções Thumb. Quando o núcleo carrega o PC a partir de um vetor (ou salta para um ponteiro de função), interpreta o bit 0 como o bit T (estado Thumb) desejado, não como parte do endereço — as instruções estão sempre alinhadas a pelo menos 2 bytes, por isso o bit 0 está livre para carregar esta flag. Um valor com bit 0 = 1 significa "continuar em estado Thumb". A toolchain define portanto o bit 0 em cada vetor de código. O hardware mascara-o de novo antes de fazer o fetch, por isso a execução começa de facto no endereço par 0x08000180. Um vetor com bit 0 = 0 pediria o estado ARM e faria fault imediatamente num Cortex-M3.

5.4 Limpeza

Remove os artefactos de build quando terminares — este módulo não disponibiliza qualquer output compilado:

make -f ../common/Makefile PREFIX=$PREFIX SRC=../solutions/04-gpio-output.c clean
O que acabaste de provar A palavra 0 da Flash = 0x20002000 = _estack (MSP inicial), e a palavra 1 = 0x08000181 = endereço de Reset_Handler com o bit Thumb ativo. As duas coisas que o silício faz no reset estão ali mesmo nos primeiros oito bytes da imagem — sem magia, apenas uma tabela.
Perguntas de Verificação

6. Verifica a tua compreensão

P1. O que faz exatamente o hardware do Cortex-M3 no reset — e o que não faz?

Exatamente duas coisas: (1) carrega o MSP inicial a partir da palavra de 32 bits em 0x08000000 (entrada de vetor 0), e (2) carrega o PC a partir da palavra em 0x08000004 (entrada de vetor 1, o vetor de reset) e começa a executar aí. Não copia o .data, não zera o .bss, não configura relógios, nem chama o main() — tudo isso é software, feito pelo Reset_Handler.

P2. MSP vs PSP: o que são e qual está ativo quando?

Ambos são cópias físicas do stack pointer R13 (é "banked"). O modo Handler usa sempre o MSP. O modo Thread usa o MSP por omissão mas pode ser comutado para PSP através do bit SPSEL do registo CONTROL — um RTOS faz isto para dar às tarefas as suas próprias pilhas enquanto as ISRs mantêm a pilha principal. No reset o hardware carrega o MSP, e este curso mantém-se no MSP em modo Thread privilegiado de princípio ao fim.

P3. Porque é que o Reset_Handler tem de zerar a secção .bss?

A norma do C garante que os globais sem inicializador (ou inicializados a zero) leem 0 no arranque do programa. Essas variáveis são agrupadas em .bss, que deliberadamente não ocupa nenhum espaço na imagem da Flash (guardar kilobytes de zeros desperdiçaria Flash). Ao ligar a energia, a RAM correspondente contém lixo indefinido, por isso o ciclo de startup tem de escrever zeros ao longo de _sbss…_ebss para honrar a garantia antes de o main() correr.

P4. Onde vivem os valores iniciais dos globais inicializados ao desligar, e porque têm de ser copiados?

Vivem na Flash, no endereço de load (LMA) da secção .data, apontado por _sidata. Têm de estar na Flash porque a RAM é volátil e perde o seu conteúdo ao desligar. Mas as próprias variáveis têm de ser escrevíveis, por isso em tempo de execução vivem na RAM no endereço de run (VMA, _sdata…_edata). O passo 1 do Reset_Handler copia a imagem da Flash para a RAM para que, por exemplo, int x = 5; leia de facto 5 quando o main() começa.

P5. Estás dentro de SysTick_Handler. O que lê o IPSR, e o que leria em main()?

Dentro de SysTick_Handler o IPSR contém o número da exceção atual, que para o SysTick é 15 (o seu índice na tabela de vetores). Para uma IRQ de periférico n leria 16 + n. Em main() — isto é, modo Thread, sem exceção ativa — o IPSR lê 0. Ler o IPSR é a forma arquitetural de um handler partilhado descobrir qual exceção está a servir.

P6. O vetor de reset na Flash lê 0x08000181, mas o Reset_Handler está em 0x08000180. Porquê o endereço ímpar?

O bit 0 de um vetor de código (ou de qualquer ponteiro de função carregado no PC) é o bit de estado Thumb (T), não parte do endereço — as instruções estão alinhadas a ≥2 bytes, por isso o bit 0 está sempre livre. O Cortex-M3 só executa código Thumb, por isso a toolchain define o bit 0 = 1 em cada vetor de código para pedir o estado Thumb. O hardware mascara-o antes de fazer o fetch, por isso a execução começa no endereço par 0x08000180. Um vetor com bit 0 = 0 pediria o estado ARM e faria fault imediatamente.
