Module 02 The ARM Programmer's Model

The Cortex-M3 core as the programmer sees it: the register file, the program status registers, operating modes, and the reset sequence that turns a cold chip into running C — walked through the real startup.c and linker.ld this course ships.

What this module is This is a theory module — there is no skeleton to fill in and no solutions/ file. Instead you will read about the core, then run a short investigation against a real compiled ELF to see the reset machinery with your own eyes. Everything here underpins every later module: when Module 04 says "main() runs," this is how it got there.
Theory

1. The core register file (R0–R15)

The Cortex-M3 is a load/store machine: the CPU operates on registers, and only dedicated load/store instructions move data to and from memory. There are sixteen 32-bit core registers visible at any instant. They are not all equal — three of them have a fixed hardware job.

RegisterRoleNotes
R0–R12General purposeScratch / variables. The C ABI passes the first 4 integer arguments in R0–R3 and returns the result in R0.
R13 = SPStack PointerBanked: there are physically two — MSP (Main) and PSP (Process). Only one is "the SP" at a time (see §3).
R14 = LRLink RegisterA BL (branch-with-link) stores the return address here. On exception entry the CPU writes a special EXC_RETURN value into LR.
R15 = PCProgram CounterAddress of the instruction being fetched. Writing it is a branch. Always reads with bit 0 = 0 even though instructions carry a Thumb bit (see §5).

1.1 The stack is full-descending

The Cortex-M3 stack grows downward (toward lower addresses) and SP points at the last item pushed — this is a full-descending stack. A PUSH first decrements SP by 4, then stores; a POP loads, then increments. That is exactly why the linker sets the initial stack pointer to the top of RAM: the first push moves it down into free space.

RAM (8 KB) 0x20002000 <- _estack (initial MSP, top of RAM) | PUSH r0 -> SP -= 4, then store stack grows DOWN ===========> | ... v 0x20000000 <- bottom of RAM (.data / .bss live here)

2. Program status: the xPSR

Condition flags, the current exception number, and the execution-state bit all live in one combined 32-bit register, the xPSR. It is really three logical views of the same register that you can read/write separately:

ViewHoldsKey fields
APSRApplication status — the arithmetic flagsN (negative), Z (zero), C (carry), V (overflow), and Q (saturation)
IPSRInterrupt status — which exception is executingException number. 0 = Thread mode (no exception). Inside SysTick_Handler it reads 15; inside an IRQ n it reads 16 + n.
EPSRExecution statusThe T bit (Thumb state) and the interrupted-instruction continuation state. T must be 1 on a Cortex-M3 — it only runs Thumb code.
xPSR (bit 31 .................................................. bit 0) |31 30 29 28 27| ... |26 ........ 16| ... |8 ..... 0| | N Z C V Q| | ICI/IT, T | | IPSR # | \___APSR_____/ \___EPSR_____/ \__IPSR_/

The N/Z/C/V flags are how C comparisons work: if (a == b) compiles to a CMP that sets Z, then a conditional branch that tests it. You rarely touch the xPSR by hand in C — but you must understand IPSR to know, inside a handler, which interrupt fired (covered in the interrupt modules), and the T bit is the reason for a subtle detail you will see in §5.

3. Operating modes and the two stacks

The Cortex-M3 has exactly two modes and two privilege levels:

AxisStateMeaning
ModeThreadNormal code — your main() and everything it calls. Entered at reset.
HandlerRunning an exception/interrupt handler. Entered automatically when an exception is taken; always privileged.
PrivilegePrivilegedFull access to all special registers and the System Control Space (NVIC, SysTick, etc.).
UnprivilegedRestricted; an RTOS may run application threads here. Cannot reach core control registers.

Which physical stack pointer is active depends on mode and a control bit:

  • Handler mode always uses MSP. Exception handlers run on the main stack.
  • Thread mode uses MSP by default, but can be switched to PSP via the CONTROL register's SPSEL bit.
Why two stacks exist An RTOS gives each task its own stack via PSP, while interrupt handlers run on a separate MSP the kernel controls — so an ISR can never overflow a task's stack, and a stack-starved task cannot corrupt the kernel. This bare-metal course never switches to PSP: we live entirely in privileged Thread mode on MSP. But you should know PSP exists, because the very first thing the hardware does at reset is load MSP (§4).

4. From cold chip to main(): reset and the vector table

There is no BIOS and no bootloader doing setup for you. When the STM32F100 leaves reset, the Cortex-M3 hardware does exactly two things, both by reading the vector table that sits at the very start of Flash, address 0x08000000:

  1. Load the initial MSP from the 32-bit word at 0x08000000 (vector entry 0).
  2. Load the PC from the word at 0x08000004 (vector entry 1) and start executing there — that address is our Reset_Handler.

That is the entire hardware contribution. From Reset_Handler onward, every byte of setup is software we write. (RM0041 §10; Cortex-M3 PM0056.)

4.1 The vector table in common/startup.c

The table is just a C array placed at 0x08000000 by the linker. The crucial detail: entry 0 is not a function pointer — it is the initial stack-pointer value. Every following entry is the address of a handler. Here is the head of g_vector_table from the shared startup file:

__attribute__((section(".isr_vector"), used))
void (* const g_vector_table[])(void) =
{
    (void (*)(void))(&_estack),  /*   0: Initial Main Stack Pointer       */
    Reset_Handler,               /*   1: Reset                            */
    NMI_Handler,                 /*   2: Non-maskable interrupt           */
    HardFault_Handler,           /*   3: Hard fault                       */
    MemManage_Handler,           /*   4: Memory management fault          */
    BusFault_Handler,            /*   5: Bus fault                        */
    UsageFault_Handler,          /*   6: Usage fault                      */
    0, 0, 0, 0,                  /* 7-10: Reserved                        */
    SVC_Handler,                 /*  11: Supervisor call (SVC)            */
    DebugMon_Handler,            /*  12: Debug monitor                    */
    0,                           /*  13: Reserved                         */
    PendSV_Handler,              /*  14: Pendable service request         */
    SysTick_Handler,             /*  15: System tick timer                */
    /* index 16+ : the STM32F100 peripheral IRQs (IRQ0 = index 16) ...    */
};

The layout is fixed by the architecture, not by us:

IndexWord addressContents
00x08000000Initial MSP = _estack (a value, not a handler)
10x08000004Reset_Handler (the reset vector)
2–60x08000008+NMI, HardFault, MemManage, BusFault, UsageFault
110x0800002CSVC
14–150x08000038+PendSV, SysTick (index 15)
16+0x08000040+Peripheral IRQs — IRQ n is at index 16 + n (WWDG = IRQ0 = index 16)

The __attribute__((section(".isr_vector"))) tag puts this array into a named section that the linker forces to the very start of Flash. In common/linker.ld:

.isr_vector :
{
    . = ALIGN(4);
    KEEP(*(.isr_vector))     /* KEEP = do not garbage-collect: nothing "calls" it */
    . = ALIGN(4);
} > FLASH                    /* FLASH ORIGIN = 0x08000000 */

Because .isr_vector is the first section in FLASH, entry 0 lands precisely at 0x08000000 — exactly where the hardware looks. KEEP() stops the linker's garbage collector from discarding the table (no code ever references it, so without KEEP it would vanish).

4.2 Where _estack comes from

_estack is not a variable — it is a linker symbol whose address is the value we want. From linker.ld:

MEMORY
{
    FLASH (rx)  : ORIGIN = 0x08000000, LENGTH = 128K
    RAM   (rwx) : ORIGIN = 0x20000000, LENGTH = 8K
}

_estack = ORIGIN(RAM) + LENGTH(RAM);   /* 0x20000000 + 0x2000 = 0x20002000 */

So the initial MSP is 0x20002000 — one past the top of the 8 KB SRAM. The first PUSH decrements it to 0x20001FFC and stores there, safely inside RAM. You will verify this exact value in the exercise below.

4.3 What Reset_Handler actually does

Once the hardware jumps to Reset_Handler, software must build the C run-time environment before main() can run. The three classic startup steps, straight from startup.c:

void Reset_Handler(void)
{
    uint32_t *src;
    uint32_t *dst;

    /* 1. Copy initialised data: Flash image (_sidata) -> RAM (_sdata.._edata) */
    src = &_sidata;
    for (dst = &_sdata; dst < &_edata; )
    {
        *dst = *src;
        dst++;
        src++;
    }

    /* 2. Zero-initialise the .bss section. */
    for (dst = &_sbss; dst < &_ebss; )
    {
        *dst = 0U;
        dst++;
    }

    /* 3. Hand control to the student's program. */
    (void)main();

    /* 4. Should never get here on bare metal. */
    for (;;) { /* trap */ }
}

Step 1 — copy .data. An initialised global like int x = 5; has a problem: its initial value (5) must survive power-off, so it has to live in non-volatile Flash; but x is writable, so at run time it must live in RAM. The linker solves this by giving .data two addresses — a load address (LMA) in Flash and a run address (VMA) in RAM:

_sidata = LOADADDR(.data);          /* LMA: where the init image sits in Flash */

.data :
{
    . = ALIGN(4);
    _sdata = .;                     /* VMA start: where the variable lives in RAM */
    *(.data) *(.data*)
    . = ALIGN(4);
    _edata = .;                     /* VMA end                                    */
} > RAM AT > FLASH                   /* run in RAM, load image in FLASH            */

At power-on, RAM holds garbage and the real initial values are only in Flash. Step 1's loop copies _edata − _sdata bytes from _sidata (Flash) down into _sdata (RAM). After it runs, x in RAM equals 5.

Step 2 — zero .bss. The C standard says uninitialised globals (int y;) and those explicitly set to zero start at 0. Storing thousands of zero bytes in Flash would be wasteful, so .bss takes no space in the image — the linker only records its RAM extent (_sbss…_ebss), and the startup loop writes zeros over that region.

Step 3 — call main(). Now the C environment matches what your code assumes (initialised globals correct, zero-globals zeroed, stack ready), so it is finally safe to enter main(). If main() ever returns there is no OS to return to, so step 4 traps in an infinite loop.

No automatic C run-time here On a hosted system (Linux, Windows) the C library's crt0 does all of this for you before main. On bare metal you are crt0. This is why the Makefile passes -nostartfiles and links startup.c explicitly: there is no hidden setup code — what you see in startup.c is the whole story.

5.3 Hints

Hint 1 — decoding the first word (TODO A)

The dump shows 00200020 at 0x08000000. readelf -x groups bytes as stored. Re-read them as a little-endian 32-bit value: the stored byte order 00 20 00 20 becomes the number 0x20002000. That is the top of the 8 KB SRAM (0x20000000 + 0x2000) — exactly the _estack the linker computed.

Hint 2 — decoding the reset vector (TODO B)

The second word is 81010008. Little-endian → 0x08000181. Compare with nm: Reset_Handler is at 0x08000180. They differ only in bit 0: 0x08000181 vs 0x08000180. The function is at the even address; the vector stores the odd value. Why? See Hint 3.

Hint 3 — the odd address (TODO C)

The Cortex-M3 only executes Thumb instructions. When the core loads the PC from a vector (or branches to a function pointer), it interprets bit 0 as the desired T (Thumb-state) bit, not as part of the address — instructions are always at least 2-byte aligned, so bit 0 is free to carry this flag. A value with bit 0 = 1 means "continue in Thumb state." The toolchain therefore sets bit 0 in every code vector. The hardware masks it back off before fetching, so execution actually begins at the even address 0x08000180. A vector with bit 0 = 0 would request ARM state and immediately fault on a Cortex-M3.

Verification Questions

6. Check your understanding

Q1. What exactly does the Cortex-M3 hardware do at reset — and what does it not do?

Exactly two things: (1) load the initial MSP from the 32-bit word at 0x08000000 (vector entry 0), and (2) load the PC from the word at 0x08000004 (vector entry 1, the reset vector) and start executing there. It does not copy .data, zero .bss, set up clocks, or call main() — all of that is software, done by Reset_Handler.

Q2. MSP vs PSP: what are they and which one is active when?

Both are physical copies of the stack pointer R13 (it is "banked"). Handler mode always uses MSP. Thread mode uses MSP by default but can be switched to PSP via the CONTROL register's SPSEL bit — an RTOS does this to give tasks their own stacks while ISRs keep the main stack. At reset the hardware loads MSP, and this course stays on MSP in privileged Thread mode throughout.

Q3. Why must Reset_Handler zero the .bss section?

The C standard guarantees that globals with no initializer (or initialized to zero) read as 0 at program start. Those variables are grouped into .bss, which deliberately occupies no space in the Flash image (storing kilobytes of zeros would waste Flash). At power-on the corresponding RAM holds undefined garbage, so the startup loop must write zeros across _sbss…_ebss to honor the guarantee before main() runs.

Q4. Where do the initial values of initialized globals live at power-off, and why must they be copied?

They live in Flash, at the .data section's load address (LMA), pointed to by _sidata. They must be in Flash because RAM is volatile and loses its contents at power-off. But the variables themselves must be writable, so at run time they live in RAM at the run address (VMA, _sdata…_edata). Reset_Handler's step 1 copies the image from Flash to RAM so that, e.g., int x = 5; actually reads 5 when main() starts.

Q5. You are inside SysTick_Handler. What does the IPSR read, and what would it read in main()?

Inside SysTick_Handler the IPSR holds the current exception number, which for SysTick is 15 (its vector-table index). For a peripheral IRQ n it would read 16 + n. In main() — i.e. Thread mode, no exception active — the IPSR reads 0. Reading IPSR is the architectural way for a shared handler to discover which exception it is servicing.

Q6. The reset vector in Flash reads 0x08000181, but Reset_Handler is at 0x08000180. Why the odd address?

Bit 0 of a code vector (or any function pointer loaded into PC) is the Thumb-state (T) bit, not part of the address — instructions are ≥2-byte aligned so bit 0 is always free. The Cortex-M3 executes only Thumb code, so the toolchain sets bit 0 = 1 on every code vector to request Thumb state. The hardware masks it off before fetching, so execution begins at the even address 0x08000180. A vector with bit 0 = 0 would request ARM state and fault immediately.
