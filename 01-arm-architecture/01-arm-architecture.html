Module 01 ARM Design Philosophy & RISC Architecture

Before writing a single register access, understand what kind of machine the STM32F100 is: a load-store RISC core (Cortex-M3) that processes data only in registers, executes one fixed-format instruction set (Thumb-2), and maps every peripheral into a single address space.

Where this module fits This is the first module of the course and has no firmware to flash — it is pure architecture theory plus one hands-on disassembly exercise. You already know C and generic CPU concepts (registers, ALU, memory). The goal here is to map that knowledge onto ARM so that everything from Module 02 onward makes sense. There is no QEMU step and no solution file.
Theory

1. RISC vs CISC

Two broad philosophies have shaped CPU instruction sets:

CISC (e.g. x86)RISC (e.g. ARM)
Instruction countMany, often hundreds, some very specializedFew, simple, general-purpose
Instruction lengthVariable (1–15 bytes on x86)Fixed or near-fixed (2 or 4 bytes on Cortex-M)
Memory operandsMost instructions can read/write memory directly (add [mem], reg)Only dedicated load/store instructions touch memory
RegistersFew architectural registers (historically 8 on x86)Large, uniform register file (16 on ARM)
DecodeComplex; one instruction may be microcoded into many stepsSimple; one instruction ≈ one fixed pipeline path

The RISC bet is: keep each instruction simple and predictable so the hardware decoding it can be small and fast, then let the compiler stitch those simple instructions together. A small, regular decoder uses less silicon and less power — which is exactly what a microcontroller wants. The name says it: Reduced Instruction Set Computer. "Reduced" means each instruction does less, not that there are necessarily fewer of them.

Not a religious war Modern high-end x86 chips internally crack CISC instructions into RISC-like "micro-ops", and ARM has plenty of instructions. The labels describe the visible instruction-set design, not a hard line in silicon. For a microcontroller like the STM32F100, the RISC traits below are what actually matter to you as a bare-metal programmer.

2. The load-store architecture

This is the single most important RISC trait for understanding ARM assembly. On a load-store machine:

  • Arithmetic and logic operate only on registers (and small immediates). add r0, r1, r2 means r0 = r1 + r2 — all three are CPU registers.
  • Memory is touched only by explicit load and store instructions: ldr (LoaD Register, memory → register) and str (STore Register, register → memory).

So the universal pattern for "do something with a value in memory" is always three phases:

load-store cycle (the rhythm of every ARM program) MEMORY REGISTER FILE (r0..r12) +-----------+ +------------------+ | array[0] | --- ldr ---> | r3 = array[0] | (1) LOAD | array[1] | --- ldr ---> | r2 = array[1] | +-----------+ | | | add r3, r3, r2 | (2) COMPUTE | (r3 = r3 + r2) | (registers only) +-----------+ | | | result | <--- str --- | r3 | (3) STORE +-----------+ +------------------+

Contrast with a CISC machine where add result, array[1] could fold the load, the add, and the store into one instruction. ARM forbids that. It feels verbose, but it makes every instruction's timing and memory behavior predictable, which is gold for real-time embedded code.

Why you care as a firmware author When you write GPIOC_BSRR = (1U << 9); later in the course, the compiler must emit: load the address of BSRR into a register, load/compute the value into another register, then str the value to that address. Memory-mapped peripheral access is just the load-store pattern pointed at a hardware register instead of RAM. Understanding this is why volatile matters — it tells the compiler "do not optimize away these loads and stores."

3. Fixed-size instruction encoding & the register file

3.1 Fixed-size encoding

On classic 32-bit ARM, every instruction is exactly 4 bytes. This makes the fetch/decode logic trivial — the CPU always knows where the next instruction begins. (Cortex-M complicates this slightly with Thumb-2, covered in §4: instructions are 2 or 4 bytes, but still drawn from a small, regular set.) Compare to x86 where an instruction can be anywhere from 1 to 15 bytes and the decoder must partly execute one instruction to find where the next starts.

3.2 A large, uniform register file

The ARM (and Cortex-M3) integer register file has 16 registers, r0–r15, each 32 bits wide. Most are general-purpose; a few have special roles by convention or hardware:

RegisterRole
r0–r12General-purpose (arguments, locals, scratch)
r13 (sp)Stack Pointer
r14 (lr)Link Register (return address)
r15 (pc)Program Counter

Having many registers means the compiler can keep more values "in hand" without spilling them to memory, which directly reduces the number of slow ldr/str operations. We dissect these registers in detail in Module 02 (Programmer's Model).

3.3 Pipelining — the Cortex-M3 has 3 stages

Because instructions are simple and regular, ARM cores overlap their execution in a pipeline. The Cortex-M3 uses a classic 3-stage pipeline:

Cortex-M3 3-stage pipeline (one instruction finishes ~every cycle) cycle: 1 2 3 4 5 instr A: [FETCH] [DECODE] [EXECUTE] instr B: [FETCH] [DECODE] [EXECUTE] instr C: [FETCH] [DECODE] [EXECUTE] FETCH = read instruction from Flash DECODE = work out what it does, read register operands EXECUTE = run it through the ALU / load-store unit

One consequence you will meet in Module 02: because the PC is read in the fetch stage, the value of pc visible to an executing instruction is ahead of that instruction's own address. Simple, fixed-format instructions are what make a clean pipeline like this cheap to build — another payoff of the RISC design.

4. ARM, Thumb, and Thumb-2 instruction sets

"ARM" the company has shipped several encodings of its instruction set over time:

Instruction setWidthIdea
ARM32-bit fixedThe original. Maximum capability per instruction, but every instruction costs 4 bytes of program memory.
Thumb (Thumb-1)16-bit fixedA compressed subset: the most common operations re-encoded into 2 bytes each for much better code density, at the cost of needing more instructions for some tasks.
Thumb-2mixed 16- & 32-bitThe best of both: keep the dense 16-bit forms for common ops, add 32-bit forms for everything else, all in one seamlessly interleaved instruction stream.

4.1 Why Cortex-M runs Thumb-2 only

Microcontrollers are dominated by code density: the STM32F100RBT6 has just 128 KB of Flash. Every byte of program counts. Thumb-2 gives near-ARM performance while packing code tightly, so the entire Cortex-M family dropped the legacy 32-bit ARM state entirely and executes Thumb-2 exclusively. There is no mode switch to worry about, no interworking — the core is always in Thumb state.

Practical consequence for the toolchain This is why every compile command in this course passes -mthumb (and -mcpu=cortex-m3). You are telling GCC: emit Thumb-2 code for an ARMv7-M core. Hand it ARM (32-bit) code and the Cortex-M3 would fault — it cannot decode the old encoding. It is also why function pointers on Cortex-M have bit 0 set to 1 (the "Thumb bit"); you will see this when we set up the vector table.

5. Where the STM32F100 sits

Putting names to layers, from the abstract ISA down to the chip on your board:

From architecture to silicon ARMv7-M <- the ARCHITECTURE (the contract: what instructions and registers exist, the memory model) | Cortex-M3 <- the CORE: ARM's specific implementation of ARMv7-M (3-stage pipeline, NVIC, SysTick, Thumb-2 only) | STM32F100RBT6 <- the CHIP: ST wraps a Cortex-M3 core with Flash, SRAM, and peripherals (GPIO, USART, timers...) | STM32VLDISCOVERY <- the BOARD: the chip + LEDs, button, debugger

Key architectural facts about this core that shape everything you write:

PropertyValueWhy it matters to you
CoreARM Cortex-M3Implements the ARMv7-M architecture.
Word size32-bitRegisters, addresses, and the native int are all 32 bits.
EndiannessLittle-endianThe least-significant byte of a word sits at the lowest address. Matters when you cast pointers or pack bytes.
Bus structureHarvard-styleSeparate buses for fetching instructions (Flash) and accessing data (SRAM/peripherals), so a fetch and a data access can happen at once.
Address spaceSingle 4 GB, memory-mappedFlash, SRAM, and every peripheral register live at fixed addresses in one flat 32-bit space. There is no separate "I/O" instruction.
"Memory-mapped everything" Because peripherals are just addresses, controlling hardware in C is identical to writing to memory: you make a volatile pointer to the peripheral's address and read/write it. No special instructions, no OS calls. That is the entire premise of bare-metal programming and the subject of Module 03. The fixed addresses you will use (GPIOC at 0x40011000, etc.) come from the reference manual RM0041.

6. Why RISC/ARM dominates embedded

  • Power. A small, simple decoder and core switch fewer transistors per instruction. For battery and always-on devices this is decisive.
  • Code density (Thumb-2). More program fits in scarce on-chip Flash, so a cheaper chip with less memory can still do the job.
  • Predictable cycle timing. Fixed-format, mostly single-purpose instructions and a short, simple pipeline make it far easier to reason about exactly how long a code path takes — essential for hard real-time control.
  • Licensable IP. ARM sells the core design; vendors like ST add their own peripherals. That ecosystem is why a Cortex-M3 shows up in thousands of different chips.
Guided Exercise

7. See the load-store pattern with your own eyes

Goal: compile a tiny C function for Cortex-M3, disassemble the object file, and identify the ldr / str / arithmetic instructions — confirming that ARM computes only in registers and touches memory only via loads and stores.

Toolchain not required to learn the lesson This exercise uses arm-none-eabi-gcc and arm-none-eabi-objdump. They may not be on your PATH yet — the course README covers installing the gcc-arm-none-eabi toolchain. If you cannot run them right now, that is fine: a real annotated disassembly is printed below in §7.3, and the lesson stands entirely on reading it. There is no QEMU and no solution file for this module.

7.1 Skeleton — create 01-arm-architecture/example.c

Write a function that sums the elements of an int array. Keep it tiny — we want short, readable assembly, not a clever algorithm. Fill in the TODO yourself before peeking at the hints.

#include <stdint.h>

/* Sum the first `n` elements of `data`.
 * We want this to disassemble into an obvious load-add-loop so we can
 * spot the load-store pattern in the machine code. */
int32_t sum_array(const int32_t *data, uint32_t n)
{
    int32_t total = 0;

    /* TODO: loop over the n elements, adding each data[i] to total.
     *       Use a plain for-loop. Do NOT do anything fancy. */

    return total;
}

7.2 Compile & disassemble

From inside the 01-arm-architecture/ folder:

# Compile to an object file for the Cortex-M3 in Thumb-2 mode.
#   -mcpu=cortex-m3  : target this exact core
#   -mthumb          : emit Thumb-2 (the only state the M3 understands)
#   -Og              : light optimization -> readable but real code
#   -c               : compile only, do not link
arm-none-eabi-gcc -mcpu=cortex-m3 -mthumb -Og -c example.c -o example.o

# Disassemble the machine code back to assembly.
arm-none-eabi-objdump -d example.o

Read the <sum_array>: section of the output. Look for three things:

  1. A ldr that reads an array element from memory into a register.
  2. An add that works only on registers (no memory operand in sight).
  3. The branch (b / cmp) that forms the loop.

Notice what is absent: there is no instruction that adds a memory location directly to a register in one step. Every value must be ldr'd in first. That is the load-store architecture, in the metal.

7.3 Sample disassembly (annotated)

Here is representative -Og output for the finished function so the lesson lands even without the toolchain. Exact registers/encodings vary by GCC version, but the shape is always this:

00000000 <sum_array>: 0: 2300 movs r3, #0 ; total = 0 (immediate -> register) 2: 2200 movs r2, #0 ; i = 0 4: e004 b.n 10 <.loop_test> ; --- loop body --- 6: f850 0022 ldr.w r0, [r0, r2, lsl #2] ; LOAD data[i] (base r0 + i*4) -> r0 ; *** memory is read ONLY here *** a: 4403 add r3, r0 ; total += data[i] (register + register) ; *** arithmetic: registers ONLY *** c: 3201 adds r2, #1 ; i++ (register + immediate) ; --- loop test --- e: ... ; .loop_test: 10: 4291 cmp r1, r2 ; compare n with i (registers) 12: d8f8 bhi.n 6 ; if i < n, branch back up to the LOAD 14: 4618 mov r0, r3 ; return value in r0 (ARM calling convention) 16: 4770 bx lr ; return (branch to Link Register)
  • ldr at 0x6 is the only instruction reading memory — it pulls data[i] into a register first. The lsl #2 is the array index scaled by 4 (the size of an int32_t).
  • add at 0xa sums two registers. The CPU cannot add memory to a register in one instruction — the value had to be loaded first.
  • Instruction widths are mixed: movs/add are 2-byte Thumb forms, ldr.w is a 4-byte Thumb-2 form (the .w = "wide"). That mix is Thumb-2 in action.
  • There is no str here because the only result is a return value, which lives in a register (r0). Add a line like out[0] = total; and recompile — you will see a str appear, completing the load-compute-store trio.
Try this variation Change the function to write the result through a pointer (void sum_into(const int32_t *data, uint32_t n, int32_t *out) with *out = total;) and disassemble again. Find the new str instruction and confirm it is the only thing that writes to memory.

7.4 Hints

Hint 1 — the loop body

A textbook accumulate loop is all you need:

for (uint32_t i = 0U; i < n; i++)
{
    total += data[i];
}

Resist optimizing it. The clearer the C, the clearer the assembly.

Hint 2 — "arm-none-eabi-gcc: command not found"

The toolchain is not installed or not on your PATH. On Debian/Ubuntu: sudo apt install gcc-arm-none-eabi. On macOS with Homebrew: brew install --cask gcc-arm-embedded. The course README has full instructions. You can still complete the lesson from the annotated disassembly in §7.3.

Hint 3 — which lines are the load-store pattern?

Scan the objdump output for the mnemonics ldr (load) and str (store). Everything between them — add, cmp, mov, adds — operates on registers and immediates only. That separation is the load-store architecture. If you see an ldr feeding an add that feeds (eventually) an str, you have found the load → compute → store rhythm.

Verification Questions

8. Check your understanding

Q1. What does "load-store architecture" mean, in one sentence?

Arithmetic and logic instructions operate only on registers (and immediates); the only way to access memory is through dedicated load (ldr) and store (str) instructions. So any computation on a value in memory is always load → compute-in-registers → store.

Q2. Why does the Cortex-M3 (and the whole Cortex-M family) execute Thumb-2 and not the legacy 32-bit ARM instruction set?

Code density. Microcontrollers have very little Flash (the STM32F100RBT6 has 128 KB), so fitting more program into fewer bytes is critical. Thumb-2 mixes 16-bit and 32-bit encodings to get near-ARM performance with much tighter code, so ARM dropped the 4-byte-only ARM state from Cortex-M entirely. The core is always in Thumb state — which is why we always compile with -mthumb.

Q3. How many bits wide is a Cortex-M3 general-purpose register, and how many are there?

Each register is 32 bits wide. There are 16 of them, r0–r15; r0–r12 are general-purpose, while r13=sp, r14=lr, and r15=pc have special roles.

Q4. State one core tradeoff of RISC versus CISC.

RISC instructions each do less, so a given task often needs more instructions (and more program memory in the naive case) than CISC — but each instruction is simple, fixed-format, and fast to decode, giving a smaller/lower-power core and far more predictable timing. CISC packs more work per instruction (and can be denser per operation) at the cost of complex, variable-length decoding. (Thumb-2 specifically attacks the code-size downside of RISC.)

Q5. Naming the layers: what is the relationship between ARMv7-M, Cortex-M3, and STM32F100?

ARMv7-M is the architecture — the contract of what instructions and registers exist. Cortex-M3 is ARM's specific core that implements ARMv7-M (3-stage pipeline, NVIC, SysTick, Thumb-2 only). STM32F100 is ST's chip that wraps a Cortex-M3 core with Flash, SRAM, and peripherals. (And STM32VLDISCOVERY is the board carrying that chip.)

Q6. Why does "memory-mapped everything" make bare-metal C possible without special I/O instructions?

Every peripheral register lives at a fixed address in the same flat 32-bit address space as RAM. Controlling hardware is therefore just reading and writing memory addresses — which in C means dereferencing a volatile pointer. No special instructions or OS calls are needed, and that is exactly what Module 03 builds on.
