Módulo 01 Filosofia de Design ARM & Arquitetura RISC

Antes de escrever um único acesso a registos, compreende que tipo de máquina é o STM32F100: um núcleo RISC load-store (Cortex-M3) que processa dados apenas em registos, executa um único conjunto de instruções de formato fixo (Thumb-2) e mapeia todos os periféricos num único espaço de endereçamento.

Onde se encaixa este módulo Este é o primeiro módulo do curso e não tem firmware para gravar — é pura teoria de arquitetura mais um exercício prático de desmontagem (disassembly). Já conheces C e os conceitos genéricos de CPU (registos, ALU, memória). O objetivo aqui é mapear esse conhecimento para ARM, de modo que tudo a partir do Módulo 02 faça sentido. Não há passo de QEMU nem ficheiro de solução.
Teoria

1. RISC vs CISC

Duas grandes filosofias moldaram os conjuntos de instruções de CPU:

CISC (ex.: x86)RISC (ex.: ARM)
Número de instruçõesMuitas, frequentemente centenas, algumas muito especializadasPoucas, simples, de uso geral
Comprimento da instruçãoVariável (1–15 bytes em x86)Fixo ou quase fixo (2 ou 4 bytes no Cortex-M)
Operandos em memóriaA maioria das instruções pode ler/escrever memória diretamente (add [mem], reg)Apenas instruções dedicadas de load/store tocam na memória
RegistosPoucos registos arquiteturais (historicamente 8 em x86)Conjunto de registos grande e uniforme (16 em ARM)
DescodificaçãoComplexa; uma instrução pode ser microcodificada em muitos passosSimples; uma instrução ≈ um caminho de pipeline fixo

A aposta do RISC é: manter cada instrução simples e previsível para que o hardware que a descodifica possa ser pequeno e rápido, deixando depois o compilador unir essas instruções simples. Um descodificador pequeno e regular usa menos silício e menos energia — que é exatamente o que um microcontrolador quer. O nome di-lo: Reduced Instruction Set Computer (Computador com Conjunto de Instruções Reduzido). "Reduzido" significa que cada instrução faz menos, não que haja necessariamente menos instruções.

Não é uma guerra religiosa Os chips x86 modernos de gama alta partem internamente as instruções CISC em "micro-ops" do tipo RISC, e o ARM tem muitas instruções. Os rótulos descrevem o design visível do conjunto de instruções, não uma linha rígida no silício. Para um microcontrolador como o STM32F100, os traços RISC abaixo são o que realmente importa para ti como programador bare-metal.

2. A arquitetura load-store

Este é o traço RISC mais importante para compreender o assembly ARM. Numa máquina load-store:

  • A aritmética e a lógica operam apenas sobre registos (e pequenos imediatos). add r0, r1, r2 significa r0 = r1 + r2 — os três são registos do CPU.
  • A memória só é tocada por instruções explícitas de load e store: ldr (LoaD Register, memória → registo) e str (STore Register, registo → memória).

Por isso o padrão universal para "fazer algo com um valor em memória" tem sempre três fases:

ciclo load-store (o ritmo de todo o programa ARM) MEMORY REGISTER FILE (r0..r12) +-----------+ +------------------+ | array[0] | --- ldr ---> | r3 = array[0] | (1) LOAD | array[1] | --- ldr ---> | r2 = array[1] | +-----------+ | | | add r3, r3, r2 | (2) COMPUTE | (r3 = r3 + r2) | (registers only) +-----------+ | | | result | <--- str --- | r3 | (3) STORE +-----------+ +------------------+

Contrasta com uma máquina CISC, onde add result, array[1] poderia juntar o load, a soma e o store numa única instrução. O ARM proíbe isso. Parece verboso, mas torna o tempo e o comportamento de memória de cada instrução previsíveis, o que é ouro para código embebido em tempo real.

Porque é que isto te interessa enquanto autor de firmware Quando escreveres GPIOC_BSRR = (1U << 9); mais à frente no curso, o compilador tem de emitir: carregar o endereço de BSRR para um registo, carregar/calcular o valor para outro registo, e depois fazer str do valor para esse endereço. O acesso a periféricos mapeados em memória é apenas o padrão load-store apontado a um registo de hardware em vez de RAM. Compreender isto é o motivo pelo qual volatile importa — diz ao compilador "não otimizes estes loads e stores para fora."

3. Codificação de instruções de tamanho fixo & o conjunto de registos

3.1 Codificação de tamanho fixo

No ARM clássico de 32 bits, todas as instruções têm exatamente 4 bytes. Isto torna a lógica de fetch/decode trivial — o CPU sabe sempre onde começa a próxima instrução. (O Cortex-M complica isto ligeiramente com o Thumb-2, abordado no §4: as instruções têm 2 ou 4 bytes, mas continuam a vir de um conjunto pequeno e regular.) Compara com o x86, onde uma instrução pode ter de 1 a 15 bytes e o descodificador tem de executar parcialmente uma instrução para descobrir onde começa a seguinte.

3.2 Um conjunto de registos grande e uniforme

O conjunto de registos inteiros do ARM (e do Cortex-M3) tem 16 registos, r0–r15, cada um com 32 bits de largura. A maioria é de uso geral; alguns têm papéis especiais por convenção ou por hardware:

RegistoPapel
r0–r12Uso geral (argumentos, locais, scratch)
r13 (sp)Stack Pointer (ponteiro de pilha)
r14 (lr)Link Register (endereço de retorno)
r15 (pc)Program Counter (contador de programa)

Ter muitos registos significa que o compilador pode manter mais valores "à mão" sem os despejar para a memória, o que reduz diretamente o número de operações lentas ldr/str. Dissecamos estes registos em detalhe no Módulo 02 (Modelo do Programador).

3.3 Pipelining — o Cortex-M3 tem 3 andares

Como as instruções são simples e regulares, os núcleos ARM sobrepõem a sua execução numa pipeline. O Cortex-M3 usa uma pipeline clássica de 3 andares:

Cortex-M3 3-stage pipeline (one instruction finishes ~every cycle) cycle: 1 2 3 4 5 instr A: [FETCH] [DECODE] [EXECUTE] instr B: [FETCH] [DECODE] [EXECUTE] instr C: [FETCH] [DECODE] [EXECUTE] FETCH = read instruction from Flash DECODE = work out what it does, read register operands EXECUTE = run it through the ALU / load-store unit

Uma consequência que vais encontrar no Módulo 02: como o PC é lido no andar de fetch, o valor de pc visível para uma instrução em execução está à frente do endereço dessa própria instrução. Instruções simples, de formato fixo, são o que torna uma pipeline limpa como esta barata de construir — outro benefício do design RISC.

4. Conjuntos de instruções ARM, Thumb e Thumb-2

A "ARM", a empresa, lançou várias codificações do seu conjunto de instruções ao longo do tempo:

Conjunto de instruçõesLarguraIdeia
ARM32 bits fixoO original. Capacidade máxima por instrução, mas cada instrução custa 4 bytes de memória de programa.
Thumb (Thumb-1)16 bits fixoUm subconjunto comprimido: as operações mais comuns recodificadas em 2 bytes cada para uma densidade de código muito melhor, à custa de precisar de mais instruções para algumas tarefas.
Thumb-2misto 16 & 32 bitsO melhor de ambos: manter as formas densas de 16 bits para operações comuns, acrescentar formas de 32 bits para tudo o resto, tudo num único fluxo de instruções entrelaçado de forma transparente.

4.1 Porque é que o Cortex-M corre apenas Thumb-2

Os microcontroladores são dominados pela densidade de código: o STM32F100RBT6 tem apenas 128 KB de Flash. Cada byte de programa conta. O Thumb-2 dá desempenho próximo do ARM ao mesmo tempo que compacta o código de forma apertada, por isso toda a família Cortex-M abandonou inteiramente o antigo estado ARM de 32 bits e executa Thumb-2 exclusivamente. Não há mudança de modo com que te preocupares, não há interworking — o núcleo está sempre em estado Thumb.

Consequência prática para a toolchain É por isto que cada comando de compilação neste curso passa -mthumb (e -mcpu=cortex-m3). Estás a dizer ao GCC: emite código Thumb-2 para um núcleo ARMv7-M. Dá-lhe código ARM (32 bits) e o Cortex-M3 entraria em falha (fault) — não consegue descodificar a codificação antiga. É também por isso que os ponteiros de função no Cortex-M têm o bit 0 a 1 (o "Thumb bit"); vais ver isto quando configurarmos a tabela de vetores.

5. Onde se situa o STM32F100

Dando nomes às camadas, do ISA abstrato até ao chip na tua placa:

From architecture to silicon ARMv7-M <- the ARCHITECTURE (the contract: what instructions and registers exist, the memory model) | Cortex-M3 <- the CORE: ARM's specific implementation of ARMv7-M (3-stage pipeline, NVIC, SysTick, Thumb-2 only) | STM32F100RBT6 <- the CHIP: ST wraps a Cortex-M3 core with Flash, SRAM, and peripherals (GPIO, USART, timers...) | STM32VLDISCOVERY <- the BOARD: the chip + LEDs, button, debugger

Factos arquiteturais-chave sobre este núcleo que moldam tudo o que escreves:

PropriedadeValorPorque te importa
NúcleoARM Cortex-M3Implementa a arquitetura ARMv7-M.
Tamanho da palavra32 bitsRegistos, endereços e o int nativo têm todos 32 bits.
EndiannessLittle-endianO byte menos significativo de uma palavra fica no endereço mais baixo. Importa quando fazes cast de ponteiros ou empacotas bytes.
Estrutura de barramentoEstilo HarvardBarramentos separados para buscar instruções (Flash) e aceder a dados (SRAM/periféricos), de modo que um fetch e um acesso a dados podem acontecer ao mesmo tempo.
Espaço de endereçamentoÚnico de 4 GB, mapeado em memóriaFlash, SRAM e todos os registos de periférico vivem em endereços fixos num único espaço plano de 32 bits. Não há instrução de "I/O" separada.
"Tudo mapeado em memória" Como os periféricos são apenas endereços, controlar hardware em C é idêntico a escrever em memória: fazes um ponteiro volatile para o endereço do periférico e lês/escreves nele. Sem instruções especiais, sem chamadas ao SO. Essa é toda a premissa da programação bare-metal e o tema do Módulo 03. Os endereços fixos que vais usar (GPIOC em 0x40011000, etc.) vêm do manual de referência RM0041.

6. Porque é que o RISC/ARM domina o mundo embebido

  • Energia. Um descodificador e um núcleo pequenos e simples comutam menos transístores por instrução. Para dispositivos a bateria e sempre ligados, isto é decisivo.
  • Densidade de código (Thumb-2). Mais programa cabe na escassa Flash on-chip, pelo que um chip mais barato com menos memória ainda consegue fazer o trabalho.
  • Tempos de ciclo previsíveis. Instruções de formato fixo, na maioria de propósito único, e uma pipeline curta e simples tornam muito mais fácil raciocinar sobre exatamente quanto tempo demora um caminho de código — essencial para controlo de tempo real estrito.
  • IP licenciável. A ARM vende o design do núcleo; fabricantes como a ST acrescentam os seus próprios periféricos. Esse ecossistema é a razão pela qual um Cortex-M3 aparece em milhares de chips diferentes.
Exercício Guiado

7. Vê o padrão load-store com os teus próprios olhos

Objetivo: compilar uma pequena função C para Cortex-M3, desmontar (disassemble) o ficheiro objeto e identificar as instruções ldr / str / aritméticas — confirmando que o ARM calcula apenas em registos e toca na memória apenas via loads e stores.

Toolchain não é necessária para aprender a lição Este exercício usa arm-none-eabi-gcc e arm-none-eabi-objdump. Podem ainda não estar no teu PATH — o README do curso aborda a instalação da toolchain gcc-arm-none-eabi. Se não os conseguires correr agora, não há problema: uma desmontagem real anotada está impressa abaixo no §7.3, e a lição assenta inteiramente na sua leitura. Não há QEMU nem ficheiro de solução para este módulo.

7.2 Compilar & desmontar

De dentro da pasta 01-arm-architecture/:

# Compile to an object file for the Cortex-M3 in Thumb-2 mode.
#   -mcpu=cortex-m3  : target this exact core
#   -mthumb          : emit Thumb-2 (the only state the M3 understands)
#   -Og              : light optimization -> readable but real code
#   -c               : compile only, do not link
arm-none-eabi-gcc -mcpu=cortex-m3 -mthumb -Og -c example.c -o example.o

# Disassemble the machine code back to assembly.
arm-none-eabi-objdump -d example.o

Lê a secção <sum_array>: do output. Procura três coisas:

  1. Um ldr que lê um elemento do array da memória para um registo.
  2. Um add que opera apenas sobre registos (sem operando de memória à vista).
  3. O salto (b / cmp) que forma o ciclo.

Repara no que está ausente: não há instrução que some uma posição de memória diretamente a um registo num único passo. Cada valor tem de ser primeiro carregado com ldr. Isso é a arquitetura load-store, no metal.

7.3 Desmontagem de exemplo (anotada)

Aqui está um output -Og representativo da função terminada, para que a lição assente mesmo sem a toolchain. Os registos/codificações exatos variam conforme a versão do GCC, mas a forma é sempre esta:

00000000 <sum_array>: 0: 2300 movs r3, #0 ; total = 0 (immediate -> register) 2: 2200 movs r2, #0 ; i = 0 4: e004 b.n 10 <.loop_test> ; --- loop body --- 6: f850 0022 ldr.w r0, [r0, r2, lsl #2] ; LOAD data[i] (base r0 + i*4) -> r0 ; *** memory is read ONLY here *** a: 4403 add r3, r0 ; total += data[i] (register + register) ; *** arithmetic: registers ONLY *** c: 3201 adds r2, #1 ; i++ (register + immediate) ; --- loop test --- e: ... ; .loop_test: 10: 4291 cmp r1, r2 ; compare n with i (registers) 12: d8f8 bhi.n 6 ; if i < n, branch back up to the LOAD 14: 4618 mov r0, r3 ; return value in r0 (ARM calling convention) 16: 4770 bx lr ; return (branch to Link Register)
  • ldr em 0x6 é a única instrução que lê memória — puxa data[i] para um registo primeiro. O lsl #2 é o índice do array escalado por 4 (o tamanho de um int32_t).
  • add em 0xa soma dois registos. O CPU não consegue somar memória a um registo numa só instrução — o valor teve de ser carregado primeiro.
  • As larguras das instruções são mistas: movs/add são formas Thumb de 2 bytes, ldr.w é uma forma Thumb-2 de 4 bytes (o .w = "wide", larga). Essa mistura é o Thumb-2 em ação.
  • Não há aqui str porque o único resultado é um valor de retorno, que vive num registo (r0). Acrescenta uma linha como out[0] = total; e recompila — verás aparecer um str, completando o trio load-compute-store.
Experimenta esta variação Altera a função para escrever o resultado através de um ponteiro (void sum_into(const int32_t *data, uint32_t n, int32_t *out) com *out = total;) e desmonta outra vez. Encontra a nova instrução str e confirma que é a única coisa que escreve em memória.

7.4 Dicas

Dica 1 — o corpo do ciclo

Um ciclo de acumulação de manual é tudo o que precisas:

for (uint32_t i = 0U; i < n; i++)
{
    total += data[i];
}

Resiste a otimizá-lo. Quanto mais claro o C, mais claro o assembly.

Dica 2 — "arm-none-eabi-gcc: command not found"

A toolchain não está instalada ou não está no teu PATH. Em Debian/Ubuntu: sudo apt install gcc-arm-none-eabi. Em macOS com Homebrew: brew install --cask gcc-arm-embedded. O README do curso tem as instruções completas. Podes ainda assim completar a lição a partir da desmontagem anotada do §7.3.

Dica 3 — que linhas são o padrão load-store?

Percorre o output do objdump à procura dos mnemónicos ldr (load) e str (store). Tudo entre eles — add, cmp, mov, adds — opera apenas sobre registos e imediatos. Essa separação é a arquitetura load-store. Se vires um ldr a alimentar um add que (eventualmente) alimenta um str, encontraste o ritmo load → compute → store.

Perguntas de Verificação

8. Verifica a tua compreensão

P1. O que significa "arquitetura load-store", numa frase?

As instruções de aritmética e lógica operam apenas sobre registos (e imediatos); a única forma de aceder à memória é através de instruções dedicadas de load (ldr) e store (str). Por isso qualquer cálculo sobre um valor em memória é sempre load → calcular-em-registos → store.

P2. Porque é que o Cortex-M3 (e toda a família Cortex-M) executa Thumb-2 e não o antigo conjunto de instruções ARM de 32 bits?

Densidade de código. Os microcontroladores têm muito pouca Flash (o STM32F100RBT6 tem 128 KB), por isso encaixar mais programa em menos bytes é crítico. O Thumb-2 mistura codificações de 16 e 32 bits para obter desempenho próximo do ARM com código muito mais apertado, pelo que a ARM removeu inteiramente do Cortex-M o estado ARM exclusivo de 4 bytes. O núcleo está sempre em estado Thumb — que é a razão pela qual compilamos sempre com -mthumb.

P3. Quantos bits de largura tem um registo de uso geral do Cortex-M3, e quantos existem?

Cada registo tem 32 bits de largura. Existem 16, r0–r15; r0–r12 são de uso geral, enquanto r13=sp, r14=lr e r15=pc têm papéis especiais.

P4. Indica um compromisso (tradeoff) central do RISC face ao CISC.

As instruções RISC fazem cada uma menos, por isso uma dada tarefa precisa frequentemente de mais instruções (e mais memória de programa no caso ingénuo) do que em CISC — mas cada instrução é simples, de formato fixo e rápida de descodificar, dando um núcleo mais pequeno/de menor consumo e tempos muito mais previsíveis. O CISC empacota mais trabalho por instrução (e pode ser mais denso por operação) à custa de uma descodificação complexa e de comprimento variável. (O Thumb-2 ataca especificamente a desvantagem de tamanho de código do RISC.)

P5. Nomeando as camadas: qual é a relação entre ARMv7-M, Cortex-M3 e STM32F100?

O ARMv7-M é a arquitetura — o contrato de quais instruções e registos existem. O Cortex-M3 é o núcleo específico da ARM que implementa o ARMv7-M (pipeline de 3 andares, NVIC, SysTick, apenas Thumb-2). O STM32F100 é o chip da ST que envolve um núcleo Cortex-M3 com Flash, SRAM e periféricos. (E o STM32VLDISCOVERY é a placa que transporta esse chip.)

P6. Porque é que "tudo mapeado em memória" torna possível o C bare-metal sem instruções de I/O especiais?

Cada registo de periférico vive num endereço fixo no mesmo espaço de endereçamento plano de 32 bits que a RAM. Controlar hardware é, portanto, apenas ler e escrever em endereços de memória — o que em C significa desreferenciar um ponteiro volatile. Não são necessárias instruções especiais nem chamadas ao SO, e é exatamente sobre isso que o Módulo 03 se constrói.
