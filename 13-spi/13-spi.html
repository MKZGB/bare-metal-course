Module 13 SPI Driver — Full-Duplex Master

Configure SPI1 as a software-NSS master, then run the classic polled byte exchange: wait TXE → write DR → wait RXNE → read DR. This is the final coding module — and one of the few peripherals where QEMU lets you read your configuration back.

Dependencies This module reuses the polled-UART printf retarget from Module 06 to narrate progress, and the GPIO alternate-function configuration from Module 04 (CRL/CRH). The bounded-wait discipline is the same one introduced for the unemulated peripherals in Modules 09–12.
Theory

1. What SPI is

The Serial Peripheral Interface is a synchronous, full-duplex, master/slave bus. Unlike I²C (which has addressing and is half-duplex), SPI uses a dedicated chip-select line per slave and four signals:

SignalDirection (from master)SPI1 pinMeaning
SCKoutputPA5Serial clock — the master always generates it
MISOinputPA6Master In, Slave Out — the slave's reply line
MOSIoutputPA7Master Out, Slave In — the master's data line
NSSoutput(software here)Slave select / chip-select, active low

There is no addressing on the bus — you pick which slave you are talking to by pulling its chip-select low. Bits are shifted out MSB-first by default (bit order is selectable via LSBFIRST).

1.1 Full-duplex means TX and RX happen together

SPI is built from two shift registers, one in the master and one in the slave, wired into a ring by MOSI and MISO and clocked by SCK:

MASTER SLAVE +-----------------+ MOSI +-----------------+ | shift reg >>--|---------------->|--> shift reg | | ^ | MISO | | | | |---------|<----------------|--------+ | +-------|---------+ SCK ------> +-----------------+ one SCK pulse shifts ONE bit each way Send a byte and you RECEIVE a byte in the same 8 clocks.

Every clocked frame shifts one TX bit out and one RX bit in. So a "transmit" and a "receive" are the same operation: to read from a slave you still have to clock out a (dummy) byte, and to write you still get back whatever the slave happened to put on MISO.

1.2 Clock modes: CPOL and CPHA

Master and slave must agree on the clock's idle level and on which edge samples data. Two bits in CR1 select one of four modes:

ModeCPOLCPHASCK idleData sampled on
000lowfirst (rising) edge
101lowsecond (falling) edge
210highfirst (falling) edge
311highsecond (rising) edge

CPOL sets the idle polarity of SCK; CPHA chooses whether data is captured on the first or second clock edge of each bit. Mode 0 (both zero) is the most common default — we use it here.

1.3 The SPI1 registers

SPI1 lives at 0x40013000 on the APB2 bus (RM0041 §25). It is a small peripheral — just four registers matter here:

RegisterAddressKey bits
SPI1_CR10x40013000bit0 CPHA, bit1 CPOL, bit2 MSTR, bits[5:3] BR, bit6 SPE, bit7 LSBFIRST, bit8 SSI, bit9 SSM, bit11 DFF
SPI1_CR20x40013004interrupt / DMA / SS-output enables (not used here)
SPI1_SR0x40013008bit0 RXNE, bit1 TXE, bit7 BSY
SPI1_DR0x4001300Cdata register (write → TX, read → RX)

The interesting CR1 bits:

  • MSTR (bit 2) — 1 = master mode (we generate SCK).
  • BR[5:3] — baud-rate prescaler: divides the APB2 clock (8 MHz here) by 2, 4, 8, 16, … up to 256 (000…111). We use 011 = ÷16 → 500 kHz SCK.
  • SPE (bit 6) — SPI peripheral enable. Set this last, after everything else is configured.
  • SSM (bit 9) + SSI (bit 8) — software slave management. With SSM=1 the hardware NSS pin is ignored and the internal NSS level comes from SSI instead. A master must see its NSS held high, so we set SSI=1 too.
  • DFF (bit 11) — data frame format: 0 = 8-bit, 1 = 16-bit. We use 8-bit.
  • LSBFIRST (bit 7) — 0 = MSB first (default), 1 = LSB first.
⚠ Master mode needs SSM + SSI (or a real NSS) In master mode, if the hardware NSS pin is allowed to go low, the SPI peripheral thinks another master grabbed the bus, clears MSTR and raises a mode-fault (MODF). When you are not wiring a hardware NSS, you must set SSM=1 (software manage NSS) and SSI=1 (drive the internal NSS high) so the master stays a master. Forgetting these is the classic "SPI configures then immediately dies" bug.

1.4 Bring-up sequence

  1. Enable the SPI1 clock (RCC_APB2ENR bit 12, SPI1EN) and the GPIOA clock.
  2. Configure PA5/PA6/PA7 as alternate-function push-pull so SPI1 owns the pins.
  3. Write CR1: MSTR | SSM | SSI | BR | SPE (with CPOL=CPHA=0, DFF=0, LSBFIRST=0).

Then for each byte: wait TXE → write DR → wait RXNE → read DR.

What QEMU shows — SPI is only PARTIALLY emulated The stm32vldiscovery machine in QEMU emulates the SPI1 registers but attaches no slave device (there is no ADXL345, no flash, nothing on MISO). Concretely:
  • CR1 reads back the value you wrote — so unlike GPIO and I²C, you can actually prove your configuration landed. We print it.
  • The status flags TXE and RXNE are reported set, so the polling flow makes progress and the program does not freeze.
  • But any byte you "receive" by reading DR is meaningless — nothing drives MISO. On real hardware, with a real slave selected, that read returns the byte the slave clocked back.
You still use bounded waits (a guard counter) on every flag, per the course rule, so a never-setting flag can never hang the program.

2.3 Hints

Hint 1 — clocks and pins

RCC_APB2ENR |= (RCC_SPI1EN | RCC_IOPAEN); then GPIOA_CRL = (GPIOA_CRL & ~SPI_PINS_MASK) | SPI_PINS_AF;. As always, a peripheral with no clock silently ignores writes.

Hint 2 — composing CR1 (don't forget SSM + SSI)

One write does it: SPI1_CR1 = SPI_CR1_MSTR | SPI_CR1_SSM | SPI_CR1_SSI | SPI_CR1_BR_DIV16 | SPI_CR1_SPE;. Leaving out SSM/SSI drops the master out of master mode. SPE is set in the same write (it is fine here because nothing else needs to happen first).

Hint 3 — bounded flag waits (never spin forever)

Pattern for both waits: guard = WAIT_GUARD; while (((SPI1_SR & SPI_SR_TXE) == 0U) && (guard > 0U)) { guard--; }. Repeat with SPI_SR_RXNE for the receive side. The guard only matters if a flag never sets; on real hardware the loop exits early.

Hint 4 — write then read DR

Writing SPI1_DR = (uint32_t)tx; launches the clocked exchange. After RXNE is set, return (uint8_t)SPI1_DR; reads the received byte and clears RXNE.

Verification Questions

3. Check your understanding

Q1. What do CPOL and CPHA each control, and what is "mode 0"?

CPOL sets the idle level of SCK (0 = idle low, 1 = idle high). CPHA selects which clock edge of each bit period samples the data (0 = first edge, 1 = second edge). Mode 0 is CPOL=0, CPHA=0: clock idles low and data is sampled on the first (rising) edge. Master and slave must use the same mode or every byte is garbled.

Q2. Why must a software-NSS master set both SSM and SSI?

In master mode the SPI watches its NSS input. If NSS goes low it assumes another master took the bus, clears MSTR and signals a mode fault, so transfers stop. With no hardware NSS wired, you set SSM=1 to tell the peripheral to ignore the physical NSS pin and use the internal level from SSI instead, and SSI=1 to drive that internal NSS high — keeping the master a master.

Q3. What does "full-duplex" mean here — how can one transfer both send and receive?

SPI is a ring of two shift registers (master + slave) joined by MOSI and MISO and driven by a shared SCK. Each clock edge shifts one bit out of the master on MOSI and one bit in from the slave on MISO simultaneously. After 8 clocks the master has sent a byte and received a byte. That is why a "read" still clocks out a dummy byte and a "write" still yields a received byte — transmit and receive are the same physical operation.

Q4. What does the BR field do, and what SCK frequency does ÷16 give here?

BR[5:3] is the baud-rate prescaler. It divides the SPI's bus clock (APB2 = 8 MHz on this board at reset) by a power of two from 2 (000) to 256 (111). We use 011 = ÷16, giving SCK = 8 MHz / 16 = 500 kHz. You pick the largest divider (slowest clock) the slave tolerates, or a fast one if it can keep up.

Q5. The received bytes print as 0x00 in QEMU. Is the code wrong?

No. QEMU emulates the SPI1 registers (you can even read CR1 back as 0x35C) and sets the TXE/RXNE flags so the polling flow runs, but it attaches no slave device — nothing drives MISO. So the received byte carries no real data and reads back as 0x00. On real hardware with a slave selected (e.g. an ADXL345 answering a register read), that same code returns the slave's reply.

Q6. What does DFF select, and why did we leave it 0?

DFF (bit 11) chooses the frame size: 0 = 8-bit, 1 = 16-bit. Many SPI devices and protocols are byte-oriented, so 8-bit frames are the common default and match our uint8_t transfer helper. Set DFF=1 only when a device expects 16-bit words; DFF must not be changed while the SPI is enabled and busy.
