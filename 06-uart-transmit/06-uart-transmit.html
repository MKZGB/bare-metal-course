Module 06 UART Transmit — Your First Visible Output

Bring up USART1 as a transmitter on PA9, write a tiny polled driver, then retarget printf() so the chip can talk to you over the serial console in QEMU.

★ This is the module everything else builds on Until now QEMU showed you nothing (it does not emulate GPIO). It does fully emulate the USARTs. By the end of this page your firmware prints real text to your terminal — the channel you will use to debug every later module.
Dependencies You should be comfortable with Module 03 (Memory-Mapped Registers) and the clock-gating / CRL/CRH pin-config pattern from Module 04 (GPIO Output). We reuse both here.
Theory

1. What a UART does

A UART (Universal Asynchronous Receiver/Transmitter — on STM32 the peripheral is called USART, the "S" adding optional synchronous modes) turns a byte into a stream of bits sent one wire at a time. "Asynchronous" means there is no clock line: sender and receiver must agree in advance on the bit rate (the baud rate) and on the frame format.

1.1 The frame: start, data, stop

The TX line idles high. To send a byte the transmitter:

  1. pulls the line low for one bit — the start bit (this falling edge is how the receiver knows a byte is coming);
  2. shifts out the data bits (8 here), LSB first;
  3. returns the line high for one (or more) bit(s) — the stop bit.
8N1 frame for the byte 'A' = 0x41 = 0b0100_0001 (LSB first) idle start | b0 b1 b2 b3 b4 b5 b6 b7 | stop idle ----, ,----------, ,----, ,---------------- | | 1 0 0 0 0 | 1 0| 0 | '---' (data bits, LSB first) (line returns high) \_ falling edge: receiver starts sampling "8N1" = 8 data bits, No parity, 1 stop bit (the USART reset default).

Each bit lasts exactly 1 / baud seconds. At 9600 baud a bit is ~104 µs and a 10-bit frame (start + 8 data + stop) takes ~1.04 ms.

1.2 Baud rate and the BRR register

The USART derives the bit timing by dividing its peripheral clock fCK (here PCLK2 = HSI = 8 MHz) by an oversampling factor of 16 times a programmable divisor USARTDIV:

baud = f_CK / (16 * USARTDIV) For 9600 baud at 8 MHz: USARTDIV = 8 000 000 / (16 * 9600) = 52.0833... BRR splits USARTDIV into a 12-bit mantissa and a 4-bit fraction: mantissa = 52 = 0x34 -> BRR[15:4] fraction = round(0.0833 * 16) = 1 = 0x1 -> BRR[3:0] BRR = 0x341

So we write USART1_BRR = 0x341. (RM0041 §27.3.4.)

QEMU ignores BRR — set it anyway QEMU does not model real bit timing, so the exact BRR value does not affect whether you see output. On real hardware a wrong BRR gives garbage on the terminal, so we always program it correctly. Set your host terminal to 9600 8N1 when connecting to a physical board.

1.3 The status flags: TXE vs TC

Inside the transmitter there are two stages: a writable data register (DR) and a shift register that actually clocks bits onto the wire. Two flags in USART1_SR tell you their state:

FlagBitMeaningUse it to…
TXE7TX data register empty — its byte has moved into the shift register, so DR can accept the next byte.decide when it is safe to write the next byte (fast, keeps the pipe full)
TC6Transmission complete — the shift register has finished and the line is idle.know the last byte has fully left the wire (e.g. before sleeping or disabling the USART)

For a simple byte-at-a-time driver you poll TXE: as soon as it is set, write the next byte. You only need TC when you must be sure the very last bit has physically left the pin.

1.4 Bring-up sequence

USART1 hangs off APB2 and its TX pin is PA9, so port A's clock is needed too. The sequence:

StepRegisterWrite
1. Clock GPIOA + USART1RCC_APB2ENR (0x40021018)set bit 2 (IOPAEN) and bit 14 (USART1EN)
2. PA9 = AF push-pullGPIOA_CRH (0x40010804)pin-9 nibble (bits[7:4]) = 0xB
3. Baud rateUSART1_BRR (0x40013808)0x341 (9600 @ 8 MHz)
4. Enable TX + USARTUSART1_CR1 (0x4001380C)set bit 3 (TE) and bit 13 (UE)

Why 0xB for PA9? The pin is driven by the peripheral, not by software, so it must be an alternate-function output. Decoding the STM32F1 4-bit field: MODE = 0b11 (output, 50 MHz) and CNF = 0b10 (AF push-pull) → (CNF<<2)|MODE = 0b1011 = 0xB. (RM0041 §8.2, §9.1.11. Remember: STM32F1 has no MODER.)

★ Early Sanity Check

2. Prove you have output BEFORE building anything

Before you write a driver, a string helper, or retarget printf, confirm the absolute minimum works: configure USART1 and push one string onto the wire. If this does not appear, nothing downstream will — so debug it here, in isolation.

2.1 Minimal "hello" (this one is complete on purpose)

Create 06-uart-transmit/hello.c:

#include <stdint.h>

#define RCC_APB2ENR  (*(volatile uint32_t *)0x40021018U)
#define GPIOA_CRH    (*(volatile uint32_t *)0x40010804U)
#define USART1_SR    (*(volatile uint32_t *)0x40013800U)
#define USART1_DR    (*(volatile uint32_t *)0x40013804U)
#define USART1_BRR   (*(volatile uint32_t *)0x40013808U)
#define USART1_CR1   (*(volatile uint32_t *)0x4001380CU)

int main(void)
{
    /* clock GPIOA (bit2) + USART1 (bit14) */
    RCC_APB2ENR |= (1U << 2) | (1U << 14);

    /* PA9 = AF push-pull 50MHz: pin-9 nibble at bits[7:4] = 0xB */
    GPIOA_CRH = (GPIOA_CRH & ~(0xFU << 4)) | (0xBU << 4);

    USART1_BRR = 0x341U;                 /* 9600 @ 8 MHz            */
    USART1_CR1 = (1U << 3) | (1U << 13);  /* TE | UE                 */

    const char *msg = "hello uart\r\n";
    while (*msg != '\0')
    {
        while ((USART1_SR & (1U << 7)) == 0U) { }  /* wait for TXE */
        USART1_DR = (uint32_t)*msg;
        msg++;
    }

    for (;;) { }
    return 0;
}

2.2 The exact run command

From inside 06-uart-transmit/:

make -f ../common/Makefile SRC=hello.c run

You should see hello uart printed on your terminal. Quit QEMU with Ctrl-A then X. If you see that line, your toolchain, board model, and USART setup are all good — proceed to the full driver.

⚠ The 3-serial gotcha — read this if QEMU refuses to start

The stm32vldiscovery machine instantiates three USARTs (USART1, USART2, USART3), and QEMU insists every serial device has a backend. A bare -serial stdio therefore fails with:

cannot use stdio by multiple character devices

The fix is to give all three a backend — USART1 to your terminal, the other two to null (discarded):

qemu-system-arm -M stm32vldiscovery -display none \
    -serial stdio -serial null -serial null \
    -kernel build/firmware.elf

The first -serial maps to USART1 (the one this course uses) → stdio = your terminal. The shared Makefile's run target already does exactly this, so make … run just works. Quit with Ctrl-A then X.

Guided Exercise

3. Build a driver, then retarget printf

Now grow the one-shot hello into reusable pieces and finish with working printf:

  1. uart_init() — the bring-up sequence from §1.4.
  2. uart_putc() — poll TXE, then write DR.
  3. uart_puts() — loop uart_putc over a string.
  4. _write() — the syscall newlib's printf calls; route it to uart_putc.
  5. setvbuf(stdout, NULL, _IONBF, 0) once, then printf away.

3.3 Hints

Hint 1 — uart_init

Clocks: RCC_APB2ENR |= (RCC_IOPAEN | RCC_USART1EN);
Pin: GPIOA_CRH = (GPIOA_CRH & ~(0xFU << 4)) | (0xBU << 4);
Baud: USART1_BRR = 0x341U;
Enable: USART1_CR1 = (USART_CR1_TE | USART_CR1_UE);

Hint 2 — uart_putc polls TXE

Wait, then write. QEMU sets TXE, so this loop is safe (no infinite hang as with unemulated peripherals):

while ((USART1_SR & USART_SR_TXE) == 0U) { }
USART1_DR = (uint32_t)c;
Hint 3 — _write and the \n → \r\n translation

A bare '\n' on a serial terminal moves down a line but not back to column 1. Send '\r' first:

for (int i = 0; i < len; i++) {
    if (buf[i] == '\n') { uart_putc((uint8_t)'\r'); }
    uart_putc((uint8_t)buf[i]);
}
return len;

Then in main: setvbuf(stdout, NULL, _IONBF, 0); once, and now printf("v=%d 0x%X\n", n, n); works.

Hint 4 — "_X is not implemented" linker warnings

Pulling in printf makes newlib reference other syscalls (_read, _close, _lseek, _isatty, _fstat). The nosys stubs warn at link time. Provide your own trivial versions (each just returns a safe value) to silence them — see the reference solution. They are never actually exercised in a transmit-only program.

Verification Questions

4. Check your understanding

Q1. QEMU prints "cannot use stdio by multiple character devices" and won't start. What's wrong, and what's the fix?

The stm32vldiscovery machine has three USARTs, and QEMU requires a backend for every serial device. A single -serial stdio tries to attach stdio to more than one device, which is illegal. Give all three a backend — the first goes to your terminal, the rest are discarded: -serial stdio -serial null -serial null. The first -serial maps to USART1, the one this course uses. The Makefile's run target already does this.

Q2. What is the difference between TXE and TC, and which should a byte-at-a-time driver poll?

TXE (TX data register empty, bit 7) sets when the byte you wrote to DR has moved into the shift register, so DR can take the next byte — even though the previous one is still being clocked onto the wire. TC (transmission complete, bit 6) sets only when the shift register has finished and the line is idle. A streaming driver polls TXE so it can refill DR immediately and keep the transmitter busy. You wait on TC only when you must guarantee the last bit has physically left the pin (e.g. before disabling the USART or entering a low-power sleep).

Q3. Why is PA9 configured with the nibble 0xB instead of a normal push-pull output like 0x2?

The pin must be driven by the USART peripheral, not by writes to the GPIO output data register. STM32F1 expresses that by selecting an alternate-function output mode in CNF. Decoding 0xB = 0b1011: MODE = 0b11 (output, 50 MHz) and CNF = 0b10 (AF push-pull). A plain output (CNF = 0b00, nibble 0x2/0x3) would leave the line under software control and the USART could not drive it.

Q4. Why does retargeting printf only require defining _write()? And why call setvbuf with _IONBF?

newlib's printf formats text into a buffer and then emits it through the single low-level syscall _write(fd, buf, len). The Makefile links nosys.specs, which supplies a do-nothing stub _write; by defining our own _write we override it and send those bytes to the UART. Calling setvbuf(stdout, NULL, _IONBF, 0) makes stdout unbuffered: characters flush immediately, and — crucially — newlib then allocates no stdout buffer, so malloc/_sbrk are never pulled in. That keeps the firmware free of dynamic memory (MISRA-friendly).

Q5. Your code does printf("%f", x) and gets nothing useful. Why?

The build links newlib-nano (--specs=nano.specs), whose printf is integer-only to save flash — floating-point conversions like %f are not included. Use integer/hex/string conversions (%d, %u, %x, %c, %s). For fixed-point you would scale and print integers (e.g. millivolts as %d). Embedded code rarely needs %f, which is why nano omits it.

Q6. Why translate '\n' into "\r\n" inside _write?

On a serial terminal the two control characters do different things: '\n' (line feed) moves the cursor down a row, and '\r' (carriage return) moves it back to column 1. A bare '\n' leaves the next line starting wherever the previous one ended (a "staircase" effect). Sending '\r' before '\n' gives the expected start-of-next-line behaviour on a raw console.
