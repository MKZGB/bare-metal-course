Módulo 06 Transmissão UART — A Sua Primeira Saída Visível

Coloca o USART1 a funcionar como transmissor em PA9, escreve um pequeno driver por polling e, depois, redireciona o printf() para que o chip possa falar contigo através da consola série no QEMU.

★ Este é o módulo sobre o qual tudo o resto assenta Até agora o QEMU não te mostrava nada (não emula GPIO). Mas emula totalmente os USARTs. No final desta página o teu firmware imprime texto real no teu terminal — o canal que vais usar para depurar todos os módulos seguintes.
Dependências Deves estar à vontade com o Módulo 03 (Registos Mapeados em Memória) e com o padrão de clock-gating / configuração de pinos CRL/CRH do Módulo 04 (Saída GPIO). Aqui reutilizamos ambos.
Teoria

1. O que faz um UART

Um UART (Universal Asynchronous Receiver/Transmitter — no STM32 o periférico chama-se USART, em que o "S" acrescenta modos síncronos opcionais) transforma um byte numa sequência de bits enviados um de cada vez por um único fio. "Assíncrono" significa que não há linha de relógio: emissor e recetor têm de concordar previamente quanto à taxa de bits (o baud rate) e ao formato da trama.

1.1 A trama: start, dados, stop

A linha TX fica em repouso a nível alto. Para enviar um byte, o transmissor:

  1. puxa a linha para nível baixo durante um bit — o start bit (é por esta transição descendente que o recetor sabe que vem aí um byte);
  2. desloca para fora os bits de dados (8 aqui), com o LSB primeiro;
  3. devolve a linha a nível alto durante um (ou mais) bit(s) — o stop bit.
8N1 frame for the byte 'A' = 0x41 = 0b0100_0001 (LSB first) idle start | b0 b1 b2 b3 b4 b5 b6 b7 | stop idle ----, ,----------, ,----, ,---------------- | | 1 0 0 0 0 | 1 0| 0 | '---' (data bits, LSB first) (line returns high) \_ falling edge: receiver starts sampling "8N1" = 8 data bits, No parity, 1 stop bit (the USART reset default).

Cada bit dura exatamente 1 / baud segundos. A 9600 baud um bit demora ~104 µs e uma trama de 10 bits (start + 8 de dados + stop) demora ~1,04 ms.

1.2 O baud rate e o registo BRR

O USART deriva o timing dos bits dividindo o seu relógio de periférico fCK (aqui PCLK2 = HSI = 8 MHz) por um fator de sobreamostragem de 16 vezes um divisor programável USARTDIV:

baud = f_CK / (16 * USARTDIV) For 9600 baud at 8 MHz: USARTDIV = 8 000 000 / (16 * 9600) = 52.0833... BRR splits USARTDIV into a 12-bit mantissa and a 4-bit fraction: mantissa = 52 = 0x34 -> BRR[15:4] fraction = round(0.0833 * 16) = 1 = 0x1 -> BRR[3:0] BRR = 0x341

Por isso escrevemos USART1_BRR = 0x341. (RM0041 §27.3.4.)

O QEMU ignora o BRR — configura-o à mesma O QEMU não modela o timing real dos bits, por isso o valor exato de BRR não afeta se vês ou não saída. Em hardware real um BRR errado dá lixo no terminal, por isso programamo-lo sempre corretamente. Configura o terminal do teu computador para 9600 8N1 quando ligares a uma placa física.

1.3 As flags de estado: TXE vs TC

Dentro do transmissor há duas etapas: um registo de dados escrevível (DR) e um registo de deslocamento (shift register) que efetivamente coloca os bits no fio. Duas flags em USART1_SR indicam o seu estado:

FlagBitSignificadoUsa-a para…
TXE7Registo de dados de TX vazio — o seu byte passou para o shift register, por isso o DR pode aceitar o byte seguinte.decidir quando é seguro escrever o próximo byte (rápido, mantém o pipe cheio)
TC6Transmissão completa — o shift register terminou e a linha está em repouso.saber que o último byte já saiu completamente do fio (p. ex. antes de adormecer ou desativar o USART)

Para um driver simples byte a byte fazes polling do TXE: assim que estiver ativo, escreves o byte seguinte. Só precisas do TC quando tens de garantir que o último bit já saiu fisicamente do pino.

1.4 Sequência de arranque

O USART1 está pendurado no APB2 e o seu pino TX é o PA9, por isso o relógio do porto A também é necessário. A sequência:

PassoRegistoEscrita
1. Relógio GPIOA + USART1RCC_APB2ENR (0x40021018)ativar bit 2 (IOPAEN) e bit 14 (USART1EN)
2. PA9 = AF push-pullGPIOA_CRH (0x40010804)nibble do pino 9 (bits[7:4]) = 0xB
3. Baud rateUSART1_BRR (0x40013808)0x341 (9600 @ 8 MHz)
4. Ativar TX + USARTUSART1_CR1 (0x4001380C)ativar bit 3 (TE) e bit 13 (UE)

Porquê 0xB para o PA9? O pino é controlado pelo periférico, não por software, por isso tem de ser uma saída em função alternada (alternate-function). Descodificando o campo de 4 bits do STM32F1: MODE = 0b11 (saída, 50 MHz) e CNF = 0b10 (AF push-pull) → (CNF<<2)|MODE = 0b1011 = 0xB. (RM0041 §8.2, §9.1.11. Lembra-te: o STM32F1 não tem MODER.)

★ Verificação Inicial

2. Prova que tens saída ANTES de construir seja o que for

Antes de escreveres um driver, uma função auxiliar de strings ou de redirecionares o printf, confirma que o mínimo absoluto funciona: configura o USART1 e coloca uma string no fio. Se isto não aparecer, nada a jusante funcionará — por isso depura-o aqui, isoladamente.

2.1 "hello" mínimo (este está completo de propósito)

Cria 06-uart-transmit/hello.c:

#include <stdint.h>

#define RCC_APB2ENR  (*(volatile uint32_t *)0x40021018U)
#define GPIOA_CRH    (*(volatile uint32_t *)0x40010804U)
#define USART1_SR    (*(volatile uint32_t *)0x40013800U)
#define USART1_DR    (*(volatile uint32_t *)0x40013804U)
#define USART1_BRR   (*(volatile uint32_t *)0x40013808U)
#define USART1_CR1   (*(volatile uint32_t *)0x4001380CU)

int main(void)
{
    /* relógio GPIOA (bit2) + USART1 (bit14) */
    RCC_APB2ENR |= (1U << 2) | (1U << 14);

    /* PA9 = AF push-pull 50MHz: nibble do pino 9 em bits[7:4] = 0xB */
    GPIOA_CRH = (GPIOA_CRH & ~(0xFU << 4)) | (0xBU << 4);

    USART1_BRR = 0x341U;                 /* 9600 @ 8 MHz            */
    USART1_CR1 = (1U << 3) | (1U << 13);  /* TE | UE                 */

    const char *msg = "hello uart\r\n";
    while (*msg != '\0')
    {
        while ((USART1_SR & (1U << 7)) == 0U) { }  /* espera por TXE */
        USART1_DR = (uint32_t)*msg;
        msg++;
    }

    for (;;) { }
    return 0;
}

2.2 O comando exato para correr

A partir de dentro de 06-uart-transmit/:

make -f ../common/Makefile SRC=hello.c run

Deverás ver hello uart impresso no teu terminal. Sai do QEMU com Ctrl-A seguido de X. Se vires essa linha, a tua toolchain, o modelo da placa e a configuração do USART estão todos bons — avança para o driver completo.

⚠ O problema das 3 portas série — lê isto se o QEMU se recusar a arrancar

A máquina stm32vldiscovery instancia três USARTs (USART1, USART2, USART3), e o QEMU insiste que todos os dispositivos série tenham um backend. Um simples -serial stdio falha, portanto, com:

cannot use stdio by multiple character devices

A solução é dar um backend aos três — o USART1 para o teu terminal, os outros dois para null (descartados):

qemu-system-arm -M stm32vldiscovery -display none \
    -serial stdio -serial null -serial null \
    -kernel build/firmware.elf

O primeiro -serial mapeia para o USART1 (o que este curso usa) → stdio = o teu terminal. O alvo run do Makefile partilhado já faz exatamente isto, por isso make … run simplesmente funciona. Sai com Ctrl-A seguido de X.

Exercício Guiado

3. Constrói um driver e depois redireciona o printf

Agora faz crescer o hello de um só disparo em peças reutilizáveis e termina com um printf funcional:

  1. uart_init() — a sequência de arranque da §1.4.
  2. uart_putc() — faz polling do TXE e depois escreve no DR.
  3. uart_puts() — itera uart_putc sobre uma string.
  4. _write() — a syscall que o printf da newlib chama; encaminha-a para uart_putc.
  5. setvbuf(stdout, NULL, _IONBF, 0) uma vez, depois é só usar printf.

3.3 Dicas

Dica 1 — uart_init

Relógios: RCC_APB2ENR |= (RCC_IOPAEN | RCC_USART1EN);
Pino: GPIOA_CRH = (GPIOA_CRH & ~(0xFU << 4)) | (0xBU << 4);
Baud: USART1_BRR = 0x341U;
Ativar: USART1_CR1 = (USART_CR1_TE | USART_CR1_UE);

Dica 2 — uart_putc faz polling do TXE

Espera, depois escreve. O QEMU ativa o TXE, por isso este ciclo é seguro (sem bloqueio infinito como acontece com periféricos não emulados):

while ((USART1_SR & USART_SR_TXE) == 0U) { }
USART1_DR = (uint32_t)c;
Dica 3 — _write e a tradução \n → \r\n

Um '\n' sozinho num terminal série desce uma linha mas não volta à coluna 1. Envia '\r' primeiro:

for (int i = 0; i < len; i++) {
    if (buf[i] == '\n') { uart_putc((uint8_t)'\r'); }
    uart_putc((uint8_t)buf[i]);
}
return len;

Depois, no main: setvbuf(stdout, NULL, _IONBF, 0); uma vez, e agora printf("v=%d 0x%X\n", n, n); funciona.

Dica 4 — avisos do linker "_X is not implemented"

Trazer o printf faz a newlib referenciar outras syscalls (_read, _close, _lseek, _isatty, _fstat). Os stubs nosys avisam na ligação. Fornece as tuas próprias versões triviais (cada uma devolve apenas um valor seguro) para os silenciar — vê a solução de referência. Nunca são realmente exercitadas num programa só de transmissão.

Perguntas de Verificação

4. Testa a tua compreensão

P1. O QEMU imprime "cannot use stdio by multiple character devices" e não arranca. O que está errado e qual é a solução?

A máquina stm32vldiscovery tem três USARTs, e o QEMU exige um backend para cada dispositivo série. Um único -serial stdio tenta ligar o stdio a mais do que um dispositivo, o que é ilegal. Dá um backend aos três — o primeiro vai para o teu terminal, os restantes são descartados: -serial stdio -serial null -serial null. O primeiro -serial mapeia para o USART1, o que este curso usa. O alvo run do Makefile já faz isto.

P2. Qual é a diferença entre TXE e TC, e qual deve um driver byte a byte fazer polling?

O TXE (registo de dados de TX vazio, bit 7) fica ativo quando o byte que escreveste em DR passou para o shift register, de modo que o DR pode receber o próximo byte — mesmo que o anterior ainda esteja a ser colocado no fio. O TC (transmissão completa, bit 6) só fica ativo quando o shift register terminou e a linha está em repouso. Um driver de streaming faz polling do TXE para poder reabastecer o DR imediatamente e manter o transmissor ocupado. Só esperas pelo TC quando tens de garantir que o último bit saiu fisicamente do pino (p. ex. antes de desativar o USART ou entrar num modo de baixo consumo).

P3. Porque é o PA9 configurado com o nibble 0xB em vez de uma saída push-pull normal como 0x2?

O pino tem de ser controlado pelo periférico USART, e não por escritas no registo de dados de saída do GPIO. O STM32F1 exprime isso selecionando um modo de saída em função alternada (alternate-function) em CNF. Descodificando 0xB = 0b1011: MODE = 0b11 (saída, 50 MHz) e CNF = 0b10 (AF push-pull). Uma saída comum (CNF = 0b00, nibble 0x2/0x3) deixaria a linha sob controlo de software e o USART não a conseguiria controlar.

P4. Porque é que redirecionar o printf só exige definir o _write()? E porquê chamar setvbuf com _IONBF?

O printf da newlib formata o texto num buffer e depois emite-o através da única syscall de baixo nível _write(fd, buf, len). O Makefile liga com nosys.specs, que fornece um stub _write que não faz nada; ao definir o nosso próprio _write substituímo-lo e enviamos esses bytes para o UART. Chamar setvbuf(stdout, NULL, _IONBF, 0) torna o stdout sem buffer: os carateres são descarregados de imediato e — crucialmente — a newlib não aloca então nenhum buffer para o stdout, pelo que malloc/_sbrk nunca são trazidos. Isso mantém o firmware livre de memória dinâmica (compatível com MISRA).

P5. O teu código faz printf("%f", x) e não obténs nada de útil. Porquê?

A build liga com a newlib-nano (--specs=nano.specs), cujo printf é apenas para inteiros para poupar flash — conversões de vírgula flutuante como %f não estão incluídas. Usa conversões de inteiros/hex/string (%d, %u, %x, %c, %s). Para vírgula fixa, escalavas e imprimias inteiros (p. ex. milivolts como %d). Código embebido raramente precisa de %f, e é por isso que a nano o omite.

P6. Porque traduzir '\n' em "\r\n" dentro do _write?

Num terminal série os dois carateres de controlo fazem coisas diferentes: '\n' (line feed) move o cursor uma linha para baixo, e '\r' (carriage return) move-o de volta à coluna 1. Um '\n' sozinho deixa a linha seguinte a começar onde a anterior acabou (um efeito de "escada"). Enviar '\r' antes de '\n' dá o comportamento esperado de início de nova linha numa consola em bruto.
