Módulo 08 Temporizador SysTick — Um Tique de Milissegundo Preciso

Usa o temporizador SysTick do núcleo Cortex-M3 para construir um verdadeiro delay_ms() e um relógio de tempo de funcionamento em milissegundos, depois imprime um contador crescente uma vez por segundo através da UART.

Dependências Este módulo reutiliza a transmissão UART + o redirecionamento do printf do Módulo 06. As interrupções são introduzidas aqui de forma informal (a exceção SysTick); o Módulo 10 cobre o NVIC na íntegra. Já deves estar à vontade com o acesso a registos via volatile dos Módulos 03–04.
Este é totalmente visível no QEMU Ao contrário do GPIO e dos temporizadores de uso geral, o SysTick é um periférico do núcleo do Cortex-M3 e o QEMU emula-o por completo — conta, ativa o COUNTFLAG e dispara a exceção SysTick. Combinado com o USART1 (também totalmente emulado), vais ver valores reais e crescentes de tempo de funcionamento a percorrer o teu terminal.
Teoria

1. Para que serve o SysTick

O Módulo 04 usou uma espera ativa com um ciclo for para a temporização — simples, mas a duração depende do compilador, do nível de otimização e da velocidade do relógio. É adivinhação não medida. Quase todos os sistemas embebidos querem, em vez disso, um tique periódico conhecido: um batimento cardíaco que dispara a cada N milissegundos para que o software possa agendar trabalho, expirar operações por tempo limite e medir o tempo decorrido.

O Cortex-M3 traz um incorporado. O SysTick é um contador decrescente de 24 bits que vive dentro do núcleo do CPU (não num barramento de periféricos), por isso existe em todos os Cortex-M3 independentemente do fabricante. Conta de forma decrescente a partir de um valor de recarga até 0, opcionalmente levanta a exceção SysTick quando atinge 0, e depois recarrega automaticamente e continua. Foi concebido precisamente para gerar o tique de um RTOS — mas funciona igualmente bem em bare-metal.

1.1 Os quatro registos (PM0056, @ 0xE000E010)

RegistoEndereçoFinalidade
SYST_CSR0xE000E010Controlo e estado
SYST_RVR0xE000E014Valor de recarga (24 bits)
SYST_CVR0xE000E018Valor atual (a escrita limpa)
SYST_CALIB0xE000E01CCalibração (não usado aqui)

1.2 O registo de controlo/estado, SYST_CSR

BitNomeSignificado
0ENABLE1 = o contador está em funcionamento.
1TICKINT1 = assere a exceção SysTick sempre que o contador atinge 0.
2CLKSOURCESeleção do relógio: 1 = relógio do processador (HCLK), 0 = relógio de referência externo.
16COUNTFLAGLê 1 se o contador atingiu 0 desde a última leitura deste registo. Ler o CSR limpa-o.

1.3 O valor de recarga, SYST_RVR

Quando o contador atinge 0, o hardware carrega-o com SYST_RVR no relógio seguinte e continua a contar. Para obter um período de N tiques do relógio de entrada escreves RVR = N − 1 (o contador visita N estados: N−1, N−2, … 1, 0).

O cálculo da recarga de 1 ms Após o reset, o chip corre no oscilador interno HSI a 8 MHz (RM0041 §7), e com CLKSOURCE=1 o SysTick é alimentado por esse relógio de processador de 8 MHz. Um milissegundo são 8 000 tiques de relógio, por isso:
RVR = (8 000 000 Hz / 1000 Hz) − 1 = 8000 − 1 = 7999
Este é o único número que tens de calcular neste módulo.

1.4 O valor atual, SYST_CVR

Ler o CVR devolve o contador em tempo real. Escrever qualquer valor nele limpa o contador para 0 e também limpa o COUNTFLAG; os dados que escreveste são ignorados. Escreves nele uma vez durante a configuração para que o primeiro período completo comece de forma limpa a partir do valor de recarga.

⚠ Contador de 24 bits — recarga máxima O RVR e o CVR têm apenas 24 bits de largura. A maior recarga é 0xFFFFFF = 16 777 215. A 8 MHz esse é o período único de SysTick mais longo possível — cerca de 16 777 216 / 8 000 000 ≈ 2.097 s. Para intervalos mais longos contas vários tiques em software (que é exatamente o que o nosso tempo de funcionamento de 1 segundo faz: 1000 × tiques de 1 ms).

2. Duas formas de o usar

2.1 Estilo A — sondar o COUNTFLAG (bloqueante)

Para um atraso bloqueante de uma única vez podes deixar o TICKINT desligado e simplesmente sondar o COUNTFLAG num ciclo. Cada vez que lê 1, decorreu um período de recarga completo:

/* atraso bloqueante de 'ticks' periodos, sem interrupcao */
SYST_RVR = 7999U;                       /* periodo de 1 ms        */
SYST_CVR = 0U;                          /* limpar                 */
SYST_CSR = CSR_CLKSOURCE | CSR_ENABLE;  /* correr, SEM TICKINT    */
while (ticks > 0U) {
    while ((SYST_CSR & CSR_COUNTFLAG) == 0U) { /* esperar um periodo */ }
    ticks--;                             /* a leitura acima limpou o COUNTFLAG */
}

Simples, mas o CPU fica preso em espera ativa e não pode fazer mais nada.

2.2 Estilo B — orientado a interrupções (um relógio em funcionamento contínuo)

Define TICKINT=1 e o temporizador levanta a exceção SysTick em cada período. Um pequeno manipulador incrementa um contador, e o resto do teu programa lê esse contador sempre que quer saber o tempo. É isto que construímos no exercício — dá um relógio de milissegundos em funcionamento contínuo que custa quase nada.

Nenhuma linha do NVIC para ativar O SysTick é uma exceção de sistema do Cortex-M3 (posição de vetor 15), não um IRQ externo, por isso não tocas nos registos ISER do NVIC para o ativar — o bit TICKINT é o seu ativador. O nome do manipulador é SysTick_Handler, que já é um símbolo WEAK em common/startup.c; definir uma função com esse nome exato sobrepõe-no.

2.3 Porque é que o contador tem de ser volatile

O contador de tiques é escrito pelo manipulador de interrupção e lido pelo main(). Do ponto de vista do main(), nada no seu próprio código linear altera alguma vez essa variável — por isso um compilador otimizador tem o direito de a ler uma vez para um registo do CPU e reutilizar essa cópia em cache. Um ciclo de atraso construído sobre um valor em cache nunca veria as atualizações do manipulador e ficaria a girar para sempre. Declará-la como volatile uint32_t diz ao compilador "isto pode mudar nas tuas costas — relê-a da memória todas as vezes." Este é o uso canónico do volatile: uma variável partilhada entre uma ISR e o código principal.

3.3 Dicas

Dica 1 — o valor de recarga e a ativação do CSR

1 ms a 8 MHz são 8000 relógios de entrada, por isso RVR = 8000 − 1 = 7999. Ativa com os três bits ao mesmo tempo: SYST_CSR = CSR_CLKSOURCE | CSR_TICKINT | CSR_ENABLE; (relógio do processador, levantar a exceção no zero, e correr).

Dica 2 — o manipulador e o contador volatile

O g_msticks tem de ser static volatile uint32_t. O corpo do manipulador é uma linha: g_msticks++;. Entrar na exceção confirma-a automaticamente — não precisas de ler/limpar o COUNTFLAG em modo de interrupção.

Dica 3 — implementar o delay_ms()

Captura o início, depois faz um ciclo sobre a diferença:

uint32_t start = g_msticks;
while ((g_msticks - start) < ms) { /* girar */ }

A subtração sem sinal significa que o teste se mantém correto mesmo quando o g_msticks dá a volta para além de 0xFFFFFFFF. Como o g_msticks é volatile, cada iteração do ciclo relê-o da memória.

Dica 4 — limpar o CVR

Qualquer escrita limpa-o; o valor é ignorado. Basta fazer SYST_CVR = 0U; antes de ativar, para que o primeiro período medido seja uma recarga completa em vez do lixo que estava no contador no reset.

Perguntas de Verificação

4. Verifica a tua compreensão

P1. Porque é que o valor de recarga é 7999 e não 8000 para um tique de 1 ms a 8 MHz?

O contador conta de forma decrescente por cada valor desde a recarga até 0 inclusive, o que são recarga + 1 estados. Para abranger exatamente 8000 tiques de relógio (1 ms a 8 MHz) precisas de 8000 estados, por isso RVR = 8000 − 1 = 7999. Escrever 8000 daria um período de 8001 tiques — errado por um.

P2. Porque é que o contador de tiques tem de ser declarado volatile?

É escrito pela ISR SysTick_Handler e lido pelo main(). O compilador não consegue ver que a ISR corre, por isso sem volatile pode pôr a variável em cache num registo e reutilizar o valor obsoleto — fazendo o delay_ms() girar para sempre. O volatile força uma leitura fresca da memória em cada acesso, que é exatamente o que uma variável partilhada entre uma interrupção e o código principal requer.

P3. Quando é que sondarias o COUNTFLAG em vez de usar a interrupção?

A sondagem (Estilo A) serve para um atraso bloqueante simples de uma única vez quando o CPU não tem mais nada para fazer e não queres nenhuma ISR. A interrupção (Estilo B) é melhor para um relógio em funcionamento contínuo: o temporizador mantém a contagem de tiques em segundo plano enquanto o main() faz outro trabalho e apenas lê a contagem quando necessário. Nota que ler o CSR limpa o COUNTFLAG, por isso um ciclo de sondagem consome a flag cada vez que a observa.

P4. O que é que o bit CLKSOURCE seleciona, e o que usamos?

O CLKSOURCE escolhe o relógio de entrada do SysTick: 1 = o relógio do processador (HCLK), 0 = um relógio de referência externo (um relógio definido pela implementação, tipicamente mais lento e dividido). Definimo-lo a 1 para que o SysTick seja acionado pelo relógio de processador HSI completo de 8 MHz, que é o que o cálculo de recarga 7999 assume.

P5. Qual é o período único mais longo que o SysTick consegue produzir a 8 MHz, e porquê?

O RVR/CVR têm 24 bits, por isso a recarga máxima é 0xFFFFFF = 16 777 215, dando 16 777 216 tiques de relógio por período. A 8 MHz isso são cerca de 16 777 216 / 8 000 000 ≈ 2.10 s. Para intervalos mais longos (como o nosso 1 segundo construído a partir de 1000 × 1 ms) acumulas vários tiques curtos em software em vez de pedir um período enorme.

P6. Precisas de ativar um IRQ de SysTick no NVIC?

Não. O SysTick é uma exceção de sistema do Cortex-M3 (ranhura de vetor 15), não uma das linhas de IRQ de periférico. O seu ativador é o bit TICKINT em SYST_CSR; os registos ISER do NVIC controlam apenas os IRQ externos (cobertos no Módulo 10). Definir TICKINT|ENABLE é tudo o que é necessário para começar a receber chamadas a SysTick_Handler.
