Module 08 SysTick Timer — An Accurate Millisecond Tick

Use the Cortex-M3 core SysTick timer to build a real delay_ms() and a millisecond uptime clock, then print an incrementing counter once per second over UART.

Dependencies This module reuses the UART transmit + printf retarget from Module 06. Interrupts are introduced informally here (the SysTick exception); Module 10 covers the NVIC in full. You should already be comfortable with volatile register access from Modules 03–04.
This one is fully visible in QEMU Unlike GPIO and the general-purpose timers, SysTick is a core peripheral of the Cortex-M3 and QEMU emulates it completely — it counts, sets COUNTFLAG, and fires the SysTick exception. Combined with the (also fully-emulated) USART1, you will watch real, increasing uptime values scroll up your terminal.
Theory

1. What SysTick is for

Module 04 used a for-loop busy-wait for timing — simple, but the duration depends on the compiler, optimisation level, and clock speed. It is unmeasured guesswork. Almost every embedded system instead wants a known, periodic tick: a heartbeat that fires every N milliseconds so software can schedule work, time out operations, and measure elapsed time.

The Cortex-M3 builds one in. SysTick is a 24-bit down-counter that lives inside the CPU core (not on a peripheral bus), so it exists on every Cortex-M3 regardless of vendor. It counts down from a reload value to 0, optionally raises the SysTick exception when it hits 0, then automatically reloads and continues. It was designed precisely to drive an RTOS tick — but it works just as well bare-metal.

1.1 The four registers (PM0056, @ 0xE000E010)

RegisterAddressPurpose
SYST_CSR0xE000E010Control & status
SYST_RVR0xE000E014Reload value (24-bit)
SYST_CVR0xE000E018Current value (write clears)
SYST_CALIB0xE000E01CCalibration (unused here)

1.2 The control/status register, SYST_CSR

BitNameMeaning
0ENABLE1 = counter is running.
1TICKINT1 = assert the SysTick exception each time the counter reaches 0.
2CLKSOURCEClock select: 1 = processor clock (HCLK), 0 = external reference clock.
16COUNTFLAGReads 1 if the counter reached 0 since this register was last read. Reading CSR clears it.

1.3 The reload value, SYST_RVR

When the counter reaches 0, the hardware loads it with SYST_RVR on the next clock and keeps counting. To get a period of N input-clock ticks you write RVR = N − 1 (the counter visits N states: N−1, N−2, … 1, 0).

The 1 ms reload calculation After reset the chip runs on the internal HSI oscillator at 8 MHz (RM0041 §7), and with CLKSOURCE=1 SysTick is fed by that 8 MHz processor clock. One millisecond is 8 000 clock ticks, so:
RVR = (8 000 000 Hz / 1000 Hz) − 1 = 8000 − 1 = 7999
This is the only number you must compute in this module.

1.4 The current value, SYST_CVR

Reading CVR returns the live counter. Writing any value to it clears the counter to 0 and also clears COUNTFLAG; the data you wrote is ignored. You write it once during setup so the first full period starts cleanly from the reload value.

⚠ 24-bit counter — maximum reload RVR and CVR are only 24 bits wide. The largest reload is 0xFFFFFF = 16 777 215. At 8 MHz that is the longest single SysTick period possible — about 16 777 216 / 8 000 000 ≈ 2.097 s. For longer intervals you count multiple ticks in software (which is exactly what our 1-second uptime does: 1000 × 1 ms ticks).

2. Two ways to use it

2.1 Style A — polling COUNTFLAG (blocking)

For a one-shot blocking delay you can leave TICKINT off and simply poll COUNTFLAG in a loop. Each time it reads 1, one full reload period has elapsed:

/* blocking delay of 'ticks' periods, no interrupt */
SYST_RVR = 7999U;                       /* 1 ms period            */
SYST_CVR = 0U;                          /* clear                  */
SYST_CSR = CSR_CLKSOURCE | CSR_ENABLE;  /* run, NO TICKINT        */
while (ticks > 0U) {
    while ((SYST_CSR & CSR_COUNTFLAG) == 0U) { /* wait one period */ }
    ticks--;                             /* the read above cleared COUNTFLAG */
}

Simple, but the CPU is stuck spinning and cannot do anything else.

2.2 Style B — interrupt-driven (a free-running clock)

Set TICKINT=1 and the timer raises the SysTick exception every period. A tiny handler increments a counter, and the rest of your program reads that counter whenever it wants the time. This is what we build in the exercise — it gives a continuously running millisecond clock that costs almost nothing.

No NVIC line to enable SysTick is a Cortex-M3 system exception (vector position 15), not an external IRQ, so you do not touch the NVIC ISER registers to enable it — the TICKINT bit is its enable. The handler name is SysTick_Handler, which is already a WEAK symbol in common/startup.c; defining a function with that exact name overrides it.

2.3 Why the counter must be volatile

The tick counter is written by the interrupt handler and read by main(). From main()'s point of view, nothing in its own straight-line code ever changes that variable — so an optimising compiler is entitled to read it once into a CPU register and reuse that cached copy. A delay loop built on a cached value would never see the handler's updates and would spin forever. Declaring it volatile uint32_t tells the compiler "this can change behind your back — re-read it from memory every single time." This is the canonical use of volatile: a variable shared between an ISR and mainline code.

3.3 Hints

Hint 1 — the reload value and CSR enable

1 ms at 8 MHz is 8000 input clocks, so RVR = 8000 − 1 = 7999. Enable with all three bits at once: SYST_CSR = CSR_CLKSOURCE | CSR_TICKINT | CSR_ENABLE; (processor clock, raise the exception on zero, and run).

Hint 2 — the handler and the volatile counter

g_msticks must be static volatile uint32_t. The handler body is one line: g_msticks++;. Entering the exception acknowledges it automatically — you do not need to read/clear COUNTFLAG in interrupt mode.

Hint 3 — implementing delay_ms()

Snapshot the start, then loop on the difference:

uint32_t start = g_msticks;
while ((g_msticks - start) < ms) { /* spin */ }

Unsigned subtraction means the test stays correct even when g_msticks wraps past 0xFFFFFFFF. Because g_msticks is volatile, each loop iteration re-reads it from memory.

Hint 4 — clearing CVR

Any write clears it; the value is ignored. Just do SYST_CVR = 0U; before enabling, so the first measured period is a full reload rather than whatever junk was in the counter at reset.

Verification Questions

4. Check your understanding

Q1. Why is the reload value 7999 and not 8000 for a 1 ms tick at 8 MHz?

The counter counts down through every value from the reload to 0 inclusive, which is reload + 1 states. To span exactly 8000 clock ticks (1 ms at 8 MHz) you need 8000 states, so RVR = 8000 − 1 = 7999. Writing 8000 would give a period of 8001 ticks — off by one.

Q2. Why must the tick counter be declared volatile?

It is written by the SysTick_Handler ISR and read by main(). The compiler cannot see that the ISR runs, so without volatile it may cache the variable in a register and reuse the stale value — making delay_ms() spin forever. volatile forces a fresh memory read on every access, which is exactly what a variable shared between an interrupt and mainline code requires.

Q3. When would you poll COUNTFLAG instead of using the interrupt?

Polling (Style A) suits a simple one-shot blocking delay when the CPU has nothing else to do and you want no ISR. The interrupt (Style B) is better for a free-running clock: the timer maintains the tick count in the background while main() does other work and merely reads the count when needed. Note that reading CSR clears COUNTFLAG, so a polling loop consumes the flag each time it observes it.

Q4. What does the CLKSOURCE bit select, and what do we use?

CLKSOURCE chooses the SysTick input clock: 1 = the processor clock (HCLK), 0 = an external reference clock (an implementation-defined, typically slower, divided clock). We set it to 1 so SysTick is driven by the full 8 MHz HSI processor clock, which is what the 7999 reload calculation assumes.

Q5. What is the longest single period SysTick can produce at 8 MHz, and why?

RVR/CVR are 24-bit, so the maximum reload is 0xFFFFFF = 16 777 215, giving 16 777 216 clock ticks per period. At 8 MHz that is about 16 777 216 / 8 000 000 ≈ 2.10 s. For longer intervals (like our 1 second built from 1000 × 1 ms) you accumulate multiple short ticks in software rather than asking for one huge period.

Q6. Do you need to enable a SysTick IRQ in the NVIC?

No. SysTick is a Cortex-M3 system exception (vector slot 15), not one of the peripheral IRQ lines. Its enable is the TICKINT bit in SYST_CSR; the NVIC ISER registers only control external IRQs (covered in Module 10). Setting TICKINT|ENABLE is all that is needed to start getting SysTick_Handler calls.
